## 二、行业调研结论

### 2.1 成熟产品怎么做

成熟桌面 Agent 和开发工具基本形成以下共识：

| 成熟做法 | 说明 | 对小蓝鲸的启发 |
|---|---|---|
| 本地先完整记录 | 桌面端产品通常把关键日志写在本地系统目录，方便断网和崩溃时排查 | 小蓝鲸不能只依赖云端，本地必须能完整排查 |
| 用户投诉自动带编号 | 用户反馈时自动带上对应任务编号，减少反复询问 | 小蓝鲸每次任务必须有唯一排查编号 |
| 模型应用要记录模型、用量、工具和耗时 | 成熟模型应用会记录模型选择、用量、流式输出、工具调用等过程 | 小蓝鲸不能只记录“失败了”，要知道失败在哪一步 |
| 用户界面回放默认遮盖敏感内容 | 成熟回放产品默认遮盖输入、正文、图片等敏感内容 | 小蓝鲸可以做场景还原，但必须默认保护用户隐私 |
| 崩溃要单独收集 | 桌面应用通常把崩溃记录和普通日志分开 | 小蓝鲸需要能定位白屏、闪退、执行中枢退出 |
| 外部工具日志要能串回任务 | 工具自身日志独立保存，但能关联到用户任务 | 小蓝鲸未来接外部工具时，要既能看工具日志，也能串回任务 |

### 2.2 核心设计原则

1. **每次任务一个全链路编号**  
   用户从输入一句话开始，到界面展示、执行中枢、模型调用、工具执行、文件改动、结果生成，全部挂在同一个编号下。

2. **每个步骤一个子编号**  
   一次任务里会有很多步骤：整理目标、模型思考、调用工具、请求确认、写文件、读取网页、生成结果。每一步都要独立编号，方便定位是哪一步慢、错、卡住。

3. **本地日志优先，云端日志增强**  
   本地日志保证“能排障”；云端日志保证“能批量分析、聚合告警、产品洞察”。

4. **异常任务全量保留，正常任务分级保留**  
   出错、卡死、用户投诉、耗时过长、高费用任务要完整保留；正常任务可只保留摘要和关键节点，控制成本。

5. **敏感内容默认不上云**  
   用户输入、文件全文、图片、密钥、完整路径、模型回复全文等，默认只在本地；只有用户明确同意提交诊断包时，才可上传必要内容。

6. **场景还原分等级**  
   不要一上来追求“完全录屏”。先做到执行链路还原，再做到逻辑还原，最后才做界面级还原。

## 三、小蓝鲸当前现状判断

### 3.1 已经具备的基础

| 现有能力 | 位置 | 当前价值 |
|---|---|---|
| 每次任务使用会话编号作为追踪入口 | 主进程任务启动逻辑 | 已能把任务和日志关联起来 |
| 本地诊断日志 | `trace-logger.ts` | 已能记录任务开始、模型、工具、审批、异常、完成等事件 |
| 设置页诊断入口 | `DiagnosticsSection.tsx` | 已能在界面查看最近任务的执行时间线 |
| 任务过程模型 | `items.ts`、`turns.ts` | 已把一轮任务拆成用户消息、思考、工具、计划、审批、问题、回复等条目 |
| 会话持久化 | 主进程会话保存、前端任务状态 | 已能恢复历史对话和执行轮次 |
| 模型配置记录 | 主进程模型配置、模型选择器 | 已知道每次任务用了哪个模型、哪个服务商 |
| 模型用量累计 | 前端任务状态、模型事件 | 已能统计输入和输出消耗 |
| 执行中枢隔离 | `agent-bridge.ts` | 执行中枢和桌面壳分离，有利于单独记录崩溃和异常 |
| 权限审批记录 | 协议和前端状态 | 已能记录用户批准、拒绝、停止、追加输入等关键行为 |

### 3.2 当前主要缺口

| 缺口 | 影响 |
|---|---|
| 全链路编号目前复用会话编号，不够规范 | 同一个会话多轮任务时，排查会混在一起 |
| 没有稳定的“步骤编号”和父子关系 | 不能清晰看出哪个工具属于哪次模型决策 |
| 本地日志和前端会话记录是两套数据 | 回放时可能出现“界面看到的”和“诊断看到的”不一致 |
| 模型调用前后的完整摘要不足 | 无法判断是模型输入问题、模型输出问题，还是工具执行问题 |
| 工具执行记录不够细 | 文件读取、网页抓取、命令执行、写文件等缺少统一前后对照 |
| 没有正式的“用户反馈诊断包” | 用户投诉时仍需要人工引导找日志 |
| 没有云端统一排查平台 | 单个用户能排查，但无法看整体失败率、成本趋势、常见问题 |
| 没有场景还原分级设计 | 当前只能看事件时间线，不能完整还原用户当时体验 |
| 隐私策略还没产品化 | 需要明确哪些只本地、哪些可上传、哪些必须遮盖 |
| 崩溃记录还没接入完整流程 | 程序白屏、闪退、执行中枢退出时不一定能串回任务 |

## 四、产品目标

### 4.1 核心目标

1. **用户投诉快速定位**  
   用户说“刚才任务失败了”，客服或研发能通过一个编号看到完整过程：用户做了什么、小蓝鲸做了什么、卡在哪一步、用了哪个模型、哪个工具报错。

2. **完整还原用户当时场景**  
   尽量还原当时的界面状态、任务输入、模型选择、文件附件、工具结果、权限决策、最终输出，让研发不必反复问用户“你当时怎么操作的”。

3. **产品行为洞察**  
   看清用户最常用哪些功能、在哪些步骤中断、哪些模型更稳定、哪些工具最容易失败、哪些任务成本过高。

4. **安全审计和信任建设**  
   对文件写入、命令执行、权限审批、模型配置修改、日志导出等敏感行为完整留痕，用户能查，团队也能排查风险。

### 4.2 成功标准

| 指标 | 目标 |
|---|---|
| 用户反馈定位时间 | 从小时级降到分钟级 |
| 任务失败根因识别率 | 大部分失败能直接定位到模型、工具、权限、网络、文件、程序崩溃中的某一类 |
| 用户诊断包收集 | 用户一键提交，不需要手动找目录 |
| 场景还原能力 | 至少能还原任务时间线、用户关键操作、模型决策、工具结果、最终界面状态 |
| 产品洞察 | 能按天看到任务成功率、失败原因、平均耗时、模型消耗、工具使用排行 |

## 五、整体方案

### 5.1 总体架构

小蓝鲸日志系统分五层：

| 层级 | 作用 |
|---|---|
| 客户端记录层 | 桌面端内部所有关键动作统一记录 |
| 本地诊断层 | 日志先落本地，支持查看、导出、过期清理 |
| 用户反馈层 | 用户一键提交诊断包，自动带上问题任务 |
| 云端排查层 | 团队内部按编号查询、聚合、告警、分析 |
| 场景还原层 | 把记录重新播放成“当时发生了什么”的时间线和界面状态 |

### 5.2 一次任务的数据流

用户发起任务后：

1. 小蓝鲸生成一个“全链路编号”。
2. 前端界面、主进程、执行中枢都携带这个编号。
3. 每个关键步骤生成“步骤编号”。
4. 日志先写入本地。
5. 任务异常、用户反馈、崩溃时触发重点保留。
6. 用户同意后，诊断包上传到云端。
7. 内部排查平台按编号展示完整过程。
8. 必要时进入场景还原模式。

## 六、编号体系

### 6.1 需要哪些编号

| 编号 | 生成时机 | 作用 |
|---|---|---|
| 设备编号 | 首次安装小蓝鲸时 | 区分是哪台电脑，默认不暴露真实设备名 |
| 用户编号 | 用户登录后 | 关联账号和反馈，不登录则为空 |
| 应用启动编号 | 每次打开小蓝鲸 | 排查“这次启动期间发生了什么” |
| 项目编号 | 用户创建项目时 | 区分不同工作区 |
| 会话编号 | 新建对话时 | 关联一段持续对话 |
| 任务编号 | 每次点击发送、继续、重试、重新生成时 | 排查一次具体执行 |
| 全链路编号 | 每次任务启动时 | 串起界面、执行中枢、模型、工具、云端 |
| 步骤编号 | 每个模型调用、工具调用、审批、文件操作时 | 定位具体步骤 |
| 反馈编号 | 用户提交问题时 | 把用户投诉和任务日志绑定 |
| 崩溃编号 | 程序崩溃时 | 把崩溃文件和任务过程绑定 |

### 6.2 小蓝鲸建议规则

- **会话编号**：代表一个长期对话。
- **任务编号**：代表一次具体执行。
- **全链路编号**：等于任务编号，不再复用会话编号。
- **步骤编号**：每一步独立生成，并记录它属于哪个上级步骤。
- **反馈编号**：用户点击“反馈问题”时生成，并自动绑定最近一次任务编号。

### 6.3 为什么不能继续只用会话编号

现在小蓝鲸很多记录复用会话编号。短期可用，但长期会有问题：

- 一个会话可能有很多轮任务，日志会混在一起。
- 用户点“继续”“重试”“重新生成”时，无法区分哪一次出错。
- 云端排查时，无法精确定位“用户投诉的那一次”。

所以要拆成：会话编号管长期对话，任务编号管一次执行。

## 七、日志分类

### 7.1 任务执行日志

用于回答“小蓝鲸当时具体做了什么”。

| 事件 | 必记内容 |
|---|---|
| 任务开始 | 用户目标摘要、工作台、项目、模型、附件数量、工作区 |
| 目标理解 | 小蓝鲸如何整理用户目标、缺什么信息 |
| 计划生成 | 当前计划、计划版本、用户是否批准 |
| 模型调用开始 | 模型、服务商、参数摘要、输入长度、是否流式输出 |
| 模型调用结束 | 首字耗时、总耗时、消耗量、结束原因、是否降级 |
| 思考过程 | 默认只记摘要，用户授权后可保存完整内容 |
| 工具准备 | 工具名、风险级别、为什么要用 |
| 工具执行 | 入参摘要、结果摘要、耗时、成功或失败 |
| 文件操作 | 文件路径摘要、读写类型、变更摘要、是否可回滚 |
| 权限审批 | 请求原因、风险说明、用户选择、审批耗时 |
| 用户追加输入 | 追加方式、追加内容摘要、发生在哪一步 |
| 用户停止任务 | 停止时间、当时正在执行哪一步 |
| 任务完成 | 最终状态、总耗时、步骤数、产物、消耗量 |
| 任务失败 | 失败分类、失败步骤、可恢复建议 |

### 7.2 界面行为日志

用于回答“用户当时怎么操作的”。

| 事件 | 必记内容 |
|---|---|
| 页面打开 | 打开了哪个页面、入口 |
| 工作台切换 | 从哪个工作台切到哪个工作台 |
| 模型切换 | 旧模型、新模型、是否在任务前切换 |
| 文件选择 | 文件数量、类型、大小，不默认记录文件内容 |
| 项目选择 | 项目编号、工作区摘要 |
| 点击发送 | 触发方式：按钮、快捷键 |
| 点击停止 | 停止时任务状态 |
| 批准或拒绝 | 关联审批请求 |
| 打开诊断日志 | 用户是否自己查看过 |
| 导出诊断包 | 导出范围、是否包含内容 |

### 7.3 模型配置日志

用于排查“是不是配置导致的”。

| 事件 | 必记内容 |
|---|---|
| 新增模型 | 服务商、模型名、连接地址摘要、上下文上限 |
| 切换模型 | 从哪个模型切到哪个模型 |
| 删除模型 | 删除的是哪条配置 |
| 修改模型配置 | 修改了哪些字段，不记录密钥明文 |
| 模型不可用 | 错误类型、状态、重试次数 |
| 模型降级 | 从哪个模型降到哪个模型，原因 |

### 7.4 工具和外部连接日志

用于排查“工具是否可靠”。

| 事件 | 必记内容 |
|---|---|
| 工具注册 | 工具名、版本、能力、风险级别 |
| 工具调用 | 调用原因、参数摘要、执行结果 |
| 外部工具连接 | 启动、断开、异常退出 |
| 外部工具日志 | 单独保存，但必须能回到任务编号 |
| 网页抓取 | 地址摘要、抓取方式、耗时、失败原因 |
| 命令执行 | 命令摘要、工作目录、退出状态、耗时 |
| 文件写入 | 写入文件、变更摘要、快照编号 |

### 7.5 系统稳定性日志

用于排查“应用本身是否坏了”。

| 事件 | 必记内容 |
|---|---|
| 应用启动 | 版本、系统、启动耗时 |
| 应用退出 | 正常退出或异常退出 |
| 主窗口白屏 | 页面加载失败原因 |
| 执行中枢启动 | 子进程启动方式、路径 |
| 执行中枢退出 | 退出状态、最后任务 |
| 未捕获异常 | 错误摘要、发生位置、当前任务 |
| 崩溃文件 | 崩溃编号、当前任务编号、应用版本 |
| 卡顿 | 卡顿时长、页面、当前任务 |

### 7.6 产品洞察日志

用于回答“用户到底怎么用小蓝鲸”。

| 指标 | 说明 |
|---|---|
| 任务成功率 | 按工作台、模型、任务类型统计 |
| 平均任务耗时 | 找出慢任务 |
| 用户中断率 | 用户在哪些步骤最常停止 |
| 权限拒绝率 | 哪些工具让用户不放心 |
| 模型消耗 | 按模型、任务、用户维度统计 |
| 工具使用排行 | 哪些工具最常用、最失败 |
| 产物类型 | 报告、文件、表格、代码修改等 |
| 复用行为 | 用户是否继续、重试、重新生成、复制结果 |

## 八、统一记录格式

### 8.1 公共字段

每条记录至少包含：

- 发生时间
- 日志级别：调试、普通、提醒、警告、错误、严重
- 来源模块：界面、主进程、执行中枢、模型、工具、文件、设置、诊断
- 应用版本
- 系统类型和版本
- 设备编号
- 用户编号
- 应用启动编号
- 项目编号
- 会话编号
- 任务编号
- 全链路编号
- 步骤编号
- 上级步骤编号
- 事件类型
- 事件摘要
- 结果状态
- 耗时
- 是否已遮盖敏感内容

### 8.2 事件内容字段

每条事件根据类型补充：

- 模型事件：模型、服务商、输入长度、输出长度、消耗量、首字耗时、总耗时
- 工具事件：工具名、风险级别、参数摘要、结果摘要、错误分类
- 文件事件：文件类型、路径摘要、大小、读写动作、快照编号
- 用户事件：点击、输入、停止、审批、反馈
- 崩溃事件：崩溃编号、崩溃类型、最后任务、最后步骤

## 九、场景还原设计

### 9.1 第一级：执行链路还原

目标：排查“哪一步出问题”。

能看到：

- 用户发起任务
- 小蓝鲸整理目标
- 模型调用
- 工具调用
- 权限审批
- 文件变更
- 最终结果
- 失败位置和失败原因

这是小蓝鲸当前诊断页的升级版，优先做。

### 9.2 第二级：逻辑还原

目标：排查“为什么它这么做”。

能看到：

- 当时用户输入摘要
- 当时模型配置
- 当时工作台
- 当时可用工具列表
- 每次模型选择工具的理由摘要
- 每个工具的输入和输出摘要
- 每次用户批准、拒绝、追加输入
- 最终回答生成依据

这一级可以复盘模型决策。

### 9.3 第三级：确定性重放

目标：让研发在本地重跑，得到同样的过程。

必须保存：

- 模型每次返回的完整结果或摘要包
- 工具每次返回的完整结果或摘要包
- 关键配置快照
- 工作区文件快照摘要
- 时间、随机结果、用户选择
- 小蓝鲸版本
- 模型配置版本
- 工具版本

注意：这一级涉及大量敏感内容，默认只能本地保存；上传必须用户确认。

### 9.4 第四级：界面体验回放

目标：还原用户当时看到的界面和操作体验。

能看到：

- 用户点击了哪里
- 哪个面板打开了
- 哪个审批卡出现了
- 哪个阶段卡住了
- 输出是如何流式出现的
- 用户什么时候停止、重试、切换模型

隐私要求：

- 默认遮盖输入框文字
- 默认遮盖模型回复全文
- 默认遮盖文件内容
- 默认遮盖图片
- 默认不录屏幕，只记录界面结构和关键状态
- 用户明确同意后，才允许提交更完整的诊断包

## 十、本地日志存储要求

### 10.1 存储位置

小蓝鲸应使用系统标准位置：

| 系统 | 建议位置 |
|---|---|
| macOS | 用户资料目录下的小蓝鲸日志目录 |
| Windows | 用户应用数据目录下的小蓝鲸日志目录 |
| Linux | 用户配置目录下的小蓝鲸日志目录 |

### 10.2 文件拆分

建议按用途拆：

| 文件 | 作用 |
|---|---|
| 任务索引 | 最近任务列表、状态、模型、耗时 |
| 单任务日志 | 一次任务完整过程 |
| 界面行为日志 | 用户关键操作 |
| 执行中枢日志 | 子进程启动、退出、运行异常 |
| 模型调用日志 | 模型请求摘要、响应摘要、消耗 |
| 工具日志 | 工具执行过程 |
| 错误日志 | 所有错误汇总 |
| 崩溃日志 | 程序崩溃文件和摘要 |
| 诊断包目录 | 用户导出的压缩包 |

### 10.3 保存周期

建议默认：

| 类型 | 默认保留 |
|---|---|
| 普通任务摘要 | 30 天 |
| 普通任务完整日志 | 7 天 |
| 异常任务完整日志 | 30 天 |
| 用户反馈相关日志 | 180 天 |
| 崩溃日志 | 90 天 |
| 用户手动导出的诊断包 | 用户自己管理 |

### 10.4 空间上限

建议默认：

- 本地日志总上限：500MB
- 单个任务日志上限：20MB
- 单个诊断包上限：100MB
- 超限后优先删除最旧的普通任务日志
- 异常任务、用户反馈任务、崩溃任务优先保留

### 11.1 用户入口

- 是否允许上传诊断日志

| 本地包 | 只导出到本地，不上传 | 用户要自己发给研发 |

## 十三、隐私与安全要求

### 13.1 数据分级

| 数据 | 默认处理 |
|---|---|
| 密钥、访问凭证、密码 | 永不记录 |
| 用户输入全文 | 默认只本地，上传需同意 |
| 文件内容全文 | 默认不上传 |
| 文件路径 | 只保留摘要或末级文件名 |
| 模型回复全文 | 默认只本地，上传需同意 |
| 图片内容 | 默认不上传，回放中遮盖 |
| 命令内容 | 高风险命令需记录摘要和审批 |
| 工具结果 | 默认摘要，完整结果需同意 |
| 费用和消耗 | 可上传，用于排障和成本分析 |
| 错误信息 | 可上传，但需二次遮盖 |

### 13.2 三道保护

1. **记录前保护**  
   写日志时就遮盖密钥、密码、长文本、完整路径。

2. **上传前保护**  
   诊断包生成时再次检查敏感内容。

3. **云端接收后保护**  
   云端发现疑似敏感内容，拒收或自动遮盖，并标记风险。

### 13.3 用户控制

用户必须能：

- 关闭云端日志上传
- 查看本地日志占用
- 一键清理本地日志
- 导出诊断包
- 选择诊断包包含范围
- 删除云端诊断数据
- 看到小蓝鲸收集了哪些类型的数据

## 十四、异常分类体系

小蓝鲸不要只展示“失败”。需要统一失败分类：

| 分类 | 典型情况 | 用户侧提示方向 |
|---|---|---|
| 模型配置问题 | 没有密钥、地址错误、模型不存在 | 引导重新配置模型 |
| 模型服务问题 | 限流、服务繁忙、响应异常 | 建议稍后重试或切换模型 |
| 网络问题 | 断网、连接超时、证书问题 | 检查网络或代理 |
| 工具输入问题 | 模型给了错误参数 | 让小蓝鲸修正参数重试 |
| 工具执行问题 | 文件不存在、网页抓取失败、命令失败 | 展示具体工具失败原因 |
| 权限被拒绝 | 用户拒绝执行 | 尊重用户选择，提供替代方案 |
| 文件权限问题 | 无权读写、路径越界 | 引导选择正确工作区 |
| 程序稳定性问题 | 子进程退出、白屏、崩溃 | 自动生成诊断包 |
| 任务设计问题 | 目标不清、计划不合理、循环执行 | 反问用户或停止并解释 |
| 成本异常 | 消耗过高、循环调用模型 | 自动提醒并停止 |

## 十六、分阶段实施建议

### 16.1 第一阶段：本地排查闭环

目标：用户投诉时，即使不上云，也能定位问题。

要做：

- 拆分会话编号和任务编号
- 完善全链路编号
- 每个模型调用、工具调用、审批、文件操作都有步骤编号
- 本地日志统一格式
- 诊断页升级为更清晰的任务时间线
- 增加本地诊断包导出
- 增加敏感内容遮盖
- 增加崩溃记录关联最近任务

验收：

- 任意一次任务失败后，能在本地看到失败步骤、模型、工具、耗时、错误摘要。
- 用户能一键导出诊断包。
- 密钥不会出现在诊断包里。

### 16.2 第二阶段：用户反馈闭环

目标：用户反馈不用手动找日志。

要做：

- 任务失败卡片增加“反馈问题”
- 设置页增加“提交诊断”
- 反馈表单自动绑定最近任务
- 诊断包分基础、增强、完整三档
- 用户可选择是否包含对话和文件摘要
- 反馈编号自动生成

验收：

- 用户反馈后，内部能根据反馈编号找到对应任务。
- 用户不需要手动找日志目录。
- 诊断包内容范围可控。

### 16.3 第三阶段：云端排查平台

目标：团队内部能批量排查和统计。

要做：

- 云端接收诊断包
- 内部按编号查询
- 单任务时间线详情页
- 失败分类统计
- 模型稳定性和消耗看板
- 工具成功率看板
- 版本对比看板
- 告警规则

验收：

- 输入反馈编号能打开完整任务详情。
- 能看到每天任务成功率、失败原因、模型消耗。
- 新版本失败率升高能被发现。

### 16.4 第四阶段：场景还原

目标：能最大程度复现用户当时体验。

要做：

- 执行链路回放
- 逻辑回放
- 关键界面状态快照
- 用户操作时间线
- 模型和工具结果回放包
- 回放差异对比
- 严格隐私控制

验收：

- 研发能按一次任务编号看到“当时从输入到失败”的完整过程。
- 对严重问题，用户授权后可复现模型和工具调用链路。
- 默认不会上传用户敏感内容。

## 十八、关键验收场景

| 场景 | 验收标准 |
|---|---|
| 模型没配置 | 日志能明确显示卡在模型配置，不误判为任务失败 |
| 模型超时 | 能看到模型、耗时、重试次数、是否切换模型 |
| 工具失败 | 能看到哪个工具、输入摘要、输出摘要、错误分类 |
| 用户拒绝权限 | 能看到用户拒绝了哪个动作，小蓝鲸是否停止重复尝试 |
| 用户中途停止 | 能看到停止时正在执行哪一步 |
| 文件写失败 | 能看到目标文件、权限问题、是否有快照 |
| 程序崩溃 | 重启后能提示用户提交崩溃诊断 |
| 用户反馈“结果不对” | 能看到模型输入摘要、工具结果、最终回答依据 |
| 任务很慢 | 能看到耗时最长的是模型、工具、文件读取还是界面渲染 |
| 费用过高 | 能看到是哪次模型调用消耗过高 |

## 十九、核心判断

小蓝鲸不应该只做“日志文件”，而应该做一套“任务事实记录系统”。

原因是：小蓝鲸不是普通聊天产品，它会调用模型、读文件、写文件、执行工具、请求权限、生成产物。用户投诉时，最关键的不是看到一堆错误，而是还原：

- 用户当时想做什么
- 小蓝鲸理解成了什么
- 小蓝鲸计划怎么做
- 模型每次让它做什么
- 工具实际返回了什么
- 用户在哪一步批准或拒绝
- 最终为什么成功或失败

当前小蓝鲸已有一个很好的雏形：任务条目模型 + 本地诊断日志 + 设置页诊断入口。下一步不要推倒重来，而是把它升级成正式的“用户反馈、排障、回放、洞察”一体化系统。
