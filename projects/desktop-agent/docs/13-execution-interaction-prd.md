## 1. 产品定位与设计哲学

### 1.1 小蓝鲸不是什么

小蓝鲸不是聊天框，不是终端日志查看器，也不是审计系统。这三者分别代表了三种失败的极端：

- **聊天框**：输入→等→输出，用户看不到过程，长任务黑盒跑，不敢交重要任务。
- **终端日志查看器**：把 stdout/stderr/exit code 全铺开，非技术用户看不懂，信息轰炸导致通知盲。
- **审计系统**：每步都留痕但体验重，像在用监控面板而非任务助手。

### 1.2 小蓝鲸是什么

一个能在本地工作区执行长程复杂任务的桌面 Agent。它做三件事：理解工作材料 → 执行多步骤任务 → 交付可检查结果。交互上，主界面像一个可靠的任务助手，详情层像一个可审计的终端。

### 1.3 三条设计主轴

这三条主轴贯穿全文，是所有交互决策的最高判断标准。

**主轴一：默认轻，展开重（渐进式披露）**

借鉴 Max Gherman 的三层模型和 Smashing Magazine 的 Decision Node Audit 方法。默认界面只展示五样东西：当前阶段、当前动作、关键结果、需要用户做的决定、最终交付物。详细信息（日志、Diff、工具返回、完整执行记录）收进可展开的详情层。

这不是"藏信息"，而是"按用户注意力预算排序"。一个复杂任务可能跑几十个工具调用，用户 90% 的时间只需要知道"现在到哪了、有没有卡住、要不要我管"。把最要紧的信号放最前面，其余折叠。

**主轴二：用户先看影响，再看动作（影响优先）**

借鉴 Smashing Magazine 的内容设计师角色——工程师写的技术准确字符串（"执行命令：npm install axios"）必须被翻译成用户听得懂的影响表达（"需要安装依赖 axios，这会联网访问 npm 并修改 package-lock.json"）。

用户不关心 Agent 调了什么工具、传了什么参数，关心的是：会不会改我的文件、会不会联网、能不能撤销、有没有风险。所有面向用户的文案，先说影响，技术动作可展开查看。

**主轴三：大事清楚，小事糊涂（风险分级展示）**

借鉴人人都是产品经理"大事清楚小事糊涂"法则。大事（改文件、删数据、花钱、发消息）必须让用户清楚、必须确认；小事（读文件、搜索、生成草稿）糊涂过去，不打扰。

这条和项目已有的权限四级（low/medium/high/critical，见 05 权限规则）天然对应：low 自动执行不展示过程细节，medium 展示"正在改"但自动，high/critical 必须确认并展示完整影响。

### 1.4 一条时间轴上的定位

借鉴人人都是产品经理的"自治度滑块"观点：今天的过程可见是一条安全带，为"还不够可靠、还不够快"的模型量身配的。小蓝鲸一期处于滑块偏"人盯"的位置（Balanced 模式），因为模型还没到能完全放手的程度。

但架构上要为"闸门退场"留口子：今天做的过程展示，要能随模型变强而减少打扰，而不是写死成每步必须确认。具体来说，权限模式从 Safe→Balanced→Auto 是一条可上滑的路径，Auto 模式（二期）就是滑块往自治方向的下一步。

## 2. 竞品对标与取舍

> 核查说明：本节结论基于已核查的 Daniel Vaughan 两篇 Codex 深度文章、腾讯云 Claude Code 架构拆解、掘金 Claude Code 架构剖析、ClaudHQ 三方对比、项目资料库 research/compiled 三方对比报告。OpenAI 官方原文因 JS 反爬未直接获取，其核心内容经 Daniel Vaughan 转述交叉核实。

### 2.1 三家的执行展示哲学

| 维度 | Codex | Claude Code | Cursor |
|---|---|---|---|
| 核心模式 | 实时可审查 + 智能审批 | while循环+ReAct，最小脚手架最厚护甲 | 提案-审核，逐个diff确认 |
| 过程展示 | 终端原生流式，diff核心，动作透明 | 动作块逐步展示，较透明但易碎碎念 | 收窄到diff review，不展示底层工具 |
| 权限交互 | Smart Approvals(guardian子代理判定) | 七级权限光谱，模式可切 | Composer逐个accept/reject |
| 透明-高效位置 | 偏高效（全自动审批可跑到底） | 中间（信任结果，看最终diff） | 偏透明（每个改动都确认） |
| 长任务 | Cloud Agent异步，/compact压缩 | 子代理委派，五层压缩 | Checkpoint恢复(较弱) |

### 2.2 从三家各借什么

**从 Codex 借**：
- 计划先行：复杂任务先给计划，用户确认再执行
- 动作可审查：每个工具调用可追溯，但默认不铺开
- diff 是核心证据：文件修改的展示以 diff 为第一优先级
- 截图可直接进入上下文：用户拖入的截图作为上下文而非技术文件展示
- approve/reject 就地完成：审批在当前界面内完成，不跳页
- **完成折叠**（用户特别关注）：执行完成后把过程收起来，只呈现最终结果——这是 Codex 在长任务结束时的关键体验优化
- Smart Approvals 思路：低风险用自动判定（对应小蓝鲸 low 自动），高风险才打扰人

**从 Claude Code 借**：
- 复杂任务先计划：Plan mode 先读文件探索，再提计划，用户批准后执行
- 模式入口放输入框附近：Plan/Default/Accept 模式切换在输入框旁，不藏深
- 权限模式用用户听得懂的名字：不暴露 Ask/Accept/Bypass 等概念名
- Auto 不是无脑自动：是减少低风险打扰，危险动作仍拦
- 动作块概念：每个工具调用是可折叠的动作块

**从 Cursor 借**：
- 高抽象主界面 + 可下钻：任务产物拉到更高抽象层，想深入时 dig deeper
- 结果验证比过程展示更重要：截图/demo/diff 是比日志更友好的证据
- 本地和云端任务能接续：任务集中在一个任务中心，不散在多个聊天和终端
- 提案-审核模式：高风险场景降级到"逐个确认 diff"

### 2.3 明确不照搬什么

**不照搬 Codex**：
- 不让所有用户都像用终端一样用 Agent——主界面是任务流，终端是详情层
- 不把完整日志默认铺开
- 不让非技术用户理解 stdout/stderr/exit code

**不照搬 Claude Code**：
- 不暴露太多模式名——新手不面对 Ask/Accept/Auto/Bypass 一堆概念
- 不把 Bypass 类能力作为普通入口——危险模式藏深
- 不让模型"碎碎念"——不每步显示思考，改阶段意图

**不照搬 Cursor**：
- 小蓝鲸不是纯代码 IDE，不把所有体验围绕 PR/commit
- Work 场景更像任务助手而非代码审查器

### 2.4 小蓝鲸的差异化定位

三家都是"纯代码 Agent"或"IDE"，面向开发者。小蓝鲸的差异化在于：

1. **双工作台**：Code（开发者，深）+ Work（知识工作者，浅但可用）。同一套执行引擎，两套展示语言。
2. **桌面原生**：不是终端 CLI，不是 IDE 插件，是独立桌面应用。主界面是任务流，不是代码编辑器。
3. **影响优先表达**：权限确认和过程展示全部用"用户影响"语言，技术细节收进开发者视图。
4. **完成折叠**：长任务结束后过程收起，只留交付物——让结果感压过日志感。

### 2.5 学术依据：为什么不直接展示原始思考链

两篇学术论文为本 PRD 的核心设计决策提供了学术支撑：

**CopilotLens（arXiv:2506.20062）** 提出"解释层"概念：不直接展示原始思考链（raw chain-of-thought），而是把代码补全重构为透明可解释的交互，通过动态两级界面（高层代码变更概览 + 细节层上下文影响）展示。这直接支撑了本 PRD 的两个决策：①用"阶段意图"替代原始思维链（3.1、4.4 节）；②中间主区域（概览）+ 右侧详情（下钻）的两级结构（4.2 节）。

**Illuminating LLM Coding Agents（arXiv:2508.12555）** 提出三层可视化分析（代码级 / 迭代级 / agent 级），用树状执行轨迹看 Agent 的迭代过程。这支撑了本 PRD 的三层展示：动作级（当前动作卡）→ 阶段级（阶段进度条）→ 任务级（完成态折叠 + 开发者视图完整 Trace）。树状结构特别适合展示 ReAct 的分支/回溯，纳入开发者视图设计。

## 3. 交互设计逻辑总览

### 3.1 执行循环如何映射到界面

小蓝鲸的执行引擎是"7 步闭环（宏观骨架）+ 第 4 步内嵌 ReAct（微观机制）"（见 01 设计基线 3.1）。这两层如何映射到 UI，是整个交互设计的地基。

**7 步闭环 → 用户可见的"阶段"**

用户不需要理解 7 步闭环的技术名称，但需要感知到任务在分阶段推进。7 步映射为用户可理解的阶段：

| 7步闭环步骤 | 用户可见的阶段 | 展示要点 |
|---|---|---|
| 1 目标契约 | "正在理解任务" | 阶段意图，不展示目标契约JSON |
| 2 计划图 | "我会这样做" | 3-5步计划，不展示DAG |
| 3 能力绑定 | （不单独展示） | 内部步骤，折叠进"准备中" |
| 4 工具执行 | "正在执行：{阶段名}" | 当前动作+进度，这是主战场 |
| 5 质量校验 | "正在验证" | 验证目标+结果 |
| 6 产物预览 | "任务完成" | 交付物面板 |
| 7 状态沉淀 | （不单独展示） | 内部步骤，完成后可查 |

**ReAct 循环 → 阶段内的"当前动作"**

第 4 步"工具执行"内部跑 ReAct（想-做-看-再想），一个阶段可能包含多轮 ReAct。用户不需要看到每一轮，只需要看到"当前在做什么动作"。

关键映射规则：
- 一轮 ReAct = 一个"当前动作"（读文件/改文件/跑命令等）
- 同类动作合并展示（连续读 4 个文件 = "已查看 4 个相关文件"）
- 动作完成变成阶段进度条上的一个 ✅
- 失败的动作才展开详情

### 3.2 三个界面状态

沿用 08 线框图的三个核心状态，但补充每个状态的交互逻辑：

**状态 A：首页输入态**
- 右侧详情面板隐藏，底部只有输入框
- 输入框右侧露两个轻量入口：模式（默认"边做边确认"）、附件（文件/图片/链接）
- 高级设置不默认露出
- 展示 3 个示例任务引导

**状态 B：任务执行态**（本 PRD 重点，见第 4 节）
- 右侧详情面板展开，实时显示阶段进度+工具日志+权限记录
- 中间主区域显示当前阶段+当前动作+思考摘要
- 底部变为：暂停/回滚/重新规划/追加/解释当前步骤

**状态 C：产物预览态**（见第 8 节）
- 中间主区域变为交付物面板
- 过程折叠为一行摘要（"已完成 6 步，点击查看详情"）
- 底部变为确认操作

### 3.3 时间线上的展示节奏

一个长程复杂任务（如"给项目加日志模块并生成说明文档"）的时间线展示节奏：

```
0s     用户提交
0-3s   "收到，正在理解任务…"（阶段意图，不展示Preflight判定）
2s+    "正在判断需要读取哪些上下文…"
5s+    "这个任务可能需要多步处理，我会先给一个简短计划。"
       → 展示计划（≤5步，每步≤20字，说明预计影响）
       → [开始] [先只看方案] [补充要求]

执行中  阶段进度条 + 当前动作卡（见第4节）
       同类动作合并，不连续刷卡片
       只在4类场景展示"原因"：即将改文件/即将执行命令/遇到失败/需要确认

压缩时  "正在整理任务上下文"（产品语言，不说token/compact）

完成时  过程折叠为一行 → 交付物面板（见第8节）
```

### 3.4 什么时候该展示，什么时候不该展示

这是 Smashing Magazine 的 Decision Node Audit 方法在小蓝鲸的落地。不是所有步骤都需要展示，只展示"判断点"。

**必须展示的时刻（判断点）**：
1. 即将改文件——展示"准备修改N个文件"+diff预览+能否回滚
2. 即将执行命令——展示"正在运行验证脚本"+验证目标
3. 即将删除——永远强确认，展示删除对象+能否恢复
4. 遇到失败或分歧——展示失败原因+已尝试+可选操作
5. 需要用户确认——展示影响+能否回滚+可选操作
6. 阶段切换——展示新阶段意图

**不该展示的时刻（糊涂过去）**：
1. 读文件——合并展示"已查看N个相关文件"
2. 搜索——合并展示"已搜索到N个结果"
3. 生成草稿——合并展示"已生成草稿"
4. 模型推理过程——不展示原始思维链，只展示阶段意图
5. 内部路由/Preflight判定——不展示模型路由L0/L1/L2/L3
6. 重试——默认隐藏，失败才暴露
7. 能力绑定/状态沉淀——内部步骤，折叠

**判断规则**：一个步骤是否需要展示，取决于它是否是"判断点"（Agent做了概率性判断/影响了用户数据/需要用户决策）。纯读、纯内部流转糊涂过去。

## 4. 长程任务的中间过程展示（核心）

> 本节是整份 PRD 的核心。长程复杂任务（跑几十个工具调用、跨多个阶段、可能触发压缩）的中间过程怎么展示，决定了用户敢不敢把重要任务交给 Agent。

### 4.1 核心问题：长任务的三重焦虑

用户面对长任务有三重焦虑，交互设计必须逐一解决：

1. **"它卡住了吗？"** —— 解决：实时阶段进度 + 当前动作，绝不让界面出现 dead air（借鉴 Max Gherman：工具执行时用户看到 nothing 除非你告诉他们）
2. **"它跑偏了吗？"** —— 解决：阶段意图展示 + 随时可暂停/追加/重新规划
3. **"它会不会搞坏我的东西？"** —— 解决：高风险前确认 + 快照可回滚 + 影响优先文案

### 4.2 执行态界面结构

执行态（状态 B）的中间主区域 + 右侧详情面板分工：

**中间主区域（用户主视线）**：

```
┌─ 中间主区域 ──────────────────────────────────┐
│ ▶ 正在执行：接入日志模块                        │
│                                                │
│ ✅ 已查看项目结构                               │
│ ✅ 已找到主入口文件                             │
│ 🔵 正在修改 src/main.py                        │
│ ⚪ 稍后运行验证                                 │
│                                                │
│ ┌─ 当前动作 ────────────────────────────────┐ │
│ │ 正在修改文件：src/main.py                   │ │
│ │ 改动目的：在应用启动时初始化日志模块         │ │
│ │ [查看 Diff]  [暂停]                        │ │
│ └────────────────────────────────────────────┘ │
└────────────────────────────────────────────────┘
```

要点：
- 顶部一行是**阶段进度条**（✅/🔵/⚪ 三态），不是完整步骤列表
- 阶段进度条只展示当前阶段的子步骤，不展示所有7步
- 当前动作独立成卡，只展示一个，不连续刷多个
- 当前动作卡只说"在做什么+为什么"，技术细节点"查看Diff"展开

**右侧详情面板（可审查）**：

```
┌─ 右侧详情 ──────────┐
│ 🎯 目标契约          │
│   定位并修复bug      │
│   不改测试文件       │
│ 📐 计划              │
│   1.读结构           │
│   2.搜调用           │
│   3.改代码           │
│   ...               │
│ 📜 工具日志           │
│   read_file ✅      │
│   shell:grep ⏳     │
│ 🔒 需审批:1项        │
│ 🧠 记忆引用          │
└─────────────────────┘
```

右侧面板是"可审计层"，默认存在但不是用户主视线。用户想审查时看右边，想了解进展看中间。

### 4.3 长任务的核心机制：动作合并

长任务最大的展示问题是"卡片洪水"——连续读 10 个文件就刷 10 张卡。借鉴 Daniel Vaughan 的工具输出截断策略和 Word 文档的合并展示，核心机制是**同类动作合并**。

**合并规则**：

| 动作类型 | 默认展示 | 展开后 |
|---|---|---|
| 连续读文件 | "已查看 4 个相关文件" + 主要发现 | 文件列表 |
| 连续搜索 | "已搜索到 12 个结果" | 搜索结果详情 |
| 连续改文件 | "已准备修改 2 个文件" + diff预览 | 每个文件的diff |
| 连续生成草稿 | "已生成草稿" | 草稿内容 |
| ReAct多轮推理 | 不展示每轮，只展示阶段结论 | 每轮的thinking+tool |

**合并的触发条件**：
- 同类工具连续调用（如连续 3 次 read_file）
- 同一阶段内的同类动作
- 合并不跨阶段（阶段切换后重新计数）

**不合的例外**：
- 即将改文件：即使只有一个也要展示（判断点）
- 即将执行命令：即使只有一个也要展示（判断点）
- 失败的动作：不合并，单独展开

### 4.4 "原因"展示的三个时机

借鉴 Word 文档的优化：不每步显示思考（避免碎碎念），只在三类场景展示"原因"：

1. **即将改文件时**——展示改动目的（"在应用启动时初始化日志模块"）
2. **即将执行命令时**——展示验证目标（"确认日志模块可以正常启动"）
3. **遇到失败或分歧时**——展示失败原因和已尝试（"缺少 pytest 依赖"）

其余时刻只展示"在做什么"，不展示"为什么"。

### 4.5 长任务的压缩体验（Compact）

长任务必然触发上下文压缩。借鉴 Daniel Vaughan 的 compaction 机制，但产品化展示。

**用户看到的**：
```
正在整理任务上下文
这个任务已经进行了较多步骤，我会把已完成内容整理成摘要，继续后面的工作。
不会丢失你的要求、文件改动和当前进度。
[查看整理摘要]
```

**展开后**：
```
保留内容：
- 当前目标
- 已完成步骤
- 用户最新要求
- 文件变更
- 失败点
- 待办事项
```

**绝对不能对用户说的**（只给开发者视图）：
- "Token 超过 32k，触发 compact"
- "上下文窗口已满"
- "轻度/中度/深度压缩"

**"不会丢失"的依据**（基于 Codex 官方原文）：Codex 的 compaction 返回一个 type=compaction 的特殊条目，带加密内容（encrypted_content），保留模型对原始对话的潜在理解。这意味着压缩不是简单删减，而是保留了模型的"记忆"。产品文案"不会丢失你的要求、文件改动和当前进度"有底层机制支撑，不是空话。

**压缩循环的兜底**：如果 Agent 反复压缩不前进（compaction loop，Daniel Vaughan 指出的已知问题），UI 要检测到并提示用户"任务较复杂，建议拆分为更小的子任务"，而不是无限转圈。

### 4.6 长任务的历史管理

长任务跑了几十步后，历史不能全铺开。借鉴 Word 文档的阶段合并：

**默认展示**：
```
✅ 阶段 1：理解项目结构  查看了 6 个文件，确认日志模块应放在 src/utils
✅ 阶段 2：生成代码变更  新建 1 个文件，修改 1 个文件
🔵 阶段 3：正在验证     运行示例脚本中
```

**展开后**：每个阶段下的具体步骤、工具调用、时间戳。

用户不需要知道内部压缩策略，只要能展开和搜索。阶段合并不等于丢失数据——所有原始事件在 event_log 里（04 契约），开发者视图可查完整 Trace。

### 4.7 长任务的中断体验

用户中断长任务后，最怕"不知道有没有损失"。借鉴 Claude Desktop Docs 原文和 Word 文档的中断优化。

**两种中断模式**（详见 5.2 节）：
- 立即中断（停止按钮）：执行流暂停，展示损失告知
- 修正方向（输入修正不打断）：不中断，当前动作完成后调整

以下为立即中断后的展示：

**无危险操作在进行时**：
```
任务已暂停
当前没有危险操作在进行。
已产生 1 个新文件，尚未修改原有代码。
[继续]  [查看新文件]  [回滚并结束]
```

**正在执行命令时**：
```
正在停止命令…
已停止：python examples/test_logger.py
没有文件被继续修改。
```

中断后必须告诉用户三件事：①当前状态（有没有危险操作在进行）②已产生的变更（新增/修改了什么）③可选操作（继续/查看/回滚）。

### 4.8 长任务的"进度感知"防卡死

借鉴 Max Gherman 的流式和工具状态事件，长任务中必须有持续的进度信号，消除 dead air：

| 等待时长 | 展示 |
|---|---|
| 0-2s | 当前动作卡（"正在修改 src/main.py"） |
| 2-5s | 动作卡 + 微动画（表示在工作） |
| 5-10s | 补充"正在处理，预计还需一会儿" |
| 10s+ | 补充"这个步骤比较耗时，你可以先做别的，完成后会通知你" |

绝不能出现：界面完全静止超过 3 秒且无任何文字变化。如果某个工具执行超过 10 秒，要允许用户切到其他任务（多任务并行），完成后通知。

## 5. 执行中的实时交互操作（V2新增）

> 本节是 V2 的核心新增。基于 OpenAI 官方《Codex agent loop》原文（turn 结束机制、streaming、plan tool）和《Codex CLI Features》原文（Tab 排队、Enter 注入、Esc fork、/copy、@ 文件搜索）、Claude Desktop Docs 原文（interrupt vs redirect、side question、parallel sessions）深化。解决"任务执行中用户能做什么操作、每种操作对执行流有什么影响"。

### 5.1 两种追加指令：注入当前 vs 排队下一条

借鉴 Codex CLI Features 原文：Enter 注入当前 turn，Tab 排队下一条 turn。这是执行中用户干预的两种根本模式。

**注入当前 turn（Enter）**：
- 用户输入修正后按 Enter，不打断当前动作
- Agent 在当前动作完成后立即读到修正，调整下一步
- 适用：方向微调（"顺便也加上日志轮转"、"用 loguru 而不是 logging"）
- UI 表现：输入框旁提示"将在当前步骤完成后生效"

**排队下一条 turn（Tab 或发送按钮）**：
- 用户输入的指令排入队列，等当前 turn 完全结束后执行
- 适用：独立的新要求（"改完后帮我跑一下 lint"、"完成后生成提交说明"）
- UI 表现：输入框上方显示"已排队：生成提交说明（将在当前任务完成后执行）"

**为什么必须区分**：注入当前是"边跑边改方向"，排队下一条是"做完这个再做那个"。混在一起会让用户不确定自己的指令什么时候生效，产生焦虑。

### 5.2 中断的两种模式：立即中断 vs 修正方向

借鉴 Claude Desktop Docs 原文："click the stop button to interrupt immediately, or type a correction and press Enter to send it without stopping the running action"。

**立即中断（停止按钮）**：
- 点击停止按钮，立刻中断当前动作
- 适用：Agent 跑偏了、方向完全错了、在做危险操作
- UI 表现：执行流暂停，展示中断损失告知（见 4.7 节）
- 中断后选项：继续 / 查看已产生变更 / 回滚并结束

**修正方向（输入修正不打断）**：
- 不点停止，直接输入修正按 Enter（即 5.1 的注入当前 turn）
- Agent 当前动作完成后读到修正，调整下一步
- 适用：方向需要调整但当前动作没风险、想追加要求
- UI 表现：不中断，输入框提示"修正将在当前步骤后生效"

### 5.3 回溯编辑与分叉

借鉴 Codex CLI Features 原文："Tap Esc twice while the composer is empty to edit your previous user message. Continue pressing Esc to walk further back in the transcript, then hit Enter to fork from that point."

**回溯编辑**：
- 在输入框为空时按 Esc，编辑上一条用户消息
- 继续按 Esc 往更早的消息回溯
- 编辑后按 Enter，从该点分叉出新分支

**适用场景**：
- 用户发现之前的需求描述不准确，想从某一步重新来
- 不想丢掉已有工作，但想换个方向从中间某点继续

**UI 表现**：
- 回溯时高亮当前编辑的消息，提示"编辑后将从此处分叉，后续步骤保留为历史分支"
- 分叉后在任务列表中显示为"分支任务"，原任务保留

**一期边界**：回溯分叉一期做基础版（编辑上一条消息并重新执行），多步回溯和分支管理二期。

### 5.4 旁支提问

借鉴 Claude Desktop Docs 原文："Ask a side question without derailing the session"。

**旁支提问**：
- 用户在任务执行中想问一个不相关的问题，但不打断当前任务
- 开启旁支提问模式，输入问题，Agent 用当前 session 上下文回答，但不影响主任务执行流
- 适用：主任务跑着，用户想问"这个项目的测试框架是什么"、"这个函数是干嘛的"

**UI 表现**：
- 旁支提问以轻量浮层或侧边对话形式展示，不占用主区域
- 明确标注"旁支提问，不影响当前任务"
- 回答完即收起，主任务继续不受影响

**一期边界**：一期做"追加要求"（影响主任务）和"旁支提问"（不影响）两种模式的区分。旁支提问一期可用"暂停后提问再继续"近似实现，独立浮层二期。

### 5.5 计划的动态更新

基于 Codex agent loop 原文：Codex 有内置 plan tool（update_plan），计划不是开头展示一次就固定，而是 Agent 在执行中可以动态更新。

**动态计划更新**：
- Agent 执行中发现原计划需要调整（发现新问题、某步比预期复杂、需要拆分），调用 plan tool 更新计划
- UI 上计划不是静态展示，而是可变动的

**UI 表现**：
- 计划更新时，右侧计划面板高亮变化部分，提示"计划已更新"
- 用户可看到：原计划 vs 调整后计划，变化点标注
- 不弹窗打断，但让用户知道方向变了

```
📐 计划已更新
原计划第3步"修改代码" → 调整为"修改代码 + 补充类型注解"
原因：发现原有代码缺少类型定义，一并修复更稳妥
```

### 5.6 Turn 结束的界面切换信号

基于 Codex agent loop 原文："In the Codex CLI, we present the assistant message to the user and focus the composer to indicate to the user that it's their turn."

**Turn 结束信号**：
- 每个 turn 以 assistant message 结束（如"我已添加你要求的日志模块"）
- 这是界面从"执行态"切换到"完成态/等待态"的触发信号
- UI 表现：assistant message 流式展示完毕 → 输入框获得焦点 → 状态切换

**与完成折叠的联动**：
- turn 结束 = 任务完成 → 触发完成态过程折叠（第 9 节）
- turn 结束 = 需要用户决策（如"是否继续修改"）→ 保持执行态，高亮决策点

### 5.7 输出复用

借鉴 Codex CLI Features 原文："Use /copy or press Ctrl+O to copy the latest completed Codex output. If a turn is still running, Codex copies the most recent finished output instead of in-progress text."

**输出复用**：
- 用户可随时复制最近完成的输出（代码、报告、diff）
- turn 运行中时，复制的是最近一次完成的输出，不是进行中的
- 适用：用户想把 Agent 生成的代码贴到别处、把报告发给同事

**UI 表现**：
- 完成态交付物面板每个产物有复制按钮
- 执行中时，已完成的阶段产物也可复制（从右侧工具日志或阶段历史中取）

### 5.8 输入态的上下文补充

借鉴 Codex CLI Features 原文："Type @ in the composer to open a fuzzy file search over the workspace root."

**@ 文件注入**：
- 输入框中输入 @ 触发文件模糊搜索
- 选中文件后路径注入消息，Agent 把该文件作为上下文
- 适用：用户想指定 Agent 关注某个文件，而不是让它自己找

**其他上下文补充入口**：
- 附件按钮：上传文件/图片/链接（见 6.6 节）
- 截图拖入：直接拖截图到输入框，作为视觉上下文（见 6.6 节）

## 6. 任务全生命周期交互设计

### 6.1 任务接收（0-5秒）

用户提交后，不要展示复杂 Preflight 判定过程。

| 时间 | 展示 |
|---|---|
| 0-2s | "收到，正在理解任务…" |
| 2-5s | "正在判断需要读取哪些上下文…" |
| 5s+ | "这个任务可能需要多步处理，我会先给一个简短计划。" |

简单问题（Preflight 判定 L0/L1/L2）直接回答，不强制进入长任务流程。只有 L3 复杂任务才进入计划展示。

### 6.2 计划展示

计划要短，不像项目管理表。

**推荐文案**：
```
我会这样做：
• 先查看项目结构，确认日志模块放在哪里
• 新建日志模块，并接入主入口
• 运行验证，确认没有破坏现有逻辑
• 生成一份接入说明文档
预计会改 2-3 个文件，运行 1 次本地命令。
[开始]  [先只看方案]  [补充要求]
```

**体验要求**：
- 不超过 5 步
- 每步不超过 20 字
- 不展示工具名
- 不展示内部路由（L0/L1/L2/L3）
- 不展示模型推理
- 必须说明预计影响：读文件 / 改文件 / 命令 / 联网

**计划不是静态的**：基于 Codex 官方原文，Agent 有内置 plan tool，执行中可动态更新计划。计划展示后不是固定的，Agent 发现新情况时可调整计划并在 UI 上标注变化（详见 5.5 节"计划的动态更新"）。用户看到的计划是"当前最新版"，不是开头那一版。

### 6.3 文件交互

**读文件：默认合并展示**

不每读一个文件刷一张卡。差体验是连续 4 张"读取 a.py / 读取 b.py / 读取 c.py / 读取 d.py"。好体验：

```
已查看 4 个相关文件
主要发现：
- 主入口在 src/main.py
- utils 目录已存在
- 当前项目没有统一日志模块
[查看文件列表]
```

**改文件：Diff 是第一优先级**

文件修改不只展示"已修改"。推荐：

```
已准备修改 2 个文件
• src/utils/logger.py  新建
• src/main.py  修改 +3 -1
[查看变更]  [接受全部]  [让 Agent 调整]
```

对于普通用户，Diff 旁边要有自然语言解释：
```
这处改动是在程序启动时初始化日志，不会改变业务逻辑。
```

**删除文件：永远强确认**

删除不做弱确认，不做倒计时：
```
需要确认删除
Agent 想删除 tmp/debug.log。
这是临时日志文件，删除后可以重新生成。
[删除]  [保留]
```

### 6.4 命令交互

**命令不要默认吓人**

不默认展示 `$ python examples/test_logger.py` + stdout + stderr + exit code。

默认展示：
```
正在运行验证脚本…
验证目标：确认日志模块可以正常启动。
[查看命令]
```

完成后：
```
验证通过
运行了：python examples/test_logger.py
耗时：1.2s
[查看输出]
```

失败时才自动展开关键错误：
```
验证失败
原因：缺少依赖 pytest。
可选操作：[允许安装依赖]  [跳过验证]  [停止]
```

**命令权限的体验表达**

不只写"是否允许执行 npm install axios"，要写：
```
需要安装一个依赖
Agent 需要安装 axios，才能运行当前项目的测试。
这会：
- 联网访问 npm
- 修改 package-lock.json
- 写入 node_modules
[允许这一次]  [不允许]  [只生成修改，不运行测试]
```

### 6.5 权限确认

**P0 不做弱确认倒计时**

弱确认倒计时（"5秒后自动执行"）有两个问题：用户紧张（是不是要赶紧取消）、用户没看清时动作已发生。P0 改为：

| 风险等级 | 处置 |
|---|---|
| 低风险 | 自动执行，执行后可撤销 |
| 中风险 | 简短确认（或 Balanced 模式下快照后自动） |
| 高风险 | 强确认，展示完整影响 |

**权限卡只保留 3 个主按钮**

避免按钮太多。主按钮：`[允许这一次] [不允许] [调整要求]`。

"本任务允许 / 当前项目允许 / 永久允许"放到二级菜单（更多授权方式），一期"批准并记住"按钮置灰（Auto Mode 才启用）。

**权限文案模板**：
```
需要你确认
Agent 想：修改 2 个文件
原因：接入新的日志模块
影响：
- 会新建 src/utils/logger.py
- 会修改 src/main.py
- 可以回滚
[允许这一次]  [不允许]  [调整要求]
```

Work 场景（外发类操作）：
```
需要你确认
Agent 想：发送一封邮件给张三
原因：你要求它同步会议纪要
影响：
- 邮件会被真实发送
- 发送后无法从对方邮箱撤回
[发送]  [先保存草稿]  [取消]
```

**BLOCKED 黑名单的扩充参考**：Claude Code 官方 auto mode 的 classifier 默认拦截清单（见调研 S-04）提供了更全面的危险操作分类，可对照补充小蓝鲸 05 权限规则的 BLOCKED 黑名单。特别是 git 类危险操作（force push、push 到 main、git reset --hard、git checkout -- .、git clean -fd、git stash drop/clear、git commit --amend 非本会话 commit）和生产类操作（terraform/pulumi/cdk destroy、生产部署、大规模删除）。一期以 05 契约的 10 条黑名单为准，二期 Auto Mode 启用时参照此清单扩充。

### 6.6 图片、链接、文档交互

**图片**：默认作为上下文展示，不作为技术文件。
```
已收到截图
Agent 将用它识别：报错信息、页面状态、相关 UI 区域
[查看截图]
```
分析后：
```
截图里的关键信息：
- 报错：401 Unauthorized
- 页面：登录后跳转失败
- 可能相关：token 失效或未写入 cookie
[查看标注]
```

**链接**：访问前说清楚是否联网。
```
需要访问外部链接
Agent 将打开 example.com 读取页面内容。
不会提交表单，也不会登录账号。
[允许访问]  [不访问，基于链接文本分析]
```

**文档**：用户关心结果，不关心解析步骤。
```
已解析文档：需求说明.docx
发现：
- 12 页
- 5 个一级标题
- 18 条需求点
- 3 处疑似重复
[查看大纲]  [开始整理]
```

### 6.7 失败态

失败时用户最怕"不知道怎么办"。不只说失败，要给路。

```
任务还没完成
卡在这里：运行测试时缺少 pytest 依赖。
已经完成：
- 已生成日志模块
- 已接入主入口
- 已生成说明文档
你可以选择：
[允许安装 pytest 后继续]  [跳过测试，保留当前修改]  [回滚所有修改]
```

失败态必须避免：只展示一堆错误日志、只说 command failed、只给 retry 一个按钮。

## 7. 两层体验：普通用户视图与开发者视图

借鉴 Max Gherman 三层模型（Answer/Evidence/Trace）和 CSDN 三级透明性（概要/过程/原始），小蓝鲸做两层体验。

### 7.1 默认：普通用户视图

**展示**：阶段、当前动作、风险确认、结果、交付物

**隐藏**：stdout、stderr、exit code、tool args、raw JSON、完整 Trace

### 7.2 专业：开发者视图

用户打开"显示技术细节"后展示：命令、工作目录、exit code、完整日志、工具参数、工具返回、审计记录、上下文整理摘要。

**入口文案**：`显示技术细节`（不叫"查看完整思考"——那是误导，思考链不是完整思考）

### 7.3 切换逻辑

- 默认普通视图
- 开发者视图是全局开关，一次切换所有任务生效
- 普通视图中"查看详情""查看命令""查看输出"是单点展开，不影响全局开关
- 开发者视图下，所有折叠默认展开

### 7.4 Code vs Work 工作台的展示语言差异

同一套执行引擎，两个工作台的展示语言不同：

| 维度 | Code 工作台 | Work 工作台 |
|---|---|---|
| 面向 | 开发者 | 知识工作者 |
| 过程展示 | 读auth.ts / 改auth.ts:42 / 跑npm test | 读example.com / 写第2节 / 引用3个来源 |
| 核心证据 | Diff、测试结果 | 报告、来源卡、表格 |
| 高频按钮 | 运行测试、查看Diff、回滚、生成提交说明、commit | 导出、复制、写入文件、继续润色、发送前确认 |
| 默认视图 | 更偏开发者视图（但仍是影响优先） | 纯普通用户视图 |

## 8. 数据事件到界面组件的映射

> 本节把 04 数据契约的事件枚举和 09 IPC 契约的 stdio 事件类型，映射到具体 UI 组件。研发实现时，事件来了该更新哪个组件，查这张表。

### 8.1 事件到组件映射表

| Agent stdout 事件 | 04契约事件类型 | 映射到的UI组件 | 展示规则 |
|---|---|---|---|
| `thinking` | THINKING | 阶段意图（中间主区域） | 不展示原始思维链，提炼为阶段意图 |
| `tool_call` | TOOL_CALL | 当前动作卡（中间） | 合并同类，判断点才独立展示 |
| `tool_result` | TOOL_RESULT | 工具日志（右侧）+ 动作卡更新 | 默认收起，失败才展开 |
| `approval_request` | APPROVAL_REQUEST | 权限确认弹窗/卡 | 影响优先文案，3个主按钮 |
| `usage` | TOKEN_USAGE | 开发者视图统计 | 普通视图不展示 |
| `status` | STATUS_CHANGE | 任务状态徽标 | 状态机转移触发 |
| `step_progress` | STEP_PROGRESS | 阶段进度条（中间） | ✅/🔵/⚪ 三态 |
| `artifact` | ARTIFACT_PRODUCED | 产物列表（预览态） | 完成后展示 |
| `error` | ERROR | 失败态卡（中间） | 给路不只给错误 |
| `completed` | TASK_COMPLETED | 触发完成态折叠 | 过程折叠→交付物面板 |
| `chunk` | — | 最终回复流式（中间） | token级流式渲染 |

### 8.2 事件聚合规则

不是每个事件都立即刷 UI。需要聚合：

- 连续 `tool_call`(同类) → 合并为一个动作卡
- 连续 `thinking` → 提炼为一句阶段意图
- `tool_result` 成功 → 更新进度条，不展开
- `tool_result` 失败 → 展开失败详情

### 8.3 实时性要求

- `chunk`、`thinking`、`tool_call`：实时流式，不缓冲（借鉴 Max Gherman：流式是最大 UX 改进）
- `step_progress`：阶段切换时立即更新
- `approval_request`：立即弹窗，阻塞执行流
- `artifact`、`completed`：任务结束批处理

## 9. 完成态：过程折叠与结果呈现

> 这是用户特别关注的优化点，借鉴 Codex 执行完成后把过程收起来只呈现最终结果的设计。

### 9.1 核心设计：过程折叠

任务完成后，执行过程中的所有阶段进度条、当前动作卡、工具日志，全部折叠为一行摘要，中间主区域完全切换为交付物面板。

**触发机制**（基于 Codex agent loop 原文）：每个 turn 以 assistant message 结束（如"我已添加你要求的日志模块"），这是 turn 完成的信号。Codex 原文的做法是"present the assistant message to the user and focus the composer"。小蓝鲸在此基础上增加完成折叠：assistant message 展示完毕 → 触发过程折叠 → 切换到交付物面板 → 输入框获得焦点等待用户下一步。详见 5.6 节"Turn 结束的界面切换信号"。

**完成时的界面变化**：

执行态（过程展开）→ 完成态（过程折叠）：

```
┌─ 中间主区域 ──────────────────────────────────┐
│ 📦 任务完成                                    │
│                                                │
│ 你现在得到：                                    │
│ 1. 日志模块：src/utils/logger.py               │
│ 2. 主入口接入：src/main.py                     │
│ 3. 接入说明：docs/logger_guide.md              │
│                                                │
│ 验证：                                         │
│ - 示例脚本运行通过                              │
│                                                │
│ 建议：                                         │
│ - 合并前跑完整测试集                            │
│                                                │
│ [查看变更]  [打开说明文档]  [生成提交信息]  [继续优化] │
│                                                │
│ ──────────────────────────────────────────── │
│ ✓ 已完成 6 步  [点击查看执行过程]               │
└────────────────────────────────────────────────┘
```

底部那行"已完成 6 步，点击查看执行过程"是折叠入口。点击后展开完整的阶段历史（4.6 节的阶段合并视图）。

### 9.2 为什么折叠

借鉴 Codex 的设计哲学和人人都是产品经理的观察：

1. **结果感压过日志感**：用户要的是结果，不是执行流水账。完成时把结果放 C 位，过程退到背景。
2. **降低认知负担**：长任务跑了几十步，完成后用户不想再面对一堆步骤卡片，只想看"我得到了什么"。
3. **信任转移**：执行中需要过程可见建立信任（安全带）；完成后信任已建立，结果本身是更好的信任凭证。
4. **符合"大事清楚小事糊涂"**：完成是大事（看结果），执行过程是小事（可查但不默认展示）。

### 9.3 交付物面板的优先级排序

完成态的交付物按用户关心程度排序，不是按执行顺序：

| 优先级 | 内容 | 说明 |
|---|---|---|
| 1 | 用户要的结果 | 文件/文档/图片/表格 |
| 2 | 验证证据 | 测试结果、运行通过 |
| 3 | 风险提示 | 改了认证逻辑建议检查 |
| 4 | 下一步建议 | 合并前跑完整测试 |
| 5 | 执行日志（折叠） | 最后，不作为主角 |

### 9.4 折叠不等于丢弃

过程折叠只是视觉收起，数据完整保留：
- 所有事件在 event_log（04 契约）
- 所有产物在 artifact_store（04 契约）
- 用户随时可"点击查看执行过程"展开
- 开发者视图下默认展开完整 Trace

### 9.5 Code 与 Work 的完成态差异

**Code 完成态**：
```
任务完成
你现在得到：
1. 修改摘要：修复了 auth.ts 的空指针检查
2. 变更文件：src/auth.ts（+3 -1）
验证：12 passed, 0 failed
风险：改了认证逻辑，建议检查会话管理
[查看 Diff]  [生成提交信息]  [回滚]  [继续修改]
```

**Work 完成态**：
```
任务完成
你现在得到：
1. Markdown 报告：docs/report.md
2. 来源：3 个网页 + 1 个文档
关键结论：
- 结论A（来源1,2）
- 结论B（来源3）
不确定项：来源2的数据时效性未确认
[打开报告]  [导出]  [继续润色]  [发送前确认]
```

### 9.6 完成态的回滚入口

完成态必须保留回滚入口（借鉴 Claude Code 的 git rollback 和项目 05 权限规则的快照机制）：
- 用户确认采纳 → 状态沉淀，任务关闭
- 用户不满意 → 可"继续修改"（回到执行态）或"回滚"（恢复快照，回到输入态）
- 回滚前再次确认："回滚将恢复 N 个文件到修改前状态，已生成的新文件会保留。"

## 11. 验收标准

> 对应 06 验收规格，这里补充交互层面的可验证验收点。每个验收点必须当场执行+看完整输出+用实际结果说明。

### 11.1 执行过程展示验收

- [ ] 长任务执行中，阶段进度条正确显示 ✅/🔵/⚪ 三态
- [ ] 当前动作卡同一时间只展示一个动作
- [ ] 连续读 4 个文件合并为"已查看 4 个相关文件"，不刷 4 张卡
- [ ] 工具执行超过 3 秒有进度文字更新，不出现 dead air
- [ ] 阶段切换时展示新阶段意图
- [ ] 只在改文件/执行命令/失败/需确认时展示"原因"

### 11.2 权限交互验收

- [ ] low 风险自动执行，执行后展示"可撤销"
- [ ] high 风险弹确认卡，展示影响+能否回滚+工作目录
- [ ] 权限卡只有 3 个主按钮，"更多授权方式"在二级菜单
- [ ] 删除操作永远强确认，无倒计时
- [ ] 拒绝后模型不重试，返回拒绝消息格式正确
- [ ] 权限文案用"用户影响"表达，不含裸技术参数

### 11.3 完成态验收

- [ ] 任务完成后，过程折叠为一行"已完成 N 步"
- [ ] 中间主区域切换为交付物面板，按优先级排序
- [ ] 交付物面板包含：结果、验证证据、风险提示、下一步、折叠的执行日志
- [ ] 点击"查看执行过程"展开阶段合并历史
- [ ] 完成态保留回滚入口，回滚前二次确认

### 11.4 失败态验收

- [ ] 失败时展示原因+已完成+可选操作
- [ ] 不只展示错误日志
- [ ] 不只给 retry 一个按钮
- [ ] 失败动作展开详情，成功动作保持折叠

### 11.5 压缩体验验收

- [ ] 压缩时展示"正在整理任务上下文"，不说 token/compact
- [ ] 展开后展示保留内容清单
- [ ] 开发者视图才展示 token 用量
- [ ] 检测到压缩循环时提示用户拆分任务

### 11.6 两层体验验收

- [ ] 默认普通视图，隐藏 stdout/stderr/exit code
- [ ] 开发者视图全局开关，一次切换所有任务
- [ ] 普通视图单点展开（查看命令/查看输出）不影响全局
- [ ] 入口文案为"显示技术细节"

### 11.7 实时交互操作验收（V2新增）

- [ ] 执行中输入修正按 Enter，提示"将在当前步骤完成后生效"，当前动作完成后 Agent 读到修正
- [ ] 执行中输入指令按 Tab/发送排入队列，提示"已排队，将在当前任务完成后执行"
- [ ] 点停止按钮立即中断，展示中断损失告知（已产生变更+可选操作）
- [ ] 输入框为空时按 Esc 可编辑上一条消息，编辑后从该点重新执行
- [ ] 计划在执行中被更新时，右侧计划面板高亮变化并提示原因
- [ ] turn 结束（assistant message 展示完）后触发完成折叠，输入框获得焦点
- [ ] 完成态交付物有复制按钮，可复制产物
- [ ] 输入框输入 @ 触发文件模糊搜索，选中后路径注入消息

### 11.8 端到端验收 case

以下 3 个 case 跑通且交互验收点全部满足：

1. **Code-加日志模块**：给真实 Python 项目加日志模块+说明文档，走完计划→执行→验证→完成折叠，产出 diff+测试通过+文档
2. **Work-整理报告**：3 个网页整理成带来源报告，走完执行→压缩→完成折叠，产出报告+来源可点击
3. **失败恢复**：故意制造缺依赖场景，验证失败态给路（安装依赖/跳过/回滚三选项）

## 12. 风险与边界场景

> 对应 AGENTS.md 的对抗式审查要求，对本 PRD 的设计做边界场景排查。

### 12.1 🔴 必须处理

| 场景 | 风险 | 处理 |
|---|---|---|
| 动作合并导致遗漏关键信息 | 用户看不到中间某步改了什么 | 判断点（改文件/命令/删除）不合，始终独立展示 |
| 完成折叠后用户找不到过程 | 审计需求无法满足 | 折叠入口常驻底部，event_log 完整保留，开发者视图默认展开 |
| 压缩循环无限转圈 | 用户以为卡死 | 检测重复压缩，提示拆分任务 |
| 权限文案误导 | 用户误以为可回滚实际不可 | 能否回滚必须如实展示（依赖安装不可回滚要明说） |
| 中断时正在执行命令 | 命令半途中断导致状态不一致 | 中断后明确告知"已停止什么""没有文件被继续修改" |

### 12.2 🟡 建议处理

| 场景 | 建议 |
|---|---|
| 超长任务（100+步）的阶段合并层级 | 一期做单层阶段合并，二期考虑按7步分组的二级折叠 |
| 多任务并行的过程展示 | 一期单任务为主，二期支持任务切换时保持各自进度 |
| 网络断开时的过程展示 | 展示"网络已断开，正在重试"，不影响本地工具执行 |
| 流式输出过长导致卡顿 | 参考Daniel Vaughan的长输出截断，展示层截断+完整存Trace |

### 12.3 🟢 延后处理

| 场景 | 说明 |
|---|---|
| 多Agent协作的过程展示 | 二期，子Agent的过程如何合并展示待设计 |
| Auto Mode下的过程展示 | 二期，Auto模式下过程展示进一步简化 |
