# 小蓝鲸桌面 Agent · 自进化架构优化方案（产品版）

> 读者：产品经理 / 项目负责人 / 研发对齐用  
> 日期：2026-07-12  
> 产品对象：**桌面通用 Agent**（Code / Work， Arbitrary 用户任务），不是车控等垂直工具  
> 技术契约：详见 `17-self-evolving-agent-architecture.md`（L0–L3）  
> 实践参照：车控必过集「离线 Loop / 在线复用 / 元反思」——只借机制，不借业务

---

## 一、先说清楚我们要解决什么问题

### 1.1 用户侧痛点

今天多数 Agent（包括我们当前桌面版）的真实体验是：

- 同一个坑，**明天还会再踩**（不会越用越聪明）  
- 长任务里「再想想」「再检查」变多，**又慢又贵，未必更准**  
- 出了问题，产品很难回答：**系统到底学到了什么？下次会不会还犯？**

### 1.2 产品目标（一句话）

> 让小蓝鲸桌面 Agent 从「每次任务从零试错」变成「**离线把坑变成资产，在线一次就尽量做对**」；  
> 并且能发现「反思/检查」本身是不是在空转，及时关掉或改掉。

### 1.3 两个设计原则（给 PM 的定义）

| 原则 | 白话 | 反面教材 |
|------|------|----------|
| **用一种循环优化另一种循环** | 外面有一层专门「改规则/改经验」的循环，让里面「干活」的循环下次更好 | 只在聊天里让模型多说两句「我再反思一下」 |
| **反思当前的反思机制** | 定期检查：我们的打分、复盘、再试，有没有变蠢、变慢、走过场 | 反思轮数越加越多，分数和体验却不涨 |

## 二、完整架构长什么样（PM 版）

可以把它想成公司里的四级分工，而不是四个平行聊天机器人：

```text
L3  进化策略组（很少开会）
    「我们到底该怎么改进产品？」——改改进流程本身
        ↓ 管
L2  流程改进组（按版本/按周）
    「执行流程、审批、压缩、要不要复查」——改办事章程
        ↓ 管
L1  经验沉淀组（每个任务结束后可跑）
    「这次踩了啥坑、下次怎么用」——写 SOP / 案例库
        ↓ 喂
L0  一线执行（用户每次任务）
    「把这件事做完」——现有桌面 Agent 主循环
```

### 2.1 各层做什么、为什么必须有

#### L0｜任务执行环（一线干活）

| 项目 | 说明 |
|------|------|
| **是什么** | 用户发任务后，Agent 读文件、跑命令、写文档、要审批……直到交付 |
| **对应现状** | 已有：`react` 循环 + 计划/提问/审批/续跑 |
| **必要性** | 没有 L0 就没有产品。进化是为了让 L0 更好，不是取代 L0 |
| **不做什么** | 不在单次任务里擅自改全局安全规则、不默认跑几十轮「自我进化」让用户干等 |

#### L1｜经验沉淀环（把失败变成下次能用的资产）

| 项目 | 说明 |
|------|------|
| **是什么** | 任务结束后（或失败时）：记下坑 → 归类 → 写成可执行经验（触发条件 + 怎么做 + 怎么验收）→ 下次同类任务自动带上 |
| **类比** | 老员工写 SOP，不是再写一篇日记 |
| **必要性** | **解决「用一万次也不变聪明」**。只存聊天记录 ≠ 会复用 |
| **没有它会怎样** | 永远重复试错；客服/用户成本线性涨，能力不涨 |

#### L2｜流程（Harness）进化环 + 元反思

| 项目 | 说明 |
|------|------|
| **是什么** | 看一批任务的过程证据，改进「怎么执行」：提示模板、工具选用策略、审批松紧、要不要复查、上下文怎么压缩、Skill 怎么触发；并定期问：「我们的反思/打分机制是不是失效了？」 |
| **类比** | 改公司流程制度 + 每季度复盘「复盘会有没有用」 |
| **必要性** | 有些问题不是「少一条经验」，而是**流程设计错了**（例如：每次都强制复查 → 又慢又假）。元反思保证进化机器本身不退化 |
| **没有它会怎样** | 经验越堆越乱；或者永远靠加人审、加轮反思硬撑 |

#### L3｜进化蓝图环（可选，后期）

| 项目 | 说明 |
|------|------|
| **是什么** | 很少触发：调整「谁负责改进、怎么验收升级、多久做一次元反思」 |
| **必要性** | 防止 L2 本身僵化；**一期可不做实现，但架构要留口子** |
| **没有它会怎样** | 短期无感；长期可能「改进流程」本身过时却没人改 |

### 2.2 两个运行阶段（产品必须讲清）

| 阶段 | 用户是否感知 | 做什么 | 必要性 |
|------|--------------|--------|--------|
| **在线** | 是 | 读已有经验，尽量一次做对；失败才短修复 | 通用 Agent 的主体验：快、稳 |
| **离线** | 否（后台/实验室） | 对「任务族」批量跑：执行→打分→修经验/流程→回测 | 深度进化放这里，避免用户等 40 轮 |

> 类比车控实践：座椅曾离线迭代几十轮；后来新模块大量「一次通过」。  
> 通用 Agent 同理：**深度迭代离线做，用户侧吃复用红利。**

### 2.2.1 纠正落点分流（不是所有经验都进 Memory）

用户纠正或 Verifier 失败后，先分流再沉淀。原则：**能确定性消灭再犯条件的，升级为 Gate/测试；能流程化的进 Skill 候选；只有偏好与软约束才进 Memory。**

| 纠正类型 | 落点 | 是否立即强制 | 门禁 |
|---------|------|-------------|------|
| 用户偏好 / 风格 | Memory / tactic(`style`) | 否 | `validated=false`，回测后才注入 |
| 项目规范 | tactic(`convention`) + 文档规则 | 半强制（注入优先） | 回测后全量信任 |
| 可确定性拦截（禁止/必须先…） | PreTool Gate / Hook **候选** | 是（落地后） | **不经** Memory 软提醒；本阶段仅记账 |
| 固定流程（每次都要…） | Skill **候选** | 否 | 人工确认后再写 Skill 文件 |
| 可复现失败 | Golden Badcase **候选** | 是（评测） | 进 `eval/golden-t1` 候选清单 |

代码挂钩：`routeLearningTarget` → outcome 记 `learning_target`；Gate/Skill/Golden 写入 `.xiaolanjing/experience/learning_candidates.jsonl`（本轮不自动改 Hook/Skill 文件）。

### 2.2.2 打断策略（何时找用户）

| 情况 | 行为 |
|------|------|
| 可回滚技术问题 | 自动处理 |
| 已有项目规范覆盖 | 自动处理 |
| 多方案但无产品目标差异 | 自主选择并记录 |
| 产品取舍 / 删除 / 权限 / 安全 | 请求用户决策 |
| 连续多次修复无效 | 停止重试并升级 |

实现：`src/agent/src/evolution/interrupt-policy.ts` → `ask_question` 前决策。

### 2.3 一条命门：评估器在环外

| 项目 | 说明 |
|------|------|
| **是什么** | 「做得好不好」由客观信号 + 黄金任务回测说了算，而不是执行模型自己说「我很好」 |
| **信号举例** | 测试/编译是否过、用户是否拒绝、是否乱改禁区、是否无意义打扰审批、相对基线是否更费 |
| **必要性** | 自评很容易「反思剧场」：话说得漂亮，事没做对 |
| **产品口径** | 我们卖的是**可验收的变强**，不是更长的思考气泡 |

## 三、现状：我们已经有什么、缺什么

### 3.1 已有（可保留作地基）

| 能力 | 说明 | 落在哪一层 |
|------|------|------------|
| ReAct 执行循环 | 想→工具→再想，有步数上限与续跑 | L0 |
| 人机卡点 | 计划确认、结构化提问、风险审批 | L0 / HITL |
| Turn/Item 事件 | 过程可展示、可追溯 | Evidence 雏形 |
| 子进程隔离 | 长任务不拖死 UI | 工程底座 |
| 权限分级与黑名单 | 安全底线 | 不可被进化环随便改 |

### 3.2 缺口（相对本方案）

| 缺口 | 产品含义 | 缺了会怎样 |
|------|----------|------------|
| **L1 经验库** | 没有「失败案例 / 缺口模式 / 可注入策略」 | 不会跨任务变聪明 |
| **环外 Verifier + 黄金任务集** | 没有稳定的「好不好」尺子和回测 | 改了不敢发；自评不可信 |
| **ReflectionGovernor** | 没有管「复查/反思」该不该开 | 易出现又慢又假的复查 |
| **离线/在线分工** | 进化和用户任务混在一起 | 要么不敢进化，要么用户体验崩 |
| **L2 流程进化** | 不能系统改进执行章程 | 只能堆提示词和人工救火 |
| **元反思门禁** | 不会检查「改进机制」是否失效 | 长期空转或越改越乱 |
| **上下文压缩 / 循环防空转** | 长任务易撞墙或空转 | 长程任务不可靠（配套能力） |
| **真实暂停/取消** | 控制感不足 | 信任感差（配套） |

## 四、补什么：按优先级的优化路线图

说明：优先级按 **「用户可感知收益 × 风险可控 × 依赖顺序」**，不是按技术炫酷程度。

---

### P0｜先让「变聪明」和「别假忙」成立（建议 1 个大迭代）

#### P0-1 证据落盘（过程资产）

| 项 | 内容 |
|----|------|
| **做什么** | 每次任务完整留下：步骤、工具结果、用户拒绝/接管、是否成功 |
| **必要性** | 没有过程证据，后面所有进化都是拍脑袋 |
| **补完标志** | 任意失败任务能回答「卡在哪一步、用户有没有改过」 |

#### P0-2 环外打分器 + 黄金任务集

| 项 | 内容 |
|----|------|
| **做什么** | 定义若干「任务族」验收方式；维护一套已通过的黄金任务；改规则必须回测 |
| **必要性** | 这是质量门，也是防「修 A 坏 B」 |
| **补完标志** | 改策略后自动跑黄金集；退化则禁止上线 |

#### P0-3 经验沉淀最小版（L1 瘦身版）

| 项 | 内容 |
|----|------|
| **做什么** | `失败案例` + `缺口→修复模板` + 任务结束后可选沉淀；下次同类任务可注入 3～6 条策略 |
| **必要性** | 直接对应「越用越聪明」的最小闭环 |
| **补完标志** | 同一类失败第二次明显减少；有可打开的经验库给产研看 |

#### P0-4 反思治理（关掉默认「假复查」）

| 项 | 内容 |
|----|------|
| **做什么** | 默认不每步复查；仅失败或高风险点才复查；限制复查次数；复查必须有外部证据 |
| **必要性** | 省成本、提速度、反「反思剧场」 |
| **补完标志** | 简单任务更短；复查次数与成功率解耦（复查少了，成功率不掉） |

**P0 做完后的产品效果（可对外讲）：**

- 同类任务**重复踩坑减少**  
- 简单任务**更干净、更快**  
- 产研能打开「我们学到了什么」的账本  
- 改配置有回测，**敢迭代**

---

### P1｜让「在线一次过、离线变强」成为产品形态（建议第 2 个大迭代）

#### P1-1 离线进化 / 在线服务 拆分

| 项 | 内容 |
|----|------|
| **做什么** | 用户会话 = 在线；夜间/实验室对任务族批跑 = 离线 |
| **必要性** | 通用 Agent 不能让用户等「41 轮进化」 |
| **补完标志** | 产品文档写清两阶段；后台有进化任务，前台无感知 |

#### P1-2 任务族经验复用

| 项 | 内容 |
|----|------|
| **做什么** | 按任务族（如「改代码并验证」「多文档报告」「本地文件整理」）隔离规则与经验 |
| **必要性** | 避免「为报告优化的规则搞坏写代码」 |
| **补完标志** | 新同类需求「一轮成功率」上升 |

#### P1-3 长任务配套：压缩 + 防空转

| 项 | 内容 |
|----|------|
| **做什么** | 上下文整理；重复空转检测并提示拆分 |
| **必要性** | 通用 Agent 的长程任务刚需；否则进化也救不了「做一半爆了」 |
| **补完标志** | 长任务完成率↑，无效重复工具调用↓ |

**P1 做完后的产品效果：**

- 宣传点可从「能做任务」升级为「**会积累能力的桌面 Agent**」  
- 内部：新场景接入成本下降（对标垂直项目里后期模块大量一次通过）  
- 用户：失败后重试更「懂行」，而不是从头乱试

---

### P2｜改流程本身（Harness 进化）+ 元反思制度化（建议第 3 个大迭代）

#### P2-1 受限的流程进化

| 项 | 内容 |
|----|------|
| **做什么** | 系统可对「提示策略 / Skill 触发 / 复查开关 / 压缩阈值 / 审批松紧」提出改进候选 → 回测 → 人工确认发布 |
| **必要性** | 有些问题必须改章程，不能只加案例 |
| **边界** | **安全黑名单等红线不自动改** |
| **补完标志** | 有版本化的 Harness 发布记录与回滚 |

#### P2-2 元反思（每 N 个版本强制）

| 项 | 内容 |
|----|------|
| **做什么** | 固定节奏问：打分维够不够？经验库够不够？复查是否空转？回测过没？**必须产出规则变更** |
| **必要性** | 落实「反思反思机制」；防止进化机器自己糊弄自己 |
| **补完标志** | 每个版本周期有元反思记录 + 至少一项规则更新或明确「无需变更」的证据 |

**P2 做完后的产品效果：**

- 流程可演进且可控（有灰度、有回滚）  
- 能回答投资人/管理层：**能力增长是否可审计**  
- 复查/审批策略可随模型变强而「减负」，而不是永远绑死

---

### P3｜可选增强（不挡主线）

| 项 | 说明 | 效果 |
|----|------|------|
| 模型微调通道 | 把高置信经验做成训练样本（后期） | 能力进权重，成本高 |
| L3 进化蓝图自动搜索 | 自动改「怎么进化」 | 极高阶，需 P2 极稳后 |
| 跨端经验共享 | 桌面 ↔ 手机同一经验 schema | 生态效应 |

## 六、按这个做了之后，能实现什么样的效果

### 6.1 对用户（桌面通用 Agent）

| 效果 | 表现 |
|------|------|
| **更稳** | 同类错误明显减少；失败后重试更「在点上」 |
| **更快** | 简单任务少无效复查；少重复读同一批文件 |
| **更敢交长任务** | 长任务不易无声卡死；可提示拆分而非空转 |
| **更可控** | 审批仍在；危险规则不会被「自动进化」改掉 |
| **体感** | 「它好像记得我项目里怎么做才对」，而不是每次像新员工 |

### 6.2 对产品与经营

| 效果 | 表现 |
|------|------|
| **差异化叙事** | 从「又一个聊天装工具的 Agent」→「会积累工作方法的桌面 Agent」 |
| **可量化改进** | 任务族一次成功率、重试率、复查次数、黄金集回归率 |
| **规模化** | 新场景主要靠迁移经验，而不是每次从零堆提示词 |
| **风险可控** | 进化有门禁、回测、回滚；安全红线人工锁死 |

### 6.3 建议的北极星与过程指标

| 类型 | 指标 | 期望方向 |
|------|------|----------|
| 北极星 | 核心任务族「一次交付成功率」 | ↑ |
| 北极星（并列） | HITL 负担（审批/提问/续跑） | ↓ |
| 北极星（并列） | 重复错误率 | ↓ |
| 过程 | 同族任务平均重试次数 | ↓ |
| 过程 | 每任务 Critic/复查 token 占比 | ↓（或不升反升成功率） |
| 质量 | 黄金集回归失败次数 | → 0（发布门禁） |
| 资产 | 可执行经验条数 / 被命中次数 | ↑ 且命中有效 |
| 健康 | 元反思按期完成率 | = 100% |

### 6.4 明确承诺边界（避免过度承诺）

本方案**不承诺**：

- 任意长尾说法/任意奇葩任务 100% 一次成功  
- 无客观信号场景下「全自动变强」（Work 场景要更多靠用户确认与产物校验）  
- 让模型在用户会话里随意改安全策略  

本方案**承诺追求**：

- 有任务族的场景下，**重复劳动与重复踩坑持续下降**  
- 进化过程**可看见、可回测、可回滚**  
- 反思机制**可被关掉或改掉**，而不是越加越重

## 七、给产品经理的评审清单（开方案会用）

开会时逐条问：

1. 用户变强的感知点是什么？（一次成功率 / 少踩坑 / 更快）  
2. 进化发生在前台还是后台？（必须能答：默认后台）  
3. 「好不好」谁说了算？（不能是模型自拍）  
4. 改坏了怎么办？（黄金集 + 回滚）  
5. 安全红线谁能动？（进化环不能动）  
6. 复查凭什么开？（有证据才开）  
7. 我们多久检查一次「改进机制」本身？（元反思节奏）  
8. 第一个任务族选哪个做样板？（建议：Code「改代码+跑验证」信号最硬）  

---

## 八、推荐的第一个样板场景（降低落地难度）

| 选项 | 为什么适合做第一个任务族 |
|------|--------------------------|
| **Code：小改动 + 跑测试/编译** | 客观信号强（exit code），最像车控里「能打分」的环境 |
| Work：多源报告 | 信号偏软，适合 P1 再上，需加强「来源/结构/用户确认」验收 |

**建议：P0 用 Code 任务族打穿闭环，再复制到 Work。**

## 十、一页结论

1. **架构**：L0 干活 → L1 沉淀经验 → L2 改流程并元反思 → L3 后期才改「怎么进化」。  
2. **形态**：离线变强，在线一次过。  
3. **命门**：环外打分 + 黄金回测；反思要治理。  
4. **现状**：执行环和人机卡点已有；**经验、尺子、治理、离线在线、流程进化**是主缺口。  
5. **优先级**：P0 证据+尺子+经验+反思治理 → P1 形态与长任务 → P2 改流程与元反思。  
6. **效果**：用户更稳更快；产品可量化、可审计地「变聪明」；不靠反思剧场硬撑。
