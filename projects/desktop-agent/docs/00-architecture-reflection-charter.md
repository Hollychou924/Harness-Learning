# 小蓝鲸桌面端 · 架构反思约定

> 状态：元规则文档，贯穿全程（2026-06-30）
> 性质：这份不是设计文档，是"怎么保证架构始终最优"的机制。每产出一份设计文档、每完成一个里程碑，都按此机制回检。

---

## 一、目标定义（先把"最优"讲清，不空喊）

"任何时候架构都是最优秀的"——这个目标的严谨表述是：

- **基于当前已确认的需求和已知信息，架构决策是最优的。**
- 不追求超越未来需求的"绝对最优"——那是过度设计，本身就不优秀。
- 信息不全时，不赌未来，但**显式标注扩展点**（哪里可能变、变了要动什么）。
- 发现新信息或需求变化时，**主动回头评估**是否冲击现有架构，该改就改。

这个定义本身符合优秀架构原则：YAGNI（不做用不上的）、为已知演进留口子（不为想象的需求留口子）。

---

## 二、反思的触发时机（什么时候必须回检）

不靠"记得反思"，靠固定触发点：

| 触发点 | 动作 |
|---|---|
| 每产出一份设计文档后 | 检查它是否与已锁定的 3 条核心决策、已有文档冲突；是否引入了不必要的复杂度 |
| 每完成一个编码里程碑后 | 检查实现是否偏离设计；是否出现了设计没预料的情况 |
| 需求或外部信息变化时 | 评估是否冲击现有架构（如跨端绑定方式调研出结果，会冲击数据契约的账号表） |
| 发现反例或踩坑时 | 回溯是设计问题还是实现问题，从根因改 |
| 每 2-3 个里程碑 | 一次架构全局体检（对抗式审查） |

---

## 三、反思的维度（按什么标准判断"合理"）

每次反思固定过这 6 个维度（源自优秀架构原则）：

1. **职责单一**：每个模块/层只做一件事，没有越界。
2. **依赖方向**：依赖指向稳定方向，不出现循环依赖、不反向依赖（UI 不该直接碰 DB 细节）。
3. **可替换性**：关键依赖（模型 provider、向量库、存储）是否可通过接口替换，不绑死实现。
4. **不过度设计**：有没有为"可能要用"的复杂度提前建设。已知演进留口子≠为想象需求建架构。
5. **不过度简化**：根因需要的复杂度有没有被为了"简单"而砍掉（如安全闸不能为了快而省）。
6. **可演进性**：现在的设计能否在不推倒重来的前提下加新能力（如跨端、新场景 Tab）。

---

## 四、反思的输出（不留空反思）

每次反思必须落到文档，不留"我觉得还行"的空话：

- ✅ 通过的维度：一句话说明为什么通过。
- ⚠️ 待办：写明什么问题、在哪个文档改、计划何时改。
- ❌ 冲突：立即修订对应文档，标注修订原因。

反思记录追加到本文件下方「反思日志」章节，按时间倒序。

---

## 五、约束：反思不变成"推倒重来"

- 反思的目的是持续校准，不是反复重造。
- 已锁定决策（见 01 设计基线）的改动门槛高：必须有根因性的新信息，否则不轻易推翻。
- 发现问题优先"在根源处最小修复"，不借反思之名做大重构。
- 这条与设计基线"根因决定改动范围"一致。

## 六、反思日志（按时间倒序追加）

### 2026-06-30 · 初次架构检视

**背景**：地基文档（01 设计基线、02 桥接协议）产出后，按本机制首次反思。

**维度逐项**：
1. 职责单一：✅ 三层（主进程/渲染/Agent子进程）职责清晰，Agent 引擎独立进程隔离高风险操作。
2. 依赖方向：✅ 渲染→主进程→Agent 子进程，单向；Agent 不反向依赖 UI。
3. 可替换性：⚠️ 模型 provider 可替换（参考已做多 provider），但向量库和 SQLite 绑定库尚未在选型文档确定——待第 3 份文档明确，确保通过接口隔离。
4. 不过度设计：✅ 跨端推迟、V2 路由推迟、多 Agent 推迟，均未提前建设。
5. 不过度简化：✅ 安全双闸（白名单+黑名单）未省，记忆四层未省。
6. 可演进性：✅ HubClient 预留接口位、设备类型开放 string、Profile 配置化可扩新 Tab。

**待办**：
- ⚠️ 向量库和存储的接口隔离，在第 3 份技术选型文档中强制要求"通过接口抽象，不绑死实现"。
- ⚠️ 跨端绑定方式调研出结果后，回头评估是否冲击 02 桥接协议和 04 数据契约的账号/会话表设计——此项单独追踪，不阻塞一期编码。

**结论**：当前架构在已知需求下合理，2 项待办在第 3 份文档和跨端调研中收口。

### 2026-06-30 · 文档体系完成后的架构反思

**背景**：7 份开工前文档（00-07）全部产出后，按本机制做全局检视。

**维度逐项**：
1. 职责单一：✅ 文档分层清晰——00元规则/01设计基线/02跨端(存档)/03选型/04数据/05权限/06验收/07评测，无交叉重复。
2. 依赖方向：✅ 文档依赖单向：01←(02,03)←04←(05,06)←07，无循环。03选型是04数据契约的地基，04是05/06的地基，06是07的地基。
3. 可替换性：✅ 03文档第五章强制 LlmProvider/VectorStore/TaskStore/HubClient/SecretStore 五个接口抽象，回应了首次反思的待办。
4. 不过度设计：✅ 跨端推迟、PPT推迟、多Agent推迟、Auto Mode推迟、A/B推迟——均为"可能要用"的复杂度未提前建设。
5. 不过度简化：✅ 安全双闸未省、记忆四层未省、评测三支柱(集+标准+决策)未省、审计留痕未省。
6. 可演进性：✅ 跨端接口位预留、设备类型开放string、Profile配置化可扩Tab、评测集可增长、向量库可替换。

**首次反思待办收口情况**：
- ✅ 向量库/存储接口隔离：03文档第五章已强制，收口。
- ⏸️ 跨端调研冲击数据契约账号表：02文档已标记存档待调研，04文档显式标注"跨端相关表暂缺待调研"，追踪中不阻塞一期。

**新发现的待办**：
- ⚠️ AgentConfig 里 embedding 配置仍含 api_base_url/model 字段（03文档沿用参考），但 key 经 SecretStore。需在编码时确认 embedding key 也走 SecretStore 不落 config——已在03文档3.2节标注，编码时落实。
- ⚠️ 状态机的"如何触发转移"属代码层，04文档只定值和规则。编码时需在 TaskStore 实现转移逻辑，并保证终态不可回退——这是实现约束，非设计缺口。

**结论**：7 份文档在已知需求下构成完整且自洽的开工契约，2 项首次待办已收口1项、1项持续追踪，2 项新待办为编码层约束非设计缺口。架构就绪，可进入编码阶段。

### 2026-06-30 · 补全 UI 与通信契约后的反思

**背景**：用户问"是不是还缺 UI 示意"，反思后补了 08 UI线框 + 09 Electron三层通信契约。9 份开工前文档齐全，再次全局检视。

**维度逐项**：
1. 职责单一：✅ 08 只管 UI 形态，09 只管通信，不交叉。09 的消息 type 与 04 的 AgentEvent 枚举对应但不重复定义。
2. 依赖方向：✅ 三层通信单向：渲染→主进程→Agent，反向靠事件推送，无循环。Agent 不反向依赖渲染/主进程。
3. 可替换性：✅ Agent 子进程纯 stdio 通信，不绑 Electron——可未来脱离 Electron 测试/复用（回应架构反思可演进性）。
4. 不过度设计：✅ 跨端消息流转标注"待调研后补"，IM/定时任务次要通道二期补。
5. 不过度简化：✅ 凭证不进渲染、contextBridge 隔离、stdin 非阻塞约束都写了，安全闸没省。
6. 可演进性：✅ IPC 通道表可扩展加新命令，stdio 协议 type 可扩展，不破坏现有。

**收口的待办**：
- ✅ "Agent↔UI 事件流映射"半缺口：09 文档第三章把 AgentEvent 映射到 stdio type 再到 IPC 事件，闭环了。
- ✅ UI 缺口：08 文档三态+双工作台+审批弹窗+流转图，AI 搭前端不再编布局。

**新待办（编码层约束，非设计缺口）**：
- ⚠️ 09 文档 2.1 的 `secret:get` 设计为"不返回明文给渲染"，编码时必须落实：主进程取 key 后注入 Agent config，渲染侧只显示布尔状态。
- ⚠️ 08 文档组件库未定（shadcn/ui 还是自建），编码时定，不阻塞。

**结论**：9 份文档构成完整自洽的开工契约，覆盖 设计基线/跨端/选型/数据/权限/验收/评测/UI/通信 九大维度。架构反思首次 2 项待办全收口，跨端调研持续追踪。开工前文档就绪，可进入编码。
