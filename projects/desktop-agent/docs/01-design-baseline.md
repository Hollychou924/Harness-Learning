## 二、桌面壳：Electron + TypeScript

### 2.1 架构分层（三层全 TS，无语言边界）

```
┌─────────────────────────────────────────────┐
│  主进程（Electron Main，Node.js）            │
│  - 窗口/托盘/全局快捷键/自动更新/三端打包     │
│  - 配置持久化、定时任务调度                   │
│  - 子进程管理（Agent 引擎以独立进程运行）     │
└──────────────────┬──────────────────────────┘
                   │ IPC（contextBridge）
┌──────────────────▼──────────────────────────┐
│  渲染进程（React + TS，WebView）             │
│  - Code/Work 双 Tab、任务列表、产物预览      │
│  - 工具日志、权限确认弹窗、记忆引用面板       │
└─────────────────────────────────────────────┘
                   │ stdio / JSON Lines
┌──────────────────▼──────────────────────────┐
│  Agent 引擎（Node.js 子进程，TS）            │
│  - 7 步闭环 + ReAct 引擎                     │
│  - 工具运行时、MCP 客户端、记忆四层          │
│  - 跨端 Hub 客户端（能力位，一期不实现）      │
└─────────────────────────────────────────────┘
```

**为什么 Agent 引擎跑成子进程而非主进程内**：Agent 跑高风险操作（Shell/Git/删文件）和长任务，独立进程能做到崩溃不影响 UI、可单独重启、资源隔离。此设计参考仓库已验证可行（Tauri 那套也是 Node 子进程跑 Agent）。

| Mac | universal（arm64+x64）.dmg | electron-builder |
| Windows | NSIS 安装包 .exe | electron-builder |

## 三、Agent 循环：7 步闭环 + 内嵌 ReAct

### 3.1 两层关系（最容易混，必须分清）

- **7 步闭环**（小蓝鲸文档第 5 章）是**任务级宏观编排**：目标契约→计划图→能力绑定→工具执行→质量校验→产物预览→状态沉淀。
- **ReAct**（想-做-看-再想）是**工具执行步内的微观机制**，只跑在 7 步的第 4 步「工具执行」内部。

它俩不是二选一，是包含关系。AI 干活时禁止把两者当替代关系。

### 3.2 7 步闭环（宏观骨架，照小蓝鲸文档）

| 步骤 | 通用职责 | 一期实现范围 |
|---|---|---|
| 1 目标契约 | 明确目标/约束/交付物/验收标准 | 一期全做 |
| 2 计划图（DAG） | 拆成可执行任务图 | 一期做单链路计划，复杂 DAG 二期 |
| 3 能力绑定 | 绑定需要的工具和 Skills | 一期做 Profile 默认绑定 + Top-K |
| 4 工具执行 | 执行工具并记录事件 | 一期全做（内嵌 ReAct） |
| 5 质量校验 | 校验和自愈 | 一期做一级自愈（格式语法），二三四级逐步 |
| 6 产物预览 | 交付可审查产物 | 一期全做 |
| 7 状态沉淀 | 状态持久化和评测回流 | 一期做状态持久化，评测回流二期 |

### 3.3 ReAct 引擎（微观机制，依据参考已验证实现）

参考仓库 `agent-xiaoai/agent.ts` 已跑通，照其设计实现，要点：

- **迭代上限**：`maxIterations` 硬上限，到顶兜底收尾，禁止无限转。
- **空回复重试**：工具执行后若 LLM 返回空，重试一次再放弃（防卡死）。
- **工具后报错兜底**：工具已执行但生成最终回复时出错，给用户一个合成收尾（已执行的工具有结果，不让对话悬空）。
- **流式产出事件**：每步产出 chunk/tool_call/tool_result/usage 事件，UI 实时渲染。

**与手机端 `AgentRuntime.kt` 的对应**：手机端的 Goal-Plan-Act-Review 循环结构同源，Review 收敛 = ReAct 的「看」步骤。两套实现思路一致，可互为验证。

### 3.4 自愈分级（小蓝鲸文档第 5.2 节，分阶段实现）

| 层级 | 触发 | 动作 | 一期 |
|---|---|---|---|
| 一级 | 格式/语法/排版 | 原地自动修复重校验 | ✅ |
| 二级 | 逻辑/功能/测试失败 | 分析原因重执行节点，有限重试 | 二期 |
| 三级 | 多次失败/跑偏 | 回滚稳定态，局部重规划 | 二期 |
| 四级 | 工具不可用/权限不足 | 说明原因和需用户介入点 | ✅（兜底必须有） |

## 四、记忆：四层混合（全上，不分期）

依据：参考仓库 `memory/manager.ts` 已实现文件式+向量+压缩；小蓝鲸文档第 11 章定义 DB 层。融合如下。

### 4.1 四层职责

| 层 | 形态 | 职责 | 实现依据 |
|---|---|---|---|
| 工作记忆层 | `MEMORY.md` 风格，**只这一份自动进上下文** | 当前任务带着什么 | 参考已验证 |
| 长期结构化层 | SQLite | 崩溃恢复、任务追溯、状态持久 | 小蓝鲸文档第 11 章 |
| 语义召回层 | 向量索引（挂长期层之上） | 相关性检索 | 参考已验证（embedding+cosine） |
| 摘要压缩层 | 摘要文件 + `compactMessages` | 防上下文无限膨胀 | 参考已验证 |

### 4.2 关键规则（依据参考已验证实现，不可省）

- **只有 MEMORY.md 自动注入上下文**，其余记忆一律靠工具按需检索。这是防止上下文爆掉的核心，参考仓库明确这么做的。
- **长会话必须有摘要压缩**：超过阈值用 LLM 生成摘要，原对话归档。参考的 `compactMessages` 直接照搬思路。
- **语义召回 hybrid**：embedding 语义 + 关键词匹配双路，参考已验证比单路准。

### 4.3 SQLite 表（依据小蓝鲸文档第 11 章，字段精确版见"数据与接口契约文档"）

task_store / event_log / workspace_index / memory_store / artifact_store / snapshot_store / skill_registry / secret_store。

## 五、工具安全：双闸（依据参考已验证实现）

参考仓库 `tools/index.ts` 的安全策略已跑通，照其设计，要点：

### 5.1 白名单免审批（SAFE_TOOLS）

只读操作免审批：read_file / list_files / memory_search / memory_get / browser_snapshot / browser_take_screenshot / list_scheduled_tasks 等。
其余工具一律需审批。

### 5.2 黑名单硬拦截（BLOCKED_PATTERNS）

危险命令直接拒绝，不进审批：
- `rm -rf /` 系列
- `mkfs` / `dd of=/dev/` / `chmod 777 /` / `shutdown` / `reboot`
- `curl/wget | sh` 管道执行远程脚本
- `sudo rm` / `python -c import os; system` 提权破坏
- fork 炸弹

### 5.3 工作目录边界

Shell 执行目录必须在用户主目录下，超出直接拒。

### 5.4 与小蓝鲸权限体系的关系

参考的双闸是「一期落地版」；小蓝鲸文档第九章的三种权限模式（Safe/Balanced/Auto）+ 风险四级是「目标态」。一期先上双闸 + Safe/Balanced 两种模式，Auto Mode 二期。
