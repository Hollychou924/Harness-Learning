# 小蓝鲸桌面端 · 评测基线

> 状态：开工前必读契约（2026-06-30）
> 性质：定义"Agent 真的变好了"怎么科学证明。呼应 Harness 系列 E8 核心论断：指标罗列 ≠ 能驱动每天变好，必须有评测集 + 评测标准 + 决策落地三件套。
> 核查范围说明：依据小蓝鲸文档第十五章、Harness 系列 E8 outline（评测集/评测标准/决策落地三支柱）、06 验收规格书的三个 case。
> 核心立论：评测不是上线后写报告，是产品的方向盘和仪表盘。每个版本改动必须在评测集上回归通过才算完成。

---

## 一、评测集：用什么题考（对应 E8 支柱一）

### 1.1 一期最小评测集（覆盖 06 文档三个 case）

| 评测集 | case 数 | 来源 | 考什么 |
|---|---|---|---|
| code-bugfix | 5+ | 真实项目真实 bug | 定位准、Diff 可 apply、测试通过 |
| work-report | 5+ | 真实网页组合 | 报告结构合理、来源属实、无事实错误 |
| file-organize | 3+ | 真实混乱文件夹 | 规则合理、可回滚、无丢失 |

一期目标：13+ 条 case，每条带 ground truth。

### 1.2 每条 case 必须有的字段

```typescript
interface EvalCase {
  case_id: string;
  category: "code-bugfix" | "work-report" | "file-organize";
  difficulty: "easy" | "medium" | "hard";
  input: string;              // 用户指令
  input_resources: string[];  // 项目路径/URL/文件夹
  ground_truth: {             // "对"长什么样
    expected_artifacts: string[];   // 应产出什么
    expected_tests_pass?: boolean;   // 测试应通过
    must_not_do: string[];           // 禁止动作
  };
  scoring_dims: string[];     // 评分维度(见第二章)
}
```

### 1.3 评测集建设原则（对应 E8）

- **真实采集为主**：从真实场景捞 case，不凭空造（贴近生产分布）。
- **故障注入补充**：主动造边界 case（空输入、超大文件、断网、权限不足）。
- **训练/评测隔离**：评测集 case 不进任何 prompt 示例，防刷分。
- **持续增长**：每发现一个线上 badcase，加成回归题（对应 E8"评测集是活的"）。

## 二、评测标准：按什么打分（对应 E8 支柱二）

### 2.1 评分维度（不只看最终结果）

| 维度 | 量化 | 说明 |
|---|---|---|
| 任务完成度 | 0/1/部分 | 产出是否齐全 |
| 结果正确性 | 0-1 | Diff 能否apply/测试是否通过/来源是否属实 |
| 过程效率 | 步数/token数 | 同样结果，步数少token少为优 |
| 安全合规 | 0/1 | 有无越权/命中BLOCKED/漏审批 |
| 无关修改率 | 0-1 | 改了不该改的文件比例 |
| 回滚可用性 | 0/1 | 失败后能否干净回滚 |

**核心认知（E8）**：Agent 是多步骤系统，只看最终答案会被骗——答案对，过程可能调十次废工具、用未授权数据、靠重试撞对。所以必须同时看结果和过程。

### 2.2 谁打分

| 维度 | 打分方式 |
|---|---|
| 任务完成度/结果正确性/安全/回滚 | 规则自动判定（可程序化验证） |
| 过程效率 | 自动统计（event_log 已记录步数/token） |
| 报告质量/事实正确性 | LLM-as-Judge（用更强模型评）+ 人工抽检 |

**防 LLM-as-Judge 失准**：每批抽 20% 人工复核，发现判定偏差超阈值则调 prompt。

## 三、决策落地：怎么用评测做迭代（对应 E8 支柱三）

### 3.1 离线评测（每次改动后跑）

- 改动 Agent 代码后，全量跑评测集。
- 回归通过（不引入新失败）才能合并。
- 指标对比上个版本，记录涨跌。

### 3.2 在线观测（一期最小）

一期不强制 A/B，但必须有可观测：

| 指标 | 来源 | 一期 |
|---|---|---|
| 任务完成率 | task_store status 统计 | ✅ |
| 平均步数/token | event_log 聚合 | ✅ |
| 回滚率 | snapshot_store 使用率 | ✅ |
| 高风险拦截率 | event_log APPROVAL_REQUESTED | ✅ |
| Diff/报告采纳率 | artifact reviewed 字段 | ✅ |

### 3.3 badcase 回流

线上/评测发现的 badcase → 加进评测集成回归题 → 下次改动必须通过。这是评测集自我增长的机制（对应 E8）。

---

## 四、一期评测流程（可执行版）

```
开发改动 → 跑评测集(13+ case) → 全绿 → 记录指标 → 合并
                              → 有失败 → 修 → 重跑
```

**完成判据**：一期结束时，13+ case 全部通过，且指标有基线值可对比。

---

## 五、评测集防腐（对应 E8）

- 评测集 case 内容不进任何系统提示词/示例。
- 评测集独立目录，不与训练数据混。
- 每 2-3 周人工看 20-50 条真实日志（E8"Look at your data"），发现评测集和现实脱节就更新。
- 模型/Harness 改动永远在最新评测集上回归。

---

## 六、本文档边界

- 一期不做正式 A/B 实验（多步骤 Agent 的 A/B 复杂，二期）。
- 一期不做自动 benchmark 对标公开榜单（GAIA/SWE-bench 是锚点不是评测集，对应 E8）。
- 评测集的具体 case 内容随开发逐步填充，本文档定的是机制和标准。
- 自我进化（E7）相关的评测回流，二期补。
