# DeepSeek Agent Harness 指标体系：如何证明 Agent 真的帮到了用户

> 目标岗位：DeepSeek Agent Harness 产品经理  
> 核心问题：如何定义与衡量“Agent 是否真的在更多场景下更深入地帮助到更多的人”

## 1. JD 原句拆解

DeepSeek Agent Harness 岗位里最关键的一句话是：

> 定义与衡量“Agent 是否真的在更多场景下更深入地帮助到更多的人”的指标。

这句话不能简单理解成“回答准确率”。Agent 的价值不是答对一句话，而是把用户交给它的任务推进到可用结果。

| JD 关键词 | 产品问题 | 指标方向 |
|---|---|---|
| 真的帮助 | 用户的任务有没有被完成，结果有没有被采用 | 任务完成率、结果采纳率、返工率 |
| 更多场景 | 能覆盖多少任务类型，不同难度是否都能稳定完成 | 场景覆盖率、分场景成功率、长尾任务成功率 |
| 更深入 | 是浅层问答，还是能推进多步骤复杂任务 | 任务深度、工具使用质量、连续执行能力 |
| 更多人 | 新手、普通用户、专业用户是否都能用起来 | 新用户首次成功率、留存、复用率 |

因此，Agent Harness 的指标对象应该从“回答”升级为“任务”。
## 1.5 更重要的判断：评测即 PRD

在定义具体指标之前，先说一个更根本的判断：对 Agent 产品来说，**评测体系本身就是 PRD**。

传统 PM 是“先写需求，再做开发”。Agent PM 是“先实验、再评测、从评测中提炼需求”。因为 Agent 的输出是随机的、动态的、场景开放的，任何静态文档都无法覆盖所有情况。评测体系——自动化测试、黄金对话、评分标准——构成了一个可运行、可验证、可演化的“活的 PRD”。

这对 DeepSeek Agent Harness PM 的直接含义是：JD 里说的“定义与衡量”，不是先做产品再补评测，而是评测体系本身就是产品定义的一部分。你定义的评分标准，就是你对“什么算做好”的定义；你收集的失败样例，就是你对产品边界的定义。


## 2. 北极星指标

建议的北极星指标是：

> 用户认可的有效任务完成率。

它包含三个约束：

1. 用户发起的是真实任务，不是演示样例。
2. Agent 在合理时间和成本内完成关键步骤。
3. 用户认可结果可用，包括采纳、确认、合并、发布或继续基于结果推进。

这个指标比“调用次数”“对话轮数”“回答准确率”更接近真实产品价值。Agent 自己说完成了不算，用户认可、外部状态变化或产物被采用才算。

为什么用“率”而不是“数”：用户量变大，完成数自然上涨，但产品能力可能没变好甚至变差。比如上周 100 个任务完成 70 个，完成率 70%；这周 1000 个任务完成 500 个，完成数涨了，完成率却只有 50%。只看“数”会误判产品变好了，看“率”才知道体验退步了。“完成数”作为辅助指标看规模，“完成率”作为北极星看质量。

## 2.5 最小可用指标集：从日志自动计算

在完整六层体系之前，先有一套能立刻落地的最小指标集。核心原则是：**所有指标全部从结构化日志自动计算，不依赖人工标注和用户反馈**。

| 指标 | 从日志怎么算 | 回答什么问题 |
|---|---|---|
| 任务完成状态 | 从日志的任务状态字段统计：正常结束、超时、步数达上限、异常退出 | 事情做成了没有 |
| 工具调用数量 | 从工具调用记录汇总：调用总次数、工具种类数、重试次数 | 过程是否合理 |
| 任务耗时 | 从时间戳计算端到端时间，统计 P50/P90 | 等待体验 |
| Agent 循环步数 | 从主循环迭代记录统计，统一“一步”定义（think→tool→observe 算 1 步） | 有没有死循环或冗余思考 |
| Token 消耗 | 从每轮 LLM 调用日志汇总 prompt/completion/total，区分上下文组成 | 成本是否可控 |

这套最小集的价值是：先把日志打全、打规范、打结构化，所有指标都能自动算出来，不需要等用户反馈，也不需要人工标注。它天然支持版本对比、A/B 测试和发版回归。完整六层体系是在这套地基上往产品价值方向延伸。

## 2.6 Harness 工程质量的三层标准化指标

除了产品价值指标，Agent Harness 还需要一套专门衡量“Harness 工程本身好不好”的指标。参考学术界和工业界的标准化建议，分成三层：

| 层级 | 指标 | 用途 |
|---|---|---|
| Tier 1 必报 | 单次成功任务 Token 消耗、单次成功任务成本、平均轮数、pass@1 准确率 | 对外可比、对外汇报 |
| Tier 2 推荐 | Token 消耗方差、失败任务 Token 浪费、输入/输出 Token 比、产物大小中位数 | 成本诊断 |
| Tier 3 深度 | 工具调用冗余率、死循环率、自纠错成功率、上下文利用率 | 内部优化 |

这套分层和前面六层的区别：六层是“产品好不好”的产品指标；这套三层是“Harness 工程好不好”的工程指标。一个回答用户有没有被帮到，一个回答底层系统跑得健不健康。两者要一起看。

## 3. 六层指标体系

### 3.1 任务结果指标：事情有没有做成

这一层回答“Agent 有没有真的帮用户完成目标”。

| 指标 | 含义 | 产品判断 |
|---|---|---|
| 任务完成率 | 用户委托的任务中，最终完成的比例 | 核心可用性 |
| 结果采纳率 | 用户是否使用、合并、发布或继续基于结果推进 | 结果是否真的有价值 |
| 任务放弃率 | 用户中途关闭、停止、换工具或不再继续 | 任务链路是否过长、过难、失控 |
| 返工率 | 用户拿到结果后需要重做或大改的比例 | 结果质量是否稳定 |
| 首次成功率 | 用户第一次表达需求后，Agent 是否理解并推进正确方向 | 使用门槛是否足够低 |
| 分难度任务完成率 | 按简单、中等、复杂三档分别统计完成率 | 避免被简单任务拉高的总分掩盖短板 |

关键判断：不要只看 Agent 有没有回复，要看用户有没有用上结果。

### 3.2 过程质量指标：它是怎么做成的

这一层回答“虽然结果完成了，但过程是否合理”。

| 指标 | 含义 | 产品判断 |
|---|---|---|
| 计划合理率 | Agent 拆任务的步骤是否完整、清晰、顺序合理 | 是否具备复杂任务规划能力 |
| 计划遵循率 | 实际执行是否沿着计划推进，还是中途漂移 | 是否容易跑偏 |
| 步骤浪费率 | 重复调用、无效搜索、反复试错等浪费动作比例 | 成本和体验是否可控 |
| 工具选择正确率 | 是否选对了完成任务所需的工具 | Harness 工具设计是否有效 |
| 工具参数正确率 | 工具调用时填写的信息是否正确 | 是否能可靠执行动作 |
| 错误恢复率 | 遇到失败后是否能解释、修正、降级或请求用户确认 | 是否具备自我纠偏能力 |

关键判断：Agent 评估不能只看最终结果，还要看过程记录。一个用 3 步完成任务的 Agent 和一个绕 20 步才碰巧完成的 Agent，产品体验和成本完全不同。

### 3.3 稳定性指标：这次能做，下次还能不能做

这一层回答“Agent 是否可靠，而不是偶然成功”。

| 指标 | 含义 | 产品判断 |
|---|---|---|
| 多次任务稳定率 | 同类任务多次执行是否都能完成 | 面向用户的可靠性 |
| 老任务退步率 | 新版本上线后，以前能做好的任务是否变差 | 是否破坏已有能力 |
| 复杂任务稳定率 | 多步骤、长时间、跨工具任务是否能持续完成 | 是否具备深度帮助能力 |
| 异常场景恢复率 | 网络失败、工具失败、用户打断后是否能继续 | 是否适合真实环境 |

这里要区分两类评估：

- 能力评估：看 Agent 能不能完成更难的新任务。
- 回归评估：看 Agent 是否还能稳定完成以前已经做好的任务。

关键判断：能力提升和稳定守住要同时看。只追求新能力，可能会把已经稳定的体验改坏。

### 3.4 用户掌控指标：用户敢不敢把任务交给它

这一层回答“用户是否信任 Agent，是否能在关键时刻掌控过程”。

| 指标 | 含义 | 产品判断 |
|---|---|---|
| 用户打回率 | 用户认为方向不对，要求重来或改方向的比例 | 任务理解是否偏航 |
| 人工接管率 | Agent 做到一半需要用户亲自接手的比例 | 自动化边界是否合理 |
| 确认点通过率 | Agent 请求放行时，用户是否愿意继续 | 用户信任是否建立 |
| 过程可理解评分 | 用户是否看得懂 Agent 正在做什么、为什么这么做 | 过程透明度是否足够 |
| 高风险动作拦截率 | 删除、发送、支付、改权限等动作是否被正确拦截 | 安全边界是否可靠 |

关键判断：Agent 产品不是越自动越好，而是要让用户知道它在做什么、能不能插手、哪里需要确认。

### 3.5 成本效率指标：产品能不能长期跑

这一层回答“Agent 的效果是否能在合理成本内规模化”。

| 指标 | 含义 | 产品判断 |
|---|---|---|
| 单任务耗时 | 从任务发起到结果交付的时间 | 等待体验是否可接受 |
| 单任务成本 | 完成一个任务消耗的模型、工具和检索成本 | 商业上是否可持续 |
| 重试次数 | 同一任务中重复失败和重来的次数 | 稳定性和浪费程度 |
| 等待时长 | 用户无反馈等待的时间 | 体验焦虑是否过高 |
| 单位成本完成率 | 同样成本下完成多少有效任务 | 效率是否变好 |

关键判断：高质量但无限贵、无限慢的 Agent 不能规模化。成本是产品约束，不是后台小问题。

### 3.6 规模化指标：是否帮助更多人和更多场景

这一层回答“Agent 是否从少数强用户扩展到更多真实用户”。

| 指标 | 含义 | 产品判断 |
|---|---|---|
| 场景覆盖率 | 核心任务类型覆盖了多少 | 是否进入更多使用场景 |
| 分场景成功率 | 不同任务类型分别完成得如何 | 是否被总分掩盖短板 |
| 新用户首次任务成功率 | 新手第一次使用是否能完成任务 | 使用门槛是否足够低 |
| 任务重复使用率 | 用户是否反复使用同一类任务 | 是否形成刚需 |
| 任务扩展率 | 用户是否从一个任务扩展到更多任务 | 信任是否在增长 |
| 人工评审比例下降 | 在质量不下降前提下，需要人工兜底的比例是否下降 | 系统是否变成熟 |

关键判断：不能只看总成功率。简单任务成功率高、复杂任务失败多，总分也可能很好看，但这并不代表 Agent 真的更深入地帮助了用户。

## 4. 指标分层：从 MVP 到规模化

Agent 产品不同阶段，指标重点不同。

| 阶段 | 核心问题 | 应重点看的指标 |
|---|---|---|
| 0 到 1 | 用户为什么需要它 | 单一核心任务完成率、首次成功率、返工率 |
| 验证期 | 用户会不会留下来 | 任务重复使用率、结果采纳率、用户打回率、人工接管率 |
| 规模化 | 增长是否可复制 | 场景覆盖率、单任务成本、人工评审比例下降、分场景成功率 |

早期不要追求“什么都能做”。对 Agent 产品来说，MVP 应该先把一个高价值任务做深，而不是堆很多浅功能。

## 5. 评估闭环

一套可落地的评估闭环可以分成 5 步：

1. 建立任务集：从真实失败、用户高频任务、边界情况和高风险场景中收集样本。
2. 明确成功标准：每条任务都要有输入、预期结果、评估标准和失败样例。
3. 记录完整过程：不仅记录最终回复，还要记录计划、工具调用、用户确认、错误和成本。
4. 分层评分：能自动判断的自动判断，主观质量用评分标准，关键样本用人工评审校准。
5. 回流迭代：每次线上失败都要归类，进入任务集，修复后跑回归验证。

这套闭环的重点是：先有证据，再谈改进。不要靠演示视频或少量手测判断产品是否变好了。

两个最有效的落地方法：

**黄金对话**：从理想体验出发，先手写一段理想的任务对话样本（用户说什么、Agent 怎么回应、怎么引导、在哪确认），再反推系统提示词、Agent 编排和评测标准。这比写功能清单更接近产品本质——你先定义“最好长什么样”，再让 Agent 去逼近它。

**错误分析 100 条**：抽样 100 个真实用户交互日志，手动逐条标注 Pass 或 Fail，写下失败原因笔记，归纳出结构化的失败模式表。这张失败模式表比任何文字描述都更真实，可以直接转化成评测任务和评分标准。认真做完 100 条，你对产品技术边界的理解深度会超过团队里大多数工程师。

更深一层：评测的真正北极星不是某个准确率数字，而是**迭代周期**——从发现一个真实问题，到定位根因，到改 prompt、改 harness、改编排，到验证修复，这一整圈要多久。指标体系最终是为了缩短这个周期。如果指标很全但迭代还是按周算，说明评测没有真正驱动改进。

再往前一步是终局判断：评测正在从“发现问题”进化成“直接驱动代码自动迭代”。评测暴露一个 badcase，Agent 自己钻进代码库分析对应的 prompt 和编排逻辑，改完跑通验证后提交修改。这在 vibe coding 时代已经跑通：用一个强 Agent 搭建评测 Harness，用提示词代替评测脚本，一个 Agent 评测另一个 Agent，全流程从周级压缩到天级。对 DeepSeek Agent Harness 产品经理来说，这意味着评测系统本身就是 Harness 产品的一部分，不是外挂的测试工具。

## 6. 面试回答版本

如果面试官问：“你怎么衡量 Agent 是否真的帮助用户？”

可以这样回答：

> 我不会只看回答准确率，因为 Agent 的价值不是答对一句话，而是完成一个真实任务。我会把指标拆成结果、过程、稳定性、用户掌控、成本和规模化六层。最核心的指标是“用户认可的有效任务完成数”，也就是用户发起的真实任务，在合理时间和成本内被 Agent 完成，并且结果被用户采纳或确认可用。然后再用任务完成率、结果采纳率、人工接管率、工具选择正确率、步骤浪费率、单任务成本和分场景成功率去拆解。这样既能判断 Agent 有没有真的帮到用户，也能判断它是不是稳定、可控、可规模化。

如果继续追问：“怎么避免指标骗人？”

可以补充：

我会避免四个坑：

第一，只看总成功率。总成功率可能被简单任务拉高，必须按场景、难度、用户类型和风险等级分桶看。同时区分能力评估和回归评估：一边看 Agent 能不能做更难的任务，一边确保老任务没有退步。

第二，Agent 自己说完成了不算。要看外部状态、产物采纳和用户确认。

第三，更多 Token 不等于更好结果。实际数据表明，消耗更多 Token 的运行通常准确率更低——好的 Harness 应该“想清楚再动手”，而不是“先动手再想”。所以 Token 消耗要和质量一起看，不能单看成本或单看准确率。

第四，评估本身有噪声。单次 pass@1 有 2 到 6 个百分点的波动，所谓“2 到 3 个百分点的改进”可能只是噪声。评估 Harness 必须多次运行，并做成本加权，不能拿一次跑分当结论。

第五，指标突然大涨，先怀疑评分器有没有 bug，而不是先庆祝 Agent 变强了。真实案例：某个模型在某个基准上得分从 42% 跳到 95%，不是因为 Agent 变好了，而是评分器对数字精度要求过严、任务描述模糊，修了评分器之后分数就上去了。所以指标数字变化，第一反应是检查评测本身有没有问题，再下结论。

## 7. 对 DeepSeek Agent Harness 的产品启示

这套指标体系对应 DeepSeek Agent Harness 的几个产品设计方向：

1. 任务工作台要天然记录任务状态、计划、工具调用、产物和用户确认。
2. 每个任务完成后要有轻量反馈，不做长问卷，而是先收集是否有用、是否需要返工。
3. 高风险动作必须有确认点和可回滚机制，不能只追求自动化。
4. 产品数据看板不能只看调用量，要看有效任务完成、过程浪费、人工接管和分场景成功率。
5. 每次失败都应该沉淀成回归样本，让产品越用越稳。

一句话总结：

> Agent Harness 产品经理的核心工作，不是证明 Agent 很聪明，而是设计一套机制，持续证明它真的、稳定地、可控地帮用户把任务做成。


## 附录：评估方法技术选择决策矩阵

不同场景该用哪种评估方法，直接查这张表。

| 产品/场景类型 | 推荐评估方法 | 为什么 |
|---|---|---|
| 事实问答（确定性答案） | 精确匹配、非结构化问答 | 答案有标准对错，字面或语义匹配即可 |
| 数据提取/格式转换 | 结构化数据比较 | 输出是 JSON 或表格，验证字段完整和值正确 |
| 实时数据系统 | 动态标准答案 | 标准答案随时间变化，需实时校准 |
| 复杂多工具 Agent | 轨迹评估 + 工具选择精度 | 要看中间步骤：计划是否合理、工具是否选对、路径是否高效 |
| RAG 检索增强生成 | 分组件 RAG + RAGAS | 拆开评：检索器召回率 + 生成器忠实度，定位是检索还是生成的问题 |
| 对话/客服 Agent | 仿真评估 + 成对比较 | 需要模拟用户多轮交互，主观质量用相对比较更稳 |
| 代码 Agent | 单元测试 + LLM 代码质量评分 | 确定性测试验证正确性，LLM 评分看过程质量 |
| 浏览器/桌面操作 Agent | 沙箱环境 + 状态检查 | 看外部状态是否变化，不只是页面是否出现 |
| 生产环境监控 | 算法反馈 + 实时反馈 | 快速、客观、可大规模运行，捕捉线上回归 |

使用原则：

1. 不依赖单一评估方法，组合客观指标和主观指标。
2. 开发阶段用离线评测集，生产阶段加实时反馈和监控。
3. 从简单方法起步（精确匹配或问答），随系统复杂度增加逐步加复杂方法。
4. 评测不是一次性活动，是贯穿产品全生命周期的持续过程。
