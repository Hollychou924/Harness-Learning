# 读感试改对照(节选:开篇 + 第二节)

> 用途:只改读感(人称温度 / 爆点密度 / 章末小结),不动观点、数字、结构。
> 看法:左为「原稿」,右为「试改」,逐段对照。读着顺再决定全篇推。

---

## 【开篇】

### 原稿

> 先说一个我反复看到的现象:
>
> 两个团队做同类 AI 产品……
>
> 如果你相信"模型决定一切",这个现象无法解释。但它在真实世界里反复发生。……腾讯一个团队……换来的是 5 个**血淋淋**的教训——问题也几乎都不在模型本身。
>
> 本文的核心论断,一句话:
>
> **Agent = Model + Harness。模型是地板,Harness 才是天花板。**

### 试改

> 先说一个我们做 Agent 时反复撞到的现象:
>
> 两个团队做同类 AI 产品。A 团队用了市面上最强的模型,B 团队用了差一档的模型。结果 B 的产品体验明显更好——更少胡说、更少卡死、更敢放手让它干活。
>
> 如果按"模型决定一切"的直觉,这个现象说不通,但它在真实世界里反复发生。阿里一个团队把同一个应用的 AI Coding 率从 **24.86% 拉到 90.54%**,靠的不是换模型,而是重做了模型外面那一整套东西。腾讯一个团队让 6 个 AI Agent 连写 4 天代码,烧掉 **4 亿 token**,换回 5 条教训——问题也几乎都不在模型本身。
>
> 这套"模型外面的东西",就是本系列的主角:**Harness(运行框架)**。
>
> 一句话先把本文的判断摆出来,后面六节都在论证它:
>
> **Agent = Model + Harness。模型决定能力上限,Harness 决定真实场景下的实际表现。** 而 PM 真正能影响产品质量的杠杆,不在选模型,在设计 Harness。

**改了什么**
- "我反复看到" → "我们反复撞到":把"我"摊给"我们",从俯视变平视。
- 删"地板/天花板"这个并列爆点(一句话里已经有"上限/实际表现",不必再叠一层口号)。
- "血淋淋的教训"→"5 条教训":去掉煽情形容词,数字本身已经够有力。
- "核心论断"→"判断……后面六节都在论证它":少一点盖章感,多一点"带你往下看"。

## 【第二节 2.2 / 2.5】

### 原稿(2.2 + 2.5 摘要)

> **把这两个数字放一起,才是诚实的 ROI:**
> 只讲 90% 是骗人,只讲 4 亿 token 是吓人。……
>
> 讲个我亲眼见过的例子。某个手机上的 Agent,我让它做一件再简单不过的事——"打开手电筒"。结果它找了几十轮……

### 试改

> 把这两个数字放一起看,才是诚实的 ROI:只报 90% 容易让人高估,只报 4 亿 token 又容易让人不敢碰。真实的 ROI = 上限收益 × 你的 Harness 成熟度——Harness 不成熟时,投入越大,翻车越惨。
>
> 再举一个很多人都遇到过的小例子。手机上的 Agent,让它做一件再简单不过的事——"打开手电筒",结果它找了几十轮、来回折腾半天,最后还调用失败了。

**改了什么**
- "只讲 90% 是骗人,只讲 4 亿 token 是吓人":这种对仗爆句改成平铺陈述,信息不丢,语气不冲。
- "我亲眼见过的例子"→"很多人都遇到过的小例子":同样是把个人经历摊成共同经历,读者代入感更强、压迫感更弱。

## 【新增:章末小结(原稿没有,建议补)】

> ## 本章小结
>
> - **Agent = Model + Harness**:模型决定能力上限,Harness 决定真实场景的实际表现;PM 的杠杆在后者。
> - **三层语言对外、七层(ETCLOVG)对内**:对外用 Prompt / Context / Harness 解释,对内用七层拆模块、做评审。
> - **判断好坏看两样**:定性看"几项做成了系统级硬约束",定量看"6 层指标里能报出几个在持续变好的数字"。
> - **对 AIPM 的要求收敛成两核**:架构设计(做什么)+ 评估(做得好不好),两者构成闭环。
>
> 下一章 E2,我们把镜头拉近到"主循环"——看任务怎么跑起来、又怎么跑飞。

**为什么补**
- 这本书每章末尾都有「本章小结」,读者带着"被收住"的安全感离开,而不是停在一句盖章式判断上。E1 现在的结尾是"下期预告",可以在它前面补一段平铺小结。
