# 写作指南:怎么写出干货,而不是水文

> 这份指南定义每篇文章的统一结构和质量底线。所有章节草稿都按这个结构写。

## 一句话标准

> **水文**回答"X 是什么";**干货**回答"在真实开发里撞上 X 时,我怎么判断、怎么做、踩过什么坑"。

每篇文章开头必须先声明「本文回答哪些问题」,且这些问题都来自真实开发场景——是积累出来的经验,不是教科书目录。

---

## 文章统一结构

### 开篇(固定)

1. **本文回答 N 个问题** — 用列表把本章要解决的真实问题摆出来。每个问题一句话,让读者一眼知道"这正是我遇到的"。
2. **PM 视角钩子** — 一个真实场景开场(例:"用户说它怎么又忘了我上次说的话?"),而不是从技术定义切入。

### 正文(每个问题一节)

每个问题按这个内核展开,缺一不可:

| 要素 | 作用 | 反面 |
|------|------|------|
| **场景** | 这个问题在什么真实情况下出现 | ❌ 抽象地说"这很重要" |
| **踩坑/反例** | 错误做法会怎样翻车(最好有真实案例) | ❌ 只讲正确答案 |
| **原创判断** | "我用 X 时发现……""我对比 X 和 Y 后认为……" | ❌ 复述别人的概念 |
| **案例/数据支撑** | wiki 里的真实案例、数字 | ❌ 空口结论 |
| **可迁移框架** | 从这个场景抽象出能用到别处的判断方法 | ❌ 就事论事 |

### 收尾(双线 + 固定)

1. **与工程师对话的检查清单** — 一组 `- [ ]`,PM 拿着就能和工程师对齐(这是本系列的标志性产出)。
2. **"对 DeepSeek 意味着什么"** — 一句话留白,呼应求职主线。
3. **下期预告** — 衔接下一章。

---

## 硬性准则(逐篇自检)

- [ ] **PM 视角优先**:开篇是真实场景钩子,不是技术定义
- [ ] **原创判断浓度**:全文 ≥ 3 处"我用/我对比……发现"
- [ ] **反例驱动**:每个核心概念 ≥ 1 个踩坑反例
- [ ] **案例可溯源**:关键结论都能指回 wiki 里的原始案例/数据
- [ ] **视觉节奏**:≥ 1 张架构图(优先 Mermaid 可编辑)、≥ 1 张表、2-3 个重点框
- [ ] **JD 关键词显性化**:Skills / MCP / Subagent / Multi-Agent / Memory / KV Cache 等出现在标题或加粗段
- [ ] **长度**:3000-5000 字(E1/E4/E5/E7/E9 可到 6000-8000)
- [ ] **收尾有检查清单**

## 读感红线(E1 实战修订,E2-E9 统一遵守)

> 来源:E1 飞书定稿手改后复盘。这些是"读起来硌人/显自大"的真实病灶,每篇发布前逐条自检。

### 1. 不写"作者自评"句(最高优先级)
禁止任何夸自己文章的话。常见违规词:「全文最硬」「最重要的产出」「干货最硬的地方」「耍流氓」「方法很简单」「这是判断,不是常识」。
- ❌ 这一节是全文干货最硬的地方
- ❌ 这是 E1 最重要的产出
- ✅ 直接给内容,让读者自己判断硬不硬

### 2. 区分两种"我"——留经验,删自夸
本系列鼓励一手经验,但要分清:
- ✅ **有信息量的"我"**:「我用 Cursor 发现它只读相关上下文」「我对比 A/B 后认为……」——保留。
- ❌ **无信息量的"我/教育腔"**:「记住这一句」「读这张表的方法很简单」「我反复看到」——删掉或改成"我们/很多团队"。
- 经验句优先用「我们经常撞到」「很多团队会发现」开头,从俯视改平视。

### 3. 短句优先,控制加粗爆点密度
- 长句拆成 2-3 个短句。
- 每节加粗"金句"不超过 1-2 个;一段里禁止堆 4-5 个加粗爆点。
- 慎用对仗爆句(如「只讲 90% 是骗人,只讲 4 亿 token 是吓人」),能平铺就平铺。

### 4. 对外正文不用内部编号
面向读者的正文里,把 E1-E9、Q1-Q6 这类内部编号换成「第 X 章」「本章第 X 节」。内部 outline/draft 可保留编号。

### 5. 引用克制
少用"论文还指出""某官方称"的二手转述堆砌。结论直接说,需要溯源时一句话带过即可。

### 6. 每章收尾补「本章小结」
在"检查清单 / 下期预告"之前,加一段平铺的「本章小结」(3-5 条要点),给读者一个软着陆,不要停在盖章式判断上。

### 7. 标点统一中文全角(含表格和列表)
中文正文里冒号、逗号、问号统一用全角 `:,?`→ `：，？`。
- ⚠️ 最易漏的地方:**表格单元格、列表项、Mermaid 节点文字之外的说明文字**。
- 例外:代码块、行内代码、纯英文/数字串(如 `24.86%`、`SWE-bench`、文件名、URL)内部不改。
- 发布前用 `scripts/check-punctuation.py` 扫一遍,确认无残留半角。

### 发布前自检(在原"硬性准则"基础上追加)
- [ ] 无任何"作者自评/夸自己"句
- [ ] "我"全部是有信息量的经验,无教育腔
- [ ] 每节加粗爆点 ≤ 2 个
- [ ] 正文无 E#/Q# 内部编号
- [ ] 有「本章小结」
- [ ] `check-punctuation.py` 零残留半角中文标点
