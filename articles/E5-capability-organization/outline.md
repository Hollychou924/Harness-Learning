## 0. 开篇钩子(约 400 字)

- 承接 E4:上下文和记忆理顺了,但任务还是会大到"一个脑子装不下"——要读几十个文件、要同时改十几处、要又写又审。这时硬塞进一个 Agent 的上下文,只会让它越跑越乱、越跑越贵(回到 E4 的 context rot)。
- **钩子**:很多团队一上来就搞"多 Agent",6 个 Agent 一起跑,结果不是更快,是更乱、更贵、互相覆盖(回放 E2 那个 4 亿 token 案例的影子)。**多 Agent 不是默认答案,是一个高成本能力。**
- 立论:这章回答三件事——什么时候该拆(拆/不拆的判断)、拆完谁协调(角色与决策权)、怎么不互相踩(隔离)。

## 1. 有哪几种拆法:子 Agent vs 多 Agent 团队(约 1000 字)= Q1

- **子 Agent(Subagent)**:一个独立上下文窗口的"分身"。Anthropic 的比喻很贴切——**像浏览器的标签页**:开一个去查支线(某函数怎么实现、有没有现成组件),干完只把结果带回来,主线对话不被一堆中间文件污染。每个子 Agent 从空白上下文开始,可以并行,可以有不同权限(研究型只读、实现型可写)。
- **多 Agent 团队(Multi-Agent)**:多个 Agent 分角色协作。典型是 Lead(技术负责人,拆任务/分配/把关)+ Worker×N(各干一块)+ Gatekeeper(审查能不能合入)。
- **两者关系**:子 Agent 是多 Agent 的"最轻形态"——单个 Agent 临时开个分身;多 Agent 团队是"重形态"——多个有角色、要协调的 Agent 长期配合。本章统称多 Agent 系统。
- **一句判断**:别一上来就上多 Agent 团队。大多数"支线探索"用一个子 Agent(开个标签页)就够了;只有任务真的需要长期、多角色、可并行协作时,才值得上多 Agent 团队这套重家伙。
- 引 `entities/claude-code.md`、Anthropic subagents 官方文(浏览器标签页比喻)。

## 2. 拆 / 不拆的量化判断:子 Agent 什么时候开(约 1400 字)= Q2【最需量化】

- **场景**:在一个大型 TypeScript monorepo 里做新功能,主线是实现,但总冒出支线:查清某服务怎么处理 auth、找日期格式化的公共 util、看设计系统有没有现成组件。这些都不需要完整项目上下文,塞进主会话只会加噪声。
- **Anthropic 官方给的 4 个适用场景 + 每个的 signal(给可抄表)**:

  | 场景 | signal(什么信号说明值得拆) |
  |---|---|
  | 收集上下文 | 要读 dozens(几十个)文件才能搞清楚一件事 |
  | 多个独立任务 | 有 ≥多个互不依赖的子任务,可并行 |
  | 无偏复核 | 需要不被主对话假设/盲点影响的干净视角 |
  | 提交前验证 | 改完要独立检查有没有过拟合测试、漏边界 |
- **子 Agent vs /clear**:/clear 也能重置上下文拿到干净视角,但代价是丢掉全部历史;子 Agent 能拿到同样的新鲜视角,同时主对话完整保留。
- **成本账(关键)**:多 Agent 往往消耗 3-10 倍 token。所以它是高成本能力,不是默认答案。**拆之前先问:这件事是不是真需要独立上下文/并行/无偏?如果不是,单 Agent 更省更稳。**
- **一句判断**:拆子 Agent 的判断标准很具体——读几十个文件、有多个独立可并行子任务、需要无偏复核、提交前验证,命中其一才拆;为了"看起来高级"而拆,只会多花 3-10 倍 token。
- 引 Anthropic subagents 官方文(4 场景+signal)、`entities/claude-code.md`(3-10 倍 token、多 Agent 取舍)。

## 3. 谁来协调:协调者职责、冲突裁决、决策权(约 1200 字)= Q3

- **核心原则**:**"别让 Agent 开车,让 Agent 当导航。"** Agent 可以出主意(提议下一步、提议方案),但"做不做、按什么预算做、要不要继续"这些方向盘动作,握在 Harness / 协调者手里。
- **协调者(Lead)的职责边界**:拆任务、分配、把关交付质量、根据实现调整方案。它对"技术方案和最终效果"负责,但**不亲自下场写每一行**——它是导航,不是司机。
- **Gatekeeper 的职责**:对要合入主干的产出做审查,拒绝时写一份**具体的反馈文件**,Worker 拿着反馈在原副本里修好再提交。关键不是"谁更聪明",是"有一个说了算的角色 + 一条明确的返工路径"。
- **冲突裁决**:多个 Worker 产出不一致时,由 Lead/Gatekeeper 拍板。**协调者的 synthesis(把多个结果综合成一个决定)是稀缺能力**——这恰恰是多 Agent 最难、也最体现设计水平的地方。
- **决策权归属(呼应 E2)**:不能让 Planner/Agent 自己决定调度、是否继续、是否重启——预算、并发、安全、成本必须由 Harness 集中管控(阿里、腾讯两边都印证:Multi-Agent 决策权在 Harness 不在 Planner)。
- **一句判断**:多 Agent 协作的灵魂不是"多开几个 Agent",是设计清楚——谁出主意、谁拍板、出分歧谁裁决、决策权握在谁手里。没有明确协调者和决策权归属的多 Agent,就是 E2 那个一推就倒的多米诺。
- 引 `topics/agentway-harness-books.md`(原则 8 协调者 synthesis 稀缺)、`topics/tencent-*`(Lead/Gatekeeper)、`topics/aliyun-*`(决策权在 Harness)。

## 4. 怎么不互相踩:上下文 / 状态隔离(约 1000 字)= Q4

- **先和 E2 划清**:E2 讲多 Agent"跑起来不跑飞"(死循环、硬闸、续跑、监控);这一节讲"多个 Agent 同时动手时,怎么不互相覆盖、不互相污染"——是隔离问题。
- **三层隔离**:
  - 上下文隔离:子 Agent 各有独立上下文窗口,天然隔离——它看不到、也不会污染主对话(这是子 Agent 的核心价值)。
  - 状态隔离:多个 Worker 各在独立代码副本(Git Worktree)里干活,完成后交 Lead 合入。**硬原则:并行 Agent 不共享可写状态。** 共享只读没问题,共享可写就是灾难。
  - 缓存隔离(呼应 E4):forked agent(分叉出去的 Agent)必须 cache-safe——别让分叉动作污染主会话的稳定前缀,否则一分叉就 cache miss、成本飙升。
- **一句判断**:多 Agent 不互相踩的前提是"可写的东西先隔离开"——独立上下文、独立副本、不共享可写状态、分叉要 cache-safe。隔离没做好,Agent 越多,踩踏越狠。
- 引 `topics/tencent-*`(Worktree、不共享可写状态)、`topics/agentway-harness-books.md`(forked agent cache-safe)。

## 5. 让用户看得懂 + 控成本(约 800 字)= Q5

- **看得懂(承接 E2 的对话流→任务流)**:多个 Agent 同时跑,用户最怕"黑箱"——不知道哪个 Agent 在干嘛、到哪了、能不能插手。解法是把每个 Agent 的状态做成可见:任务卡片(运行中/等待确认/已完成)、执行计划、推理过程、决策点(放行/改向/打回)。**多 Agent 越多,可观测越重要——否则一团乱麻没人看得懂。**
- **控成本**:多 Agent 3-10 倍 token 不是白烧的借口。打法和 E2 一致——**便宜模型干粗活(拆任务、读文件、跑测试),贵模型只用在关键决策(方案行不行、代码能不能合)**;4 亿 token 案例里 92% 被低成本模型吃掉、真正调贵模型不到 2.5%,正是这个分工的体现。再配合"该不该拆"的判断(第二节),别为了多 Agent 而多 Agent。
- **一句判断**:多 Agent 系统的两条命脉是"看得见"和"花得值"——可观测让协作不变黑箱,模型分级让 3-10 倍 token 花在刀刃上。
- 引 `concepts/quest-mode-agent-development.md`(任务流/决策点)、`topics/tencent-*`(92% token、3-10 倍)。

## 6. 拔高 + 自查表 + 下期预告(约 600 字)

- **拔高(呼应 E1 双核)**:多 Agent 系统设计,正是 E1 第六节"架构设计能力"里那条**"多 Agent 协作架构"**的落地——设计 Agent 之间怎么通信、怎么调度、怎么决策、怎么隔离。
  - 架构设计能力的体现 = 角色边界(Lead/Worker/Gatekeeper 各管什么)、协调结构(谁拍板、决策权在谁)、隔离设计(独立上下文/副本/cache-safe)。
  - 一句话:**多 Agent 不是"人多力量大",是"分工清楚、协调到位、隔离干净"才力量大。设计一支能协作的 Agent 团队,考的就是 PM 的架构设计能力。**
- **文末:多 Agent 系统自查清单**:
  - [ ] 这件事真需要拆吗?命中"读几十文件/可并行/无偏复核/提交前验证"了吗?还是单 Agent 更省?
  - [ ] 拆出来的是子 Agent(开个标签页)还是多 Agent 团队(长期多角色)?选对形态了吗?
  - [ ] 有明确的协调者吗?谁拍板、冲突谁裁决、决策权在 Harness 还是 Planner?
  - [ ] 并行 Agent 隔离了吗?独立上下文/独立副本/不共享可写状态/分叉 cache-safe?
  - [ ] 多 Agent 状态用户看得见、插得上手吗?成本是不是便宜模型干粗活、贵模型干关键?
- **下期预告**:E6 安全与权限——多个 Agent 能并行动手了,可"权限大了出了问题谁负责、哪些操作要人工确认、被提示注入攻击怎么防、做错了怎么回滚"。
