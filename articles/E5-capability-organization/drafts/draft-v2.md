## 开篇：多 Agent 的难点不是把任务分出去，是把结果收回来

E4 把上下文和记忆理顺了。但有些任务，就是会大到"一个脑子装不下"——要读几十个文件才能搞清来龙去脉、要同时改十几处互不相关的代码、要又写又审。这时候硬把所有东西塞进一个 Agent 的上下文，只会让它越跑越乱、越跑越贵（就是 E4 说的 context rot）。

很自然地，大家会想到：**那就多开几个 Agent 一起干。**

但这里有个普遍的误区。很多团队一上来就搞"多 Agent 系统",6 个 Agent 同时跑，期待 6 倍的速度。结果往往不是更快，而是更乱、更贵、互相覆盖——这正是 E2 那个"4 亿 token、6 个 Agent 连写 4 天"案例的影子。

> **一句话先立住：多 Agent 不是默认答案，是一个高成本能力。** 实践中，多 Agent 常见消耗单 Agent 的 3-10 倍 token;Anthropic 官方也明确说，Agent Team 的 token 随活跃 teammate 数**线性增长**、会显著高于单会话。

所以这一章不鼓吹"多 Agent 很厉害"，而是冷静回答几件事：**什么时候该拆、怎么拆、拆完谁协调、结果怎么收回来、怎么不互相踩、怎么证明真的值得。**

提前剧透本章的灵魂判断：

> **多 Agent 的难点不是把任务分出去，而是把结果收回来；不是让一群 Agent 一起干活，而是设计一套能分工、能对齐、能裁决、能合并、能停下来的组织系统。**

## 一、多 Agent 系统到底有几种形态？（Q1）

"多个 Agent 一起干"其实有四种形态，从轻到重、成本从低到高。先用一张表看清全貌：

| 形态 | 本质 | 谁协调 | 适合任务 | 成本 |
|---|---|---|---|---|
| **单 Agent** | 一个 Agent 走完全流程 | 它自己 | 小任务、强顺序任务 | 最低 |
| **子 Agent(Subagent)** | 主 Agent 临时开个分身 | 主 Agent | 支线探索、独立查证、提交前复核 | 中 |
| **Agent Team** | 多个长期角色协作 | Lead + 共享任务列表 | 多模块开发、复杂审查、竞争性假设 | 高 |
| **Dynamic Workflow** | 用脚本编排大量 Agent | 可执行脚本 + 独立 Runtime | 大规模迁移、全仓审计、交叉验证 | 很高 |

用三个比喻就记住了：

- **子 Agent = 开个浏览器标签页**：你在主线写功能，顺手开个标签页去查支线（这个服务怎么处理鉴权？有没有现成的日期格式化工具？），它在自己的上下文里查完，**只把结果带回来**，主线不被一堆中间文件污染。
- **Agent Team = 组建一个临时项目组**：多个 Agent 各有角色（Lead/Worker/Reviewer），长期协作。
- **Dynamic Workflow = 把项目经理写成一段可执行脚本**：协调逻辑不再活在某个 Agent 的脑子（上下文）里，而是变成代码。

**子 Agent 和 Agent Team 有一个关键区别（官方明确）**：子 Agent 在单个 session 内运行，**只能把结果汇报给主 Agent**，彼此之间不通信；而 Agent Team 的 teammates **各有独立上下文窗口、可以直接互相通信**。一个是"主 Agent 派出去的侦察兵，回来交报告"；一个是"能互相喊话的项目组成员"。

### 1.1 第四形态详解：Dynamic Workflows——把协调逻辑从上下文搬进代码

前三种形态有一个共同天花板：**编排逻辑活在主 Agent 的上下文里。** 负责协调的 Agent 要在自己的上下文窗口里记着所有子任务的进度、状态、结果。子 Agent 一多（超过十来个），上下文被中间结果填满，协调者开始"失忆"和逻辑漂移——这就是 E4 讲的 context rot 在多 Agent 场景的放大。

Anthropic 2026 年 5 月推出的 **Dynamic Workflows（动态工作流）**，给了这个天花板一个很聪明的破法：

> **上下文装不下协调状态时，就把协调逻辑从上下文搬到代码里。**

具体是：主 Claude 只负责**写**一段 JavaScript 编排脚本（怎么拆、怎么调度、怎么复核、怎么返工），不亲自执行；一个独立、不懂业务的 **Runtime** 执行脚本、管状态、生成和协调子 Agent；所有中间结果存在**脚本变量**里，不回灌主上下文；最终只把聚合结果返回主会话。官方对四种形态"谁决定下一步、中间结果存哪"的对比：

| 机制 | 谁决定下一步 | 中间结果存哪 |
|---|---|---|
| 子 Agent | Claude，逐轮决定 | 主上下文 |
| Skills | Claude 按指令 | 主上下文 |
| Agent Team | Lead 监督同侪 | 共享任务列表 |
| **Dynamic Workflow** | **脚本（代码）** | **脚本变量** |

官方把这件事概括成一句很到位的话——"**A harness for every task**":Claude 能为每个任务即时写出一套定制的多 Agent harness（正好呼应 E1 的"Harness > Model"）。配套细节（官方文档）：脚本本身**没有文件系统/shell 访问权**（只负责协调，真正读写和命令由 agents 做）、**最多 16 个并发 agent、单次 run 最多 1000 个 agent**（防失控的 backstop）；一键开关 `/effort ultracode`（把思考预算拉满 + 自动决定何时启用 workflow）；因为状态存在脚本变量，所以天然支持断点续跑、可复用、可观察。

还有一个 PM 值得记的工程细节：Dynamic Workflows 能跑起上百个子 Agent，有一半功劳在 cache——Opus 4.8 把"长跑中途注入新指令也不破坏 prompt cache"做出来了，否则一个跑一小时、不断注入新上下文的 workflow 早被 cache miss 的成本拖垮（这正是 E4"cache 是 Harness 经济学基础设施"的活例子）。

> **一句判断**：从单 Agent → 子 Agent → Agent Team → Dynamic Workflow，是同一条主线越走越极致——协调者从"它自己"，到"主 Agent"，到"Lead + 共享列表"，最后到"一段代码 + 独立 Runtime"。越往后越强，也越贵。别拿重武器打小仗。

## 二、什么时候该拆？任务又该怎么拆？（Q2）

拆 Agent 有两个问题，大多数人只想了第一个（"要不要拆"），忘了第二个（"怎么拆"）——而**拆分方式直接决定协作质量**。

### 2.1 该不该拆：子 Agent 的量化判断

什么时候该开子 Agent?Anthropic 给了 4 个适用场景，每个都配了明确的 **signal（信号）**:

| 场景 | signal：出现这个信号，就值得拆 |
|---|---|
| **收集上下文** | 搞清一件事，要读 dozens（几十个）文件 |
| **多个独立任务** | 有多个互不依赖的子任务，可以并行 |
| **无偏复核** | 需要一个不被主对话假设、盲点影响的干净视角 |
| **提交前验证** | 改完要独立检查有没有过拟合测试、漏掉边界 |

一个常见疑问：要"干净视角"，用 `/clear` 清空上下文不也行？区别在于：`/clear` 能拿到新鲜视角，但代价是把全部历史也丢了；开子 Agent 能拿到同样新鲜的视角，**同时主对话完整保留**。

但别忘成本：多 Agent 常见 3-10 倍 token。拆之前先问：这件事真需要独立上下文 / 真能并行 / 真需要无偏吗？都不是，单 Agent 一口气做完更省更稳。

### 2.2 怎么拆：六种拆法

确定要拆之后，按什么维度拆？这决定了 Agent 之间会不会打架：

| 拆法 | 适合场景 | 例子 | 风险 |
|---|---|---|---|
| **按模块拆** | 多模块独立改造 | 前端 / 后端 / 测试各一个 Agent | 模块边界不清会冲突 |
| **按文件拆** | 大批量同类改动 | 上千文件迁移，每批分给一个 Agent | 跨文件依赖被切断 |
| **按阶段拆** | 流程清晰的串行任务 | 探索→实现→测试→复核，各一个阶段 | 阶段间交接信息易丢 |
| **按角色拆** | 需要不同专业视角 | 安全审计员 / 性能优化员 / 代码审查员 | 角色重叠或留白 |
| **按假设拆** | 一个问题有多种可能 | flaky bug：三个 Agent 各验证一种竞争性假设 | 假设设计不全面 |
| **按方案拆** | 需要多方案比选 | 命名/架构：多个 Agent 各出方案再 PK | 收敛成本高 |

**一句判断**：拆的维度选错，Agent 越多越乱。按模块拆要先划清模块边界，按文件拆要先理清跨文件依赖，按阶段拆要设计好交接格式。**拆分不是把活随便切几块，是先想清楚"切口在哪不会互相撕扯"。**

## 三、谁来协调？结果怎么收回来、冲突谁拍板？（Q3）

任务拆出去之后，真正的难点来了——**怎么把结果收回来。** 这一节是全章的核心。

### 3.1 协调者（Lead）的职责边界

核心原则一句话：**"别让 Agent 开车，让 Agent 当导航。"** Agent 可以出主意，但"做不做、按什么预算做、要不要继续"这些方向盘动作，握在协调者 / Harness 手里。

- **Lead** 拆任务、分配、把关交付质量、根据实现调整方案。它对"技术方案和最终效果"负责，但**不亲自下场写每一行**——它是导航不是司机。一旦 Lead 自己陷进某个具体实现，团队就失去了那个"站在全局看的人"。
- **决策权归属（呼应 E2）**：不能让 Planner / 某个 Agent 自己决定调度、是否继续、是否重启——预算、并发、安全、成本必须由 Harness 集中管控。阿里、腾讯两边实践都印证：**Multi-Agent 的决策权在 Harness，不在 Planner。**

### 3.2 每个 Agent 必须有"交付协议"

要把分散的结果收回来、还能用，前提是每个 Agent 不能随便吐一段自由文本，而要按**结构化的交付协议**输出。一个好用的协议至少包含：

| 字段 | 含义 |
|---|---|
| conclusion | 一句话结论 |
| evidence | 支撑结论的证据（文件、行号、报错） |
| files | 改动 / 涉及的文件清单 |
| risks | 风险与不确定点 |
| confidence | 自评置信度 |

有了协议，协调者才能机器化地比对、合并、裁决，而不是去读一堆散文。（Dynamic Workflows 里这就是 `schema:` 强制子 Agent 走 StructuredOutput 工具，校验在 tool-call 层，不用自己 JSON.parse。）

### 3.3 fan-out / fan-in：扇出去，再收回来

多 Agent 的标准结构是 **fan-out（扇出）→ fan-in（收口）**：把任务扇出给 N 个 Agent 并行干，再把 N 份结果收回来合并成一个。收口这一步最考设计：

- **合并不是"拼接"，是"综合"**：协调者要做 synthesis——把多份结构化结果对齐、去重、消解矛盾，产出一个对的结论。**这是多 Agent 最稀缺的能力**：难的从来不是把活分出去，是把结果收回来综合对。
- **冲突按证据裁决，不按投票**：三个 Agent 说 A、一个说 B，不代表 A 就对。要看谁的 evidence 更硬。**多数票在 Agent 世界里靠不住**——可能只是同一个偏见被复制了三份。
- **独立验证收口**：对高风险结论，再派一个"对抗性"Agent 专门去推翻它，推不翻才采纳（这是 Dynamic Workflows 里 /deep-research 和大型审计的质量模式：fan-out + 独立 verifier）。

> **一句判断**：多 Agent 协作的灵魂是"收口"——有结构化交付协议才能合并，按证据而非投票才能裁决，有独立验证才敢采纳。没有收口设计的多 Agent，扇出去就收不回来，就是 E2 那个一推就倒的纸牌屋。

## 四、怎么不互相踩？上下文 / 状态 / 工具权限隔离（Q4）

先和 E2 划清：E2 讲多 Agent"跑起来不跑飞"（死循环、四道硬闸、断点续跑、监控）；这一节讲"多个 Agent 同时动手时，怎么不互相覆盖、不互相污染"——是隔离问题。隔离分三层：

**① 上下文隔离**：子 Agent / teammate 各有独立上下文窗口，天然隔离——看不到也不会污染别人（回到"浏览器标签页"的比喻，每个标签页互不干扰）。

**② 状态隔离**：多个 Worker 同时改代码，改同一份文件必然互相覆盖。正确做法是让每个 Worker 在**独立的代码副本（Git Worktree）**里干活，完成后交 Lead 合入。硬原则：

> **并行的 Agent，不共享可写状态。** 共享只读没问题，共享可写就是灾难。

外加缓存隔离（呼应 E4）：分叉出去的 Agent 必须 cache-safe，别让分叉污染主会话稳定前缀，否则一分叉就 cache miss、成本飙升。

**③ 工具权限隔离（按角色给权限，不是人人全开）**：不同角色该有不同的工具权限，这是隔离也是安全：

| 角色 | 工具权限 |
|---|---|
| Explorer / 研究型 | 只读（读文件、搜代码） |
| Worker / 实现型 | 可写（编辑文件、跑命令） |
| Reviewer / 审查型 | 只读 + 评论，不直接改代码 |
| Gatekeeper / 把关 | 决定能不能合入，不下场实现 |

Anthropic 官方也明确：子 Agent 可以有不同权限——研究型只读、实现型可写。**权限按角色最小化，既防互相踩，也防越权闯祸（完整的权限/审批是 E6 的主场）。**

> **一句判断**：多 Agent 不互相踩的前提，是"可写的东西先隔离开"——独立上下文、独立代码副本、不共享可写状态、按角色最小化工具权限、分叉 cache-safe。隔离没做好，Agent 越多，踩踏越狠。

## 五、看得懂 + 控成本 + 怎么证明值得（Q5）

多 Agent 跑起来后，还有三道关：用户**看得懂**吗？成本**花得值**吗？到底**比单 Agent 强**吗？

**看得懂（承接 E2 的"对话流 → 任务流"）**：多个 Agent 同时跑，用户最怕"黑箱"——不知道哪个 Agent 在干嘛、到哪了、能不能插手。解法是把每个 Agent 的状态做成可见：任务卡片（运行中/等待确认/已完成）、执行计划、决策点（放行/改向/打回）。Dynamic Workflows 的 `/workflows` 视图就是范例——每个阶段的 agent 数、token 总量、耗时都看得见，可以钻进任一阶段看每个 agent 找到了什么、必要时停掉。**Agent 越多，可观测越重要。**

**控成本**:
- **3-5 个起步**（官方建议）：多数场景从 3-5 个 teammate 开始，平衡并行收益和协调成本；并发不是越高越好，超过一定数量边际收益递减。
- **模型分级**：便宜模型干粗活（拆任务、读文件、跑测试），贵模型只用在关键决策和最终 synthesis（4 亿 token 案例里 92% token 被低成本模型吃掉，真正调贵模型不到 2.5%）。
- **每个 Agent 给 token budget + 阶段验收条件 + 早停**：别让某个 Agent 无限烧。

**怎么证明值得（呼应 E1 双核里的"评估能力"）**：多 Agent 的成功标准不是"跑了很多 Agent"，是**单位成本下的交付质量更高**。要和单 Agent 做 baseline 对比：

| 对比项 | 看什么 |
|---|---|
| 成功率 | 任务完成率是否提升 |
| 耗时 | 总 elapsed time 是否降低 |
| 成本 | token / 钱是否可接受 |
| 质量 | 测试通过 / review / bug 数是否更好 |
| 用户负担 | 人工介入次数是否增加 |
| 冲突率 | 文件覆盖 / 合并失败是否可控 |

（完整的评估体系是 E8 的主场，这里只给一张起手对比表。）

> **一句判断**：多 Agent 系统的三道关是"看得见、花得值、证得明"——可观测让协作不黑箱，3-5 起步+模型分级让成本花在刀刃上，baseline 对比让你能证明它真比单 Agent 强，而不是自我感觉良好。

## 六、多 Agent 决策树 + 拔高

把整章打穿，其实就是一棵决策树：

```text
任务简单、顺序明确、文件范围小?
  是 → 单 Agent

需要支线探索、独立查证、提交前复核?
  是 → 开子 Agent(开个浏览器标签页)

有多个独立模块/角色,需要长期协作或互相挑战?
  是 → Agent Team(组建临时项目组)

任务大到一个 Lead 的上下文都协调不过来,需要几十到上百个并行子任务?
  是 → Dynamic Workflow(把项目经理写成脚本)

以上都不是?
  不拆。先用单 Agent。
```

### 拔高：设计一支能协作的 Agent 团队，考的是架构设计能力

回到 E1 的"双核能力"。E1 第六节讲架构设计能力时，列过一条**"多 Agent 协作架构"**——设计多个 Agent 之间怎么通信、调度、决策、隔离。这一整章就是那条能力的落地：

- **架构设计能力**的体现 = 选对形态（四形态）、拆对维度（六拆法）、设计收口（交付协议+fan-in+按证据裁决）、按角色隔离权限。
- **评估能力**的体现 = 用 baseline 对比（成功率/成本/质量/冲突率）证明多 Agent 真的值得，而不是堆 Agent 自嗨。
- 这些都不是"功能"，是 PM 设计一支 Agent 团队时必须拍板的**架构决策**。

一句话收束：

> **多 Agent 的难点不是把任务分出去，而是把结果收回来；不是让一群 Agent 一起干活，而是设计一套能分工、能对齐、能裁决、能合并、能停下来的组织系统。** 设计这套系统——什么时候拆、怎么拆、谁协调、怎么收口、怎么隔离、怎么证明值得——考的正是 Harness 时代 PM 的架构设计能力。传统 PM 设计一个功能，Harness PM 设计一支能自主协作的智能团队。
