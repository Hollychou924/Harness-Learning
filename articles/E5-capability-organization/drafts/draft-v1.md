## 开篇:多 Agent 不是"人多力量大",是高成本能力

E4 把上下文和记忆理顺了。但有些任务,就是会大到"一个脑子装不下"——要读几十个文件才能搞清来龙去脉、要同时改十几处互不相关的代码、要又写又审。这时候硬把所有东西塞进一个 Agent 的上下文,只会让它越跑越乱、越跑越贵(就是 E4 说的 context rot)。

很自然地,大家会想到:**那就多开几个 Agent 一起干。**

但这里有个普遍的误区。很多团队一上来就搞"多 Agent 系统",6 个 Agent 同时跑,期待 6 倍的速度。结果往往不是更快,而是更乱、更贵、互相覆盖——这正是 E2 那个"4 亿 token、6 个 Agent 连写 4 天"案例的影子。

> **一句话先立住:多 Agent 不是默认答案,是一个高成本能力。** 官方数据是,多 Agent 往往消耗单 Agent 的 3-10 倍 token。

所以这一章不鼓吹"多 Agent 很厉害",而是冷静回答三件事:

1. **什么时候该拆**(拆 / 不拆的判断);
2. **拆完谁来协调**(角色分工与决策权);
3. **怎么不互相踩**(隔离)。

把这三件事想清楚,多 Agent 才是力量;想不清楚,就是 E2 那个一推就倒的多米诺。

## 一、有哪几种拆法:子 Agent vs 多 Agent 团队(Q1)

"拆"这件事,有轻有重,先把两种形态分清。

**子 Agent(Subagent)——最轻的拆法。** 它是一个拥有独立上下文窗口的"分身"。Anthropic 官方给了一个特别贴切的比喻:**子 Agent 就像 Claude Code 会话里的"浏览器标签页"**——你正在主线写功能,需要顺手查个支线(这个服务怎么处理鉴权?有没有现成的日期格式化工具?设计系统里有没有现成组件?),就开一个标签页(子 Agent)去查,它在自己的上下文里读文件、探索代码,**干完只把结果带回来**,主线对话不会被一堆中间文件污染。每个子 Agent 从空白上下文开始,可以并行,还能有不同权限——研究型的只读、实现型的可写。

**多 Agent 团队(Multi-Agent)——重的拆法。** 多个 Agent 分角色、长期协作。典型结构是:Lead(技术负责人,拆任务、分配、把关)+ Worker×N(各干一块)+ Gatekeeper(审查能不能合入主干)。这是一支有组织、要协调的"团队"。

**两者的关系**:子 Agent 是多 Agent 的"最轻形态"——单个 Agent 临时开个分身去办支线;多 Agent 团队是"重形态"——多个有角色、需要协调的 Agent 长期配合。本章把它们统称为多 Agent 系统。

> **一句判断**:别一上来就上多 Agent 团队。大多数"支线探索"用一个子 Agent(开个浏览器标签页)就够了;只有当任务真的需要长期、多角色、可并行地协作时,才值得上多 Agent 团队这套重家伙。

### 1.1 第四层:Dynamic Workflows——把编排逻辑写成代码

子 Agent 和多 Agent 团队都有一个共同天花板:**编排逻辑活在主 Agent 的上下文里。** 负责协调的那个 Agent,要在自己的上下文窗口里记着所有子任务的进度、状态、结果。子 Agent 一多(超过十来个),上下文就被中间结果填满,协调者开始"失忆"和逻辑漂移——这就是 E4 讲的 context rot 在多 Agent 场景的放大。

Anthropic 2026 年 5 月推出的 **Dynamic Workflows(动态工作流)**,给了这个天花板一个很聪明的破法:**把编排逻辑从主上下文里搬出来,变成一段可执行的 JavaScript 脚本。**

- 主 Claude 只负责**写**这段编排脚本(怎么拆、怎么调度、怎么复核、怎么返工),自己不执行;
- 一个独立的、不懂业务的 **Runtime** 负责执行脚本、管理状态、生成和协调子 Agent;
- 所有中间结果存在**脚本变量**里,不回灌主上下文;
- 最终只把**聚合后的结果**返回主会话。

它和前三种拆法的本质区别,可以用一张表看清(官方对比):

| 机制 | 谁决定下一步 | 中间结果存哪 | 典型规模 |
|---|---|---|---|
| 子 Agent | Claude,逐轮决定 | 主上下文 | 每轮几个 |
| Skills | Claude 按指令 | 主上下文 | 同上 |
| 多 Agent 团队 | Lead 监督同侪 | 共享任务列表 | 少数长跑同侪 |
| **Dynamic Workflows** | **脚本(代码)** | **脚本变量** | **几十到上百(防失控上限 1000),并发约 16** |

关键差异就一句:**计划(plan)活在可执行代码里,而不是一段不断变长的聊天记录里。** 这才是它能突破"上下文窗口瓶颈"、在一个 session 里调度上百个子 Agent 的根因。官方把这件事概括成一句很到位的话——"**A harness for every task**":Claude 现在能为每个任务即时写出一套定制的多 Agent harness(正好呼应 E1 的"Harness > Model")。

它还顺带解决了 E2 讲过的两个老问题:因为状态存在脚本变量而非主上下文,所以**天然支持断点续跑**(关机/断网都不影响,从断点接着跑);又因为编排是代码,所以可以用 `while`/`if`/动态扇出做**动态拓扑**(比如安全审计中一个子 Agent 发现高危漏洞,脚本自动再起 3 个从不同角度交叉验证)。一键开关是 `/effort ultracode`——把思考预算拉满 + 自动决定何时启用 workflow。

**一个值得 PM 记住的工程细节**:Dynamic Workflows 能跑起上百个子 Agent,有一半功劳在 cache——Opus 4.8 把"长跑中途注入新指令也不破坏 prompt cache"做出来了,否则一个跑一小时、不断注入新上下文的 workflow 早就被 cache miss 的成本拖垮(这正是 E4 那句"cache 是 Harness 经济学基础设施"的活例子)。

> **一句判断**:从子 Agent → 多 Agent 团队 → Dynamic Workflows,是同一条主线越走越极致——协调者从"用上下文记账的主 Agent",升级成"一段代码 + 一个独立 Runtime"。但越强越贵:Dynamic Workflows 官方明确警告它"比普通会话烧多得多的 token",只适合复杂、高价值任务。它不是日常拆活的默认选项,是处理"一个 Agent 扛不动的大规模任务"时的重武器。

## 二、拆 / 不拆的量化判断:子 Agent 什么时候开(Q2)

这是多 Agent 里最需要"量化依据"的决策。先看一个真实场景(来自 Anthropic 官方):你在一个大型 TypeScript monorepo 里做新功能,主线是实现,但支线任务不断冒出来——查清某个服务怎么处理 auth、找日期格式化的公共 util、看设计系统里有没有接近的组件。这些都不需要完整的项目上下文,塞进主会话只会加噪声。

那到底什么时候该开子 Agent?Anthropic 给了 4 个适用场景,而且每个都配了一个明确的 **signal(信号)**:

| 场景 | signal:出现这个信号,就值得拆 |
|---|---|
| **收集上下文** | 搞清一件事,要读 dozens(几十个)文件 |
| **多个独立任务** | 有多个互不依赖的子任务,可以并行 |
| **无偏复核** | 需要一个不被主对话假设、盲点影响的干净视角 |
| **提交前验证** | 改完要独立检查有没有过拟合测试、漏掉边界 |

这里还有一个常见疑问:既然要"干净视角",用 `/clear` 清空上下文不也行?区别在于:`/clear` 确实能重置、拿到新鲜视角,但代价是**把全部历史也丢了**;而开一个子 Agent 能拿到同样新鲜的视角,**同时主对话完整保留**。这就是子 Agent 比"清空重来"更优雅的地方。

但别忘了那个成本数字:**多 Agent 往往消耗 3-10 倍 token。** 所以拆之前一定要先问自己:这件事是不是真需要独立上下文、真能并行、真需要无偏?如果都不是,单 Agent 一口气做完,更省也更稳。

> **一句判断**:拆子 Agent 的标准很具体——读几十个文件、有多个独立可并行的子任务、需要无偏复核、提交前要独立验证,命中其中之一才拆;为了"看起来高级"而拆,只会多花 3-10 倍 token。

## 三、谁来协调:协调者职责、冲突裁决、决策权(Q3)

任务拆给多个 Agent 之后,马上面临一个新问题:**谁说了算?**

核心原则一句话:**"别让 Agent 开车,让 Agent 当导航。"** Agent 可以出主意——提议下一步、提议方案;但"做不做、按什么预算做、要不要继续"这些方向盘动作,必须握在协调者 / Harness 手里。

**协调者(Lead)的职责边界**:拆任务、分配、把关交付质量、根据实现情况调整技术方案。它对"技术方案和最终效果"负责,但**不亲自下场写每一行代码**——它是导航,不是司机。一旦 Lead 自己也陷进某个具体实现,整个团队就失去了那个"站在全局看的人"。

**Gatekeeper 的职责**:对要合入主干的产出做审查。拒绝时,它不是甩一句"不行",而是写一份**具体的反馈文件**,Worker 拿着这份反馈在原来的代码副本里修好、再提交。关键不在于"谁更聪明",而在于"有一个说了算的角色 + 一条明确的返工路径"。

**冲突裁决**:多个 Worker 产出不一致时,由 Lead / Gatekeeper 拍板。这里要点出一个被低估的能力——**协调者的 synthesis(把多个结果综合成一个决定)是稀缺能力**。多 Agent 系统最难、也最体现设计水平的地方,恰恰不是"把活分出去",而是"把分散的结果收回来、综合成一个对的决定"。

**决策权归属(呼应 E2)**:不能让 Planner 或某个 Agent 自己决定调度、是否继续、是否重启——预算、并发、安全、成本必须由 Harness 集中管控。这一点阿里和腾讯两边的实践都反复印证:**Multi-Agent 的决策权在 Harness,不在 Planner。** 让 Planner 自己当家做主,就是 E2 那条多米诺失败链的起点。

> **一句判断**:多 Agent 协作的灵魂不是"多开几个 Agent",是设计清楚——谁出主意、谁拍板、出分歧谁裁决、决策权握在谁手里。没有明确协调者和决策权归属的多 Agent,就是一推就倒的纸牌屋。

## 四、怎么不互相踩:上下文 / 状态隔离(Q4)

先和 E2 划清界限:E2 讲多 Agent"跑起来不跑飞"(死循环、四道硬闸、断点续跑、监控);这一节讲的是另一件事——**多个 Agent 同时动手时,怎么不互相覆盖、不互相污染。** 这是隔离问题。

隔离分三层:

**① 上下文隔离**:子 Agent 各有独立的上下文窗口,天然隔离——它看不到、也不会污染主对话。这正是子 Agent 的核心价值(回到第一节那个"浏览器标签页"的比喻:每个标签页互不干扰)。

**② 状态隔离**:多个 Worker 同时改代码,如果改同一份文件,必然互相覆盖。正确做法是让每个 Worker 在自己**独立的代码副本(Git Worktree)**里干活,完成后交给 Lead 合入。这背后是一条硬原则:

> **并行的 Agent,不共享可写状态。**

共享只读没问题(大家都能看同一份资料);共享可写就是灾难(大家都往同一个文件里写)。想并行,就先把"可写的东西"隔离开。

**③ 缓存隔离(呼应 E4)**:分叉出去的 Agent(forked agent)必须 **cache-safe**——别让分叉动作污染主会话那段需要稳定的前缀。否则你每分叉一次就触发一次 cache miss,成本立刻飙升(这正是 E4 讲的 cache × 动态内容的冲突,在多 Agent 场景的放大)。

> **一句判断**:多 Agent 不互相踩的前提,是"可写的东西先隔离开"——独立上下文、独立代码副本、不共享可写状态、分叉要 cache-safe。隔离没做好,Agent 越多,踩踏越狠。

## 五、让用户看得懂 + 控成本(Q5)

多 Agent 系统跑起来之后,还有两条命脉:用户**看得懂**吗?成本**花得值**吗?

**看得懂(承接 E2 的"对话流 → 任务流")**:多个 Agent 同时跑,用户最怕的是"黑箱"——不知道哪个 Agent 在干嘛、跑到哪了、能不能插手。解法是把每个 Agent 的状态做成可见的:任务卡片(运行中 / 等待确认 / 已完成)、执行计划、推理过程、决策点(放行 / 改方向 / 打回)。**Agent 越多,可观测越重要**——否则就是一团没人看得懂的乱麻,出了问题也无从下手。

**花得值**:那个"3-10 倍 token"不是白烧的借口。控成本的打法和 E2 一致——**便宜模型干粗活(拆任务、读文件、跑测试),贵模型只用在关键决策上(这个方案行不行、这段代码能不能合入)**。4 亿 token 案例里有个数字很说明问题:92% 的 token 被低成本模型吃掉,真正调用昂贵模型的请求不到 2.5%。这正是"模型分级"分工的体现。再配合第二节"该不该拆"的判断——别为了多 Agent 而多 Agent——成本就压下来了。

> **一句判断**:多 Agent 系统的两条命脉是"看得见"和"花得值"——可观测让协作不变黑箱,模型分级让 3-10 倍 token 花在刀刃上。

## 六、拔高:设计一支能协作的 Agent 团队,考的是架构设计能力

回到 E1 立下的"双核能力"。E1 第六节讲架构设计能力时,列过一条叫**"多 Agent 协作架构"**——设计多个 Agent 之间怎么通信、怎么调度、怎么决策、怎么隔离。这一整章,就是那条能力的落地。

- **架构设计能力**在这里的体现 = 角色边界(Lead / Worker / Gatekeeper 各管什么)、协调结构(谁拍板、决策权在谁手里)、隔离设计(独立上下文 / 独立副本 / cache-safe)。
- 这些都不是"功能",而是 PM 在设计一支 Agent 团队时必须拍板的**架构决策**。

一句话收束:

> **多 Agent 不是"人多力量大",而是"分工清楚、协调到位、隔离干净"才力量大。** 设计一支能协作的 Agent 团队——什么时候拆、谁来协调、谁说了算、怎么不互相踩——考的正是 PM 的架构设计能力。这也是 Harness 时代的 PM 和传统 PM 的又一处分水岭:传统 PM 设计一个功能,Harness PM 设计一支能自主协作的智能团队。
