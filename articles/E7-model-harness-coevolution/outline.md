# E7 自进化篇新版大纲

> 对外标题：《AI 产品经理读懂 Harness》(7/9) · 自进化篇——怎么让 Agent 越用越聪明
> 新主线：**把失败变成资产**。

## 0. 开场：为什么 Agent 用了一万次也没变聪明

场景：

- 今天帮你改一个表格，绕了 5 次才成功。
- 明天换个类似表格，它又从头试错。
- 用户以为“AI 会学习”，实际系统只把最后答案给了你，过程里的坑全丢了。

开场判断：

> 大多数 Agent 不会越用越聪明，不是因为模型不努力，而是因为产品没有设计“长记性”的机制。

引出定义：自进化不是魔法，而是把失败、接管、复盘、验证，变成下次能复用的资产。

---

## 1. 什么是真正的自进化：不是“自己改自己”，而是“失败资产化”

### 1.1 三种伪自进化

- 只把聊天记录存起来：知道发生过，不代表下次会做。
- 只写一句“下次注意”：没有触发条件和验证，下次还是不会用。
- 只盯评测分上涨：可能只是会刷题，不是真实变强。

### 1.2 真自进化的 6 步

```text
看见：记录过程、失败点、用户接管点
判断：分清是信息不够、工具错、规则缺、技能没触发，还是模型不会
沉淀：放到正确层级，变成记忆、规则、技能、工具说明或评测题
验证：用旧题和新题验证有没有真变好
发布：灰度进入正式能力库，保留回滚
复用：下一次类似任务能自动找到并遵守
```

一句话：**没进验证集、没进能力库、下次用不上，都不叫自进化。**

---

## 2. 第一层：Skill 自进化——把一次踩坑变成下次的办事手册

### 2.1 Memory 和 Skill 的区别

- Memory：记住“用户不喜欢什么、项目叫什么”。
- Skill：记住“这类任务应该怎么办、先后顺序是什么、哪些坑不能踩”。

产品比喻：Memory 像个人偏好，Skill 像老员工留下的 SOP。

### 2.2 Skill 怎么从任务里长出来

结合 SkillOpt、Hermes、Cloud Agent Skill：

```text
复杂任务完成
→ 回看过程
→ 找到关键成功路径和踩坑
→ 写成 Skill
→ 标明触发条件、适用边界、验证方式
→ 下次命中类似任务时自动调用
→ 新坑继续更新 Skill
```

重点不是“自动生成一份文档”，而是“下次真的能调起来”。

### 2.3 产品经理要追问的 5 个问题

1. 这个 Skill 是从哪些真实任务沉淀来的？
2. 它什么时候会被触发？
3. 它不适用于什么场景？
4. 有没有验证它真的减少了失败？
5. 旧 Skill 会不会被合并、废弃、回滚？

## 3. 第二层：Meta-Harness——连“改 Harness”也可以被产品化

### 3.1 Meta-Harness 的大白话解释

它不是让 Agent 凭空变聪明，而是让一个“改进官”看过去所有方案、分数和执行过程，然后提出下一版更好的办事流程。

```text
上一轮方案 + 执行过程 + 分数 + 失败日志
→ 改进官查证据
→ 提出新版 Harness
→ 跑验证
→ 记录结果
→ 继续下一轮
```

### 3.2 硬数据

- 项目页展示：TerminalBench-2 小规模实验从 28.5% 提到 46.5%。
- 论文记录：最重场景每步可利用约千万级诊断材料。
- TerminalBench-2 中，改进者每轮中位读取 82 个文件。
- 数学推理任务中，对 5 个未参与搜索的模型平均提升 4.7 个点。

### 3.3 对 PM 的启发

自进化系统的第一资产不是模型，而是“过程证据库”：

- 每次任务怎么跑的？
- 哪一步失败？
- 哪个工具选错？
- 用户在哪里接管？
- 修复后有没有复测？

没有这些，所谓自进化就是拍脑袋。

## 4. 第三层：写经验和用经验，是两种不同能力

### 4.1 最容易被忽略的坑

很多团队以为“经验写好了，系统就变强了”。但真实问题可能是：

- 下次任务时没找到这条经验。
- 找到了，但没判断出该用。
- 用了，但没有遵守关键步骤。
- 经验太多、太乱，互相冲突。

### 4.2 论文结论的大白话

最新研究把自进化拆成两件事：

- 写经验：能不能从失败记录里写出有用改进。
- 用经验：真正做任务时能不能调用并遵守这些改进。

关键判断：

> 资源不应该只投给“更强的经验写手”，还要投给“任务执行时真的会用经验”。

### 4.3 产品设计落点

要设计：

- Skill 触发机制。
- 命中失败的反馈。
- Skill 使用后的效果对比。
- 冲突 Skill 的合并和废弃。
- “该用没用”的专门评测。

---

## 5. 第四层：Harness 和模型怎么形成双飞轮

### 5.1 短期：先让 Harness 变好

短期最现实的是改外部系统：

- 规则更清楚。
- 工具选择更稳。
- Skill 更好用。
- 上下文更完整。
- 评测集更贴近真实失败。

### 5.2 长期：把高质量过程变成训练资产

长期才进入模型训练：

- 成功路径。
- 失败纠偏。
- 用户接管。
- 高质量 Skill。
- 真实评测题。

但必须讲清两条红线：

- 用户数据授权和隐私边界。
- 训练数据和评测数据隔离，防止刷题。

### 5.3 三种路线

| 路线 | 适合谁 | 风险 |
|---|---|---|
| 轻训练、重 Harness | 个人项目、早期产品、快速验证 | 上限依赖外部模型 |
| 重数据、重训练 | 大厂、自研模型团队 | 成本高、合规重 |
| Harness 先自进化，成熟后再反哺训练 | 大多数 Agent 产品 | 需要长期数据治理 |

给 DeepSeek 桌面端 Agent 的建议：先做第三条。

---

## 6. 自进化怎么不把自己改坏

### 6.1 最大风险

不是进化太慢，而是越改越脏：

- Skill 越来越多但没人清理。
- 规则互相打架。
- 为了评测分学会刷题。
- 安全边界被自动改松。
- 提示越来越长、成本越来越高。

### 6.2 生产发布链路

```text
候选改动
→ 评测通过
→ 高风险人工审核
→ 小范围灰度
→ 线上观察
→ 可一键回滚
→ 正式进入能力库
```

红线：

> Agent 可以建议改进，但不能直接改正式能力，尤其不能直接改安全规则。

---

## 7. PM 自进化方案评审清单

拿到任何“自进化 Agent”方案，直接问：

- [ ] 它记录的是最后答案，还是完整过程？
- [ ] 失败有没有分类归因？
- [ ] 经验沉淀到哪一层：记忆、规则、Skill、工具说明、评测题，还是训练数据？
- [ ] 下次任务怎么触发这条经验？
- [ ] 有没有验证“用了经验以后更好”？
- [ ] 有没有发现“该用没用”的机制？
- [ ] 旧经验怎么合并、废弃、回滚？
- [ ] 高风险改动谁审核？
- [ ] 训练数据和评测数据是否隔离？
- [ ] 用户数据是否有授权和脱敏？

---

## 8. 收尾金句

> Agent 自进化不是让 AI 自己长脑子，而是让产品系统长记性。每一次失败，如果没有变成评测题、规则、Skill 或训练资产，它就只是一次浪费；只有能被验证、能被复用、能被治理的经验，才叫真正的进化。
