## 开篇:别只答"看准确率"

E7 反复强调,Harness 自进化靠的是"能评好坏"。可到底怎么科学地评好坏?这一篇专门讲。

先看个面试场景。面试官问:"你怎么评测一个 Agent?"——只会答"看准确率、看成功率"的,基本出局。因为 **Agent 是多步骤系统,最终答案对,过程可能全错**:

- 报告对了,但用了未授权的数据源;
- 代码能跑,但中间乱调了 30 次工具,贵到不可部署;
- 回答完整,但关键事实来自一次错误检索,只是碰巧蒙对;
- 任务成功了,只因为重试撞上了正确答案。

只看终点,这些全看不见。所以这章会反复强调一句:**Agent 评估不能只评 final answer,要评 trajectory(轨迹)**——看中间的计划、工具调用、权限判断、错误恢复、成本和最终结果。

而评好坏,本质就两件事:**用什么题考(评测集),按什么标准打分(评测标准)。** 但在这两件事之前,还有一个最容易被跳过的前置问题。

---

## Q0:你到底在评什么?——不是评模型,是评整条 Harness 链路

大多数人一上来就建评测集、看成功率。但**先得想清楚:你评的到底是什么?**

Agent 不是一个单点模型,而是一串系统——回顾前 7 章,它是:

```text
模型 + Prompt/Rules + 上下文组装 + 记忆 + 工具路由 + 工具
+ 编排 + 权限层 + 压缩/缓存 + 多 Agent + UI/人机回环
```

如果只看一个"最终成功率",一旦效果变差,你根本不知道该修哪一环——是模型退化、上下文没带对、工具调错、记忆污染、还是权限拦错了?(这正是 E7 第一节说的"先定位")。所以 Harness 评估的第一步,是**把评估对象拆开**:

| 评估对象 | 要回答的问题 | 典型指标 | 对应章 |
|---|---|---|---|
| 模型能力 | 模型本身会不会推理/生成/理解 | 单步正确率、推理质量 | — |
| 上下文管理 | 该带的信息带对了吗 | context recall、无关上下文比例、compact 后信息保留率 | E4 |
| 工具调用 | 选对工具、参数对吗 | 工具选择准确率、参数合法率、工具成功率 | E3 |
| 记忆系统 | 记的东西对吗、有没有污染 | 记忆 precision/recall、过期记忆率 | E4 |
| 权限安全 | 拦住高风险动作了吗 | 越权率、误拦率、注入拦截率 | E6 |
| 多 Agent 编排 | 拆得对吗、协作有效吗 | 冲突率、合并成功率、协调开销 | E5 |
| 整体任务 | 用户目标完成了吗 | 任务成功率、采纳率、人工接管率 | 全链路 |
| 成本体验 | 能不能上线 | token、延迟、缓存命中率、单任务成本 | E4/E5 |

> **一句判断**:**Harness 评估不是给模型打分,而是给一条"理解 → 取上下文 → 调工具 → 执行 → 验证 → 交付"的系统链路打分。** 先想清楚这次评的是哪一环,效果掉了才知道该回哪一章去修。这也是为什么 E8 是全系列的收口——它把 E1-E7 每一环都变成了可被衡量的对象。

# 支柱一 · 评测集:用什么题考

## 一、评测集从哪来、怎么建得合理(Q1)

两种常见错误:① 没有评测集,凭感觉上线;② 直接拿公开榜单(GAIA/AgentBench/SWE-bench)当评测集。

**公开榜单的正确用法**:它们是"入门锚点"——对标行业水位、防自己刷榜自嗨;但**垂直业务必须自建**,因为公开集覆盖不了你的真实用户分布。

**自建评测集,三路并用**:

| 来源 | 作用 | 说明 |
|---|---|---|
| 真实采集 | 地基 | 从线上真实日志捞 case,最贴近生产分布 |
| 故障注入 | 补边界 | 主动构造异常/边界 case |
| 模拟生成 | 补长尾 | 生成低频但重要的场景 |

**真实案例(带数字)**:阿里云智能运维故障根因 Agent——自建覆盖"输入/代码/资源/状态"四维评测集,三路并用;2000+ 原始案例、200+ 已发布;首批 200 案例根因召回 87.5%、定位准确率超 80%。

**一条 case 到底长什么样(Case Schema,给可抄)**:光说"建评测集"不够,得知道一条样本怎么落表:

```json
{
  "case_id": "car_voice_0421",
  "domain": "车载语音",
  "scenario": "高速行驶中、带方言、说到一半被打断",
  "input": "用户原始语音转写 + 上下文(车速/是否有乘客)",
  "expected": "正确意图 + 应执行的动作 + 不该执行的动作",
  "eval_dims": ["意图识别", "响应延迟", "安全(分心驾驶)", "是否误触发"],
  "ground_truth": "人工标注的标准答案",
  "difficulty": "hard",
  "tags": ["方言", "多轮中断", "L2舒适控制"]
}
```

每条带 **难度等级 + 评分维度 + ground truth**,且要"Golden 标准答案 + 真实采样 + 反例"三种都有。

**评测集要分层(给可抄)**:不是一坨题,按用途分几层,各管各的:

| 层 | 用途 | 何时跑 |
|---|---|---|
| Smoke(冒烟) | 最小核心集,快速过一遍 | 每次改动 |
| Regression(回归) | 历史 badcase 沉淀的题(呼应 E7) | 每次改动 |
| Domain(领域) | 覆盖核心业务场景 | 定期 |
| Adversarial(对抗) | 攻击/边界/注入 | 定期/安全相关改动 |
| Long-horizon(长程) | 多步骤长任务 | 重大改动 |
| Holdout(留出) | 永不用于训练,防污染 | 独立验证 |

**车载差异化(我所在领域的真实考量)**:车载 Agent 不能套 SWE-bench,必须自建覆盖语音识别、方言、多轮中断、分心驾驶的 Benchmark——这些公开集里完全没有。

> **一句判断**:"先评测集,再上线"是生产心态。公开榜单告诉你"行业到哪了",**自建、分层、带 schema 的评测集才告诉你"你的用户在你这儿过得好不好"。**

## 二、评测集怎么不过时、不被污染(Q2)

两个坑:**过时**和**被污染**。

**会过时**:评测集建完三个月,业务/用户/场景变了,它就和现实脱节。最危险的信号——**评测分还在涨、线上却在崩**,因为你在考一套过期的题。
破解靠 **Look at your data**(Hamel Husain 三年心法):定期(每周)拿 20-50 条真实线上日志亲自看,而不是只盯分数。分数是"薛定谔的好看",真实 case 才告诉你哪里真出问题。
配套机制(呼应 E7):每发现一个新 badcase,就加进 Regression 层变成回归题——评测集跟着真实数据自我增长。还要**按场景分桶抽样**(别只看总分,看每个分桶——总分涨了但车载方言桶在掉,你得知道)。

**被污染**(呼应 E7 数据飞轮):训练数据绝不能污染评测集,Holdout 必须独立。否则"分数提升"可能只是模型把测试题背下来了。

> **一句判断**:评测集是活的。**不"看真数据"会过时,不做"训练/评测隔离"会自欺,不"分桶看"会被总分掩盖局部退步。**

# 支柱二 · 评测标准:按什么标准打分

## 三、除了成功率,按什么标准打分、怎么量化(Q3)

Agent 是多步骤系统,**只看最终答案会被骗**。评测标准必须同时覆盖结果和过程,可量化。一套 5 类指标:

| 类别 | 量化指标 | 回答什么 |
|---|---|---|
| 任务完成率 | 成功率、测试通过率、回归失败数 | 它干成没 |
| 过程质量 | 平均步数、工具调用正确率、上下文命中率、错误恢复次数、轨迹合理性 | 怎么干成的 |
| 成本 | token、缓存命中率、首 token 延迟、单任务耗时 | 划不划算 |
| 安全合规 | 危险操作拦截数、越权次数、注入拦截率、隐私合规 | 安不安全 |
| 用户满意度 | 采纳率、合并率、人工接管率、取消/重试率、赞踩 | 用户认不认 |

**轨迹级评估(这是 Agent 评估和传统评估最大的不同)**:不只给 final answer 打分,而是给整条轨迹的每一环打分——

- plan(计划):任务拆解合不合理?
- tool(工具):每一步选对工具、参数对吗?
- observation(观察):有没有正确读取并利用工具结果?
- recovery(恢复):出错后有没有恢复,还是一路错下去?
- policy(权限):高风险动作有没有过权限层?

LangChain 的轨迹评估器(Trajectory Evaluator)就是把"输入 + 最终输出 + 中间步骤"一起评——这样才抓得住"答案对但过程烂"。

几个关键:成本是硬约束不是加分项(高性能但资源密集 = 不可部署);不同业务权重不同(车载重"识别+延迟+安全",coding 重"测试通过+采纳+token");过程指标靠 Trace 拿(接第五节)。

> **一句判断**:评测标准不是一个"成功率"数字,是"结果 + 过程(轨迹)+ 成本 + 安全 + 满意度"的多维仪表盘。**只看成功率,等于开车只看车速、不看油表和方向——迟早出事。**

## 四、谁来打分:人工 / 程序 / LLM-as-Judge(Q4)

三种打分员各有位置:
- **程序化判断**:能确定性判的优先(测试通过、regex、SQL 结果)——最可靠最便宜,能用就用。
- **人工**:ground truth(金标准),但贵、不可扩量。
- **LLM-as-Judge**:可扩量,但有噪声、可能不准。

**LLM-as-Judge 什么条件下能信(放我家 91.9% 实战)**:三件套缺一不可——结构化 Benchmark + 多评审投票 + 人工抽样校准;**先小批量人工标注,验证一致性达 90%+ 才能上规模**(实战参考线)。

**哪些绝不能交给 LLM judge 当最终裁判(禁区)**:
- 安全红线(越权、注入)——必须程序判/人工;
- 数值精确(算对没)——程序判;
- 代码能否运行——跑测试,不是让 LLM 猜;
- 权限合规最终裁决——不能交给概率性的 LLM。
LLM judge 适合的是**语义质量、摘要质量、对话自然度、方案完整性**这类主观多维 rubric 打分。

**降噪**:同一 Skill 同一 GT,LLM judge 跑 4 次分数 0.79~0.92 漂移。解法:多次采样取均值、**pairwise 对比优先于绝对打分**(只比 A/B 谁好,比给绝对分稳,更适合版本对比)、固定 judge prompt 版本、关键 case 程序判优先。**judge 分数提升 0.02 不一定代表更好**——要看噪声、方差、置信区间、样本量,以及是否在关键分桶上也提升。

**裁判要被元评估**:裁判是 LLM 时,要有 ground truth 集定期验证它准不准(芝麻信用"评测 Agent"机审率 80%+ 靠元评估兜底)。

**瑞士奶酪模型(Anthropic 官方)**:任何单层评估都有洞,多层叠加(程序判确定性的 + LLM 扩量主观的 + 人工抽样校准)才补得上。

> **一句判断**:能程序判的别用 LLM,主观的才交给校准过的 LLM,安全/数值/代码/权限这些红线绝不交给 LLM 当裁判。**一把没校准的 LLM-as-Judge,就是没刻度的尺子——量得越多,错得越自信。**

# 决策落地 · 怎么用评测做上线/迭代决策

## 五、离线过了线上怎么验证:A/B 多步骤挑战 + Replay + 可观测(Q5)

完整的评估 Sequence:

```text
离线评测集(上线前拦明显问题)
 → 灰度(小流量先试)
 → 在线 A/B(真实用户对比新旧)
 → 持续回归(每次改动跑回归集)
```

**A/B 在多步骤 Agent 的特殊挑战(AI PM 面试高频)**:传统网页 A/B 一次交互一个结果;Agent 一次任务跑 20 步、每步影响结果——
- **实验单位不能按 step**,要按"任务/会话/用户/项目/设备"(车载就是按车辆),否则一个任务的 20 步被当 20 个独立样本,严重高估样本量;
- 样本量、置信区间、显著性检验算法都要重算;
- 有**状态污染**:记忆和权限状态会跨任务影响,A 组的记忆可能污染后续——要隔离;
- 成本和安全要和质量一起看,结果还可能延迟产生。

**Replay Evaluation(Harness 评估里很专业的一块)**:把历史任务在**固定上下文 + 固定工具返回**下重放,对比新旧 Harness 的行为差异。它特别适合评估 prompt、tool router、context builder、多 Agent workflow 这类**系统改动**——因为不需要真实用户参与,可以离线把上万条历史 trace 重放一遍,得到新版本的预期表现(呼应 E7 的数据回流模式 E)。

**为什么离线过了线上还可能崩**:真实输入更脏、工具状态不同、用户打断更多、长尾没覆盖、线上延迟影响体验、评测集过拟合。所以离线分数高 ≠ 线上好用,必须走完整链路。

**可观测三支柱升级(AWS 秘籍七)**:传统 Metrics → Logs → Traces 只回答"发生了什么",Agent 黑盒必须加三层——决策原因(为什么选这个动作)、行为链条(完整 think-act-observe 轨迹)、结果质量(每步产出好不好)。Trace 该记什么(脱敏后):task_id、user_input、context 引用、plan、tool_calls/args/results、policy_decisions、model/prompt 版本、memory 读写、errors、retries、cost、latency、final_output、user_feedback。

> **一句判断**:离线分数高 ≠ 线上好用。**A/B 要按 Agent 多步骤特性重设实验单位、防状态污染;系统改动用 Replay 离线验证;可观测要看见"为什么",不只"发生了什么"。**

## 六、拔高:评测是产品的方向盘和仪表盘

回到 E1 的"双核能力"。评测集 + 评测标准,正是"评估能力"的两根支柱;而评估又是 E7 自进化飞轮的**引擎**——没有"能评好坏",E7 的错误固化、门控、回归、防刷分全转不起来。

把整章连起来:
- **先定对象(Q0)**:评的不是模型,是整条 Harness 链路——这把 E1-E7 每一环都变成了可衡量的对象;
- **评测集 = 用什么题考**(架构设计:覆盖真实分布、分层、不断生长的题库);
- **评测标准 = 按什么标准打分**(评估能力:多维量化 + 轨迹级 + 校准过的裁判 + 多层叠加);
- **决策落地 = 让评测真正驱动上线/迭代**(离线→灰度→A/B→回归 + Replay + 可观测)。

一句话收束:

> **评估不是给 Agent 打一个分,而是把真实任务拆成可复现的题、可校准的标准、可追踪的轨迹和可执行的上线决策。只有这样,团队才知道每次迭代到底是模型变强了、工具变准了、上下文变好了,还是只是分数变漂亮了。一个不会评测的团队,只能凭感觉做 AI 产品;一个会评测的团队,能让产品每天踏在实地上自动变好。**
