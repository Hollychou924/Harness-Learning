## 核心结构:评好坏 = 两根支柱 + 决策落地(2026-06-07 按周浩拆法重排)

> 周浩的判断:评测核心就两件事——**用什么题考(评测集)、按什么标准打分(评测标准)**。本章以这两根支柱为主轴,再加"怎么用评测做上线/迭代决策"的落地,把之前散的 5 问(指标/LLM-as-Judge/从零建集/A-B/可观测)系统化地装进去。

**支柱一 · 评测集怎么建得合理**
1. **Q1:评测集从哪来、怎么建?** — 三路并用:真实采集 + 故障注入 + 模拟长尾(阿里云智能运维 2000+ 案例)。"先评测集再上线"是生产心态。行业参考:GAIA/AgentBench/SWE-bench 只是入门锚点,垂直业务必自建(车载要自建覆盖语音/方言/多轮中断/分心驾驶)。
2. **Q2:评测集怎么不过时、不被污染?** — Look at your data(每周看真实数据)、每发现一个新问题就加进评测集(呼应 E7 错误固化)、训练/评测隔离防污染(holdout)。

**支柱二 · 评测标准怎么定得合理、可量化**
3. **Q3:除了成功率,按什么标准打分?怎么量化?** — Agent 是多步骤系统,不能只看最终答案,要看过程。5 类指标量化:任务完成率 / 过程质量(步数·工具调用正确率·轨迹)/ 成本(token·延迟)/ 安全合规 / 用户满意度。不同业务权重不同;高性能但资源密集=不可部署。
4. **Q4:谁来打分?人工 vs LLM-as-Judge 怎么保证准?** — LLM-as-Judge 放我家 91.9%,但三件套缺一不可:结构化 Benchmark + 多评审投票 + 人工抽样校准(90%+ 一致才上规模)。噪声问题(0.79-0.92 漂移);裁判要被元评估;瑞士奶酪——任何单层都不可信,多层叠加才可靠。

**决策落地 · 怎么用评测做上线/迭代决策**
5. **Q5:离线评测过了,线上怎么验证?A/B 在多步骤 Agent 怎么做?可观测怎么补?** — 评估 Sequence:离线 → 灰度 → 在线 A/B → 持续回归。A/B 多步骤挑战(一次跑 20 步、每步影响结果,样本量/置信区间算法不同)。可观测三支柱升级:传统 Metrics/Logs/Traces 只回答"发生了什么",Agent 必须加"决策原因/行为链条/结果质量"三层。

> **拔高(收尾)**:评测集 + 评测标准 = E1 双核里"评估能力"的两根支柱,也是 E7 自进化飞轮的引擎——没有评估,自进化无从谈起。评测不是写报告,是让产品每天自动变好的反馈系统。
