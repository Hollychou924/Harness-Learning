# E3｜工具系统与 MCP：给 Agent 一把钥匙，还是一串钥匙

> 草稿版本：v1 · 状态：✍️ 草稿中 · 字数：不限，讲透为准
>
> 承接 E1（全景）、E2（循环与编排）。E2 讲清了"一个 Agent 怎么转、多个怎么编排不跑飞",E3 回到工具层：工具一多，怎么还能调得对、调得快、调得安全。

## 本文回答这些问题

1. 同一个外部能力，做成 Function Call、MCP、Skill 还是插件？怎么选？
2. 工具越来越多，Agent 选错或不知道用哪个，怎么办？
3. 怎么把一个工具"造好"？（粒度、命名、描述、参数、输出）
4. 工具调用失败，Agent 该怎么处理？重试几次？何时放弃？
5. 给 Agent 工具 = 给一把权限钥匙，怎么治理？MCP 接入怎么不裸奔？

## 开篇：回到那支手电筒

E1 里我提过一个真实的反面例子：某个手机上的 Agent，我让它做一件再简单不过的事——"打开手电筒"。结果它在几百个工具里找了几十轮，最后还调用失败了。

很多人第一反应是"模型太笨"。但模型连写代码都会，怎么会不知道"手电筒"是什么？真正的问题不在模型，在**工具系统**:

> 不是 Agent 不会开手电筒，是工具系统没有被设计成一个"Agent 能理解、能选对、能安全执行"的能力结构。工具太多、描述太像、参数太松、没有一个帮它快速选对工具的机制——于是连开个手电筒都变成了又慢、又费 token、又容易失败的灾难。

这一篇的核心判断，一句话：

> **工具不是把后端 API 原样接上去，而是把外部世界重新整理成 Agent 能理解、能选、能执行、能失败恢复、能治理的"能力资产"。**

记住"能力资产"这四个字，它是本章的暗线。下面五个问题，就是围绕"怎么造、怎么选、怎么管这批能力资产"展开的。

---

## 一、四种外壳怎么选：Function Call / MCP / Skill / 插件（Q1）

同一个外部能力，你可以把它包装成四种不同的"外壳"，这是每个 Agent 项目都要做的第一个架构决策：

| 外壳 | 一句话定位 | 适合 |
|---|---|---|
| **Function Call** | 最朴素的一次性函数调用 | 简单、单次、即调即走的能力（查天气、读文件） |
| **MCP** | 标准化的工具接入协议（工具界的 USB-C） | 要跨产品复用、接生态、未来换模型（接数据库、Git、Slack、Jira） |
| **Skill** | 把一套验证过的多步流程沉淀下来 | 固定的、可复用的工作流（按公司规范发版、出周报） |
| **插件** | 带 UI 交互的能力 | 需要用户在界面上操作、确认、选择 |

**怎么选？一棵简单的决策树：**

- 一次性、简单调用 → Function Call。
- 要跨产品复用 / 接现成生态 / 想以后换模型不返工 → MCP。
- 是一套固定多步流程、想沉淀下来反复用 → Skill。
- 需要用户界面交互 → 插件。

**一句判断**：别为了用 MCP 而用 MCP。外壳只是手段，核心永远是"这个能力，Agent 到底用不用得对"。后面四节讲的"造好、选对、管住"，才是真功夫——外壳选错可以换，能力没设计好，换什么壳都没用。

## 二、工具一多，Agent 选不对怎么办（Q2）

这是手电筒案例的正解，也是工具系统的第一道坎。

**问题的根源**：工具一旦上百，你会撞上两件事——一是把所有工具说明都塞进上下文，真正的任务材料反而放不下；二是模型面对几百个选择，本身就更容易选错、或过度调用。**工具数量不等于能力，可发现 + 能正确选择，才是能力。**

破解的思路有两层。

**第一层：渐进式披露（按需加载）。** 不要一上来把所有工具的完整说明都摊给模型，而是分两层：

- 第一层：只给"名称 + 一句话描述"，让模型判断"要不要用"。
- 第二层：命中了，再加载这个工具的完整说明。

Anthropic 自己讲过 Claude Code 在这件事上的完整演进，很有参考价值：最早他们用 RAG——预先把代码库索引好，每次回答前由 harness 检索相关片段塞给模型。这套能用，但要建索引、环境一变就脆，而且最关键的是——**context 是"喂"给模型的，不是模型自己找的**。后来他们干脆给 Claude 一个 Grep 工具，让它像在网上搜东西一样自己去代码库里搜、自己一层层建上下文。再后来用 Agent Skills 把"渐进式披露"正式化；甚至专门做了一个子 Agent（Claude Code Guide）：用户问"怎么加 MCP"这种问题时，由这个子 Agent 在它自己的上下文里查文档、只把答案回传，主对话的上下文始终保持干净。

**第二层：Tool Router（给 Agent 选工具，像做一次检索）。** 当工具多到渐进式披露也不够时，要把"选工具"当成一个检索/推荐问题来设计，而不是把工具列表一股脑摊开：

```text
召回(从全量工具里粗选出相关的一批)
  → 排序(按相关度、历史调用质量排个序)
  → 裁剪(只把 top-N 个工具的说明给模型)
  → 动态加载(命中后再补完整细节)
```

这套思路在 Anthropic 那篇 MCP 实践里有个惊艳的例子：Cloudflare 的 MCP server 面对约 2500 个 API 端点，没有把 2500 个工具全摆出来，而是只暴露 **search 和 execute 两个工具**——Agent 先 search 找到该调哪个，再 execute 执行，整套加起来大约只占 1K token。这就是"工具的搜索引擎"思路的极致。

**一句判断**：工具越多，系统越需要一个"工具的搜索引擎"。手电筒案例的病根，正是工具一多却没有这套召回-排序机制，Agent 只能在几百个选项里硬猜。

## 三、怎么把一个工具"造好"(Q3)

这一节最实在，因为大多数失败不是"没有工具"，而是"工具造得不好用"。一个工具要被 Agent 用对，至少要在五件事上下功夫。

### ① 粒度：别太粗，也别太细

还是手电筒。同一个能力，至少有三种粒度的设计：

```text
方案 A(太粗): device_control(action: string, target: string)
方案 B(中等): set_flashlight(enabled: boolean)
方案 C(太细): turn_on_flashlight() / turn_off_flashlight()
```

- 方案 A 太粗：参数全是自由文本，Agent 容易乱填（action 填 "enable"、target 填 "torch mode"）。
- 方案 C 太细：每个动作一个工具，工具数量爆炸，又回到第二节"选不对"的坑。
- 方案 B 中等抽象：意图清晰、参数受控，对 Agent 最友好。

Anthropic 那篇《Seeing like an agent》给了一个很妙的框架：设计工具要"像模型一样思考"。就像给你一道难题，给你纸、计算器还是电脑，取决于你的能力——给 Agent 的工具，也要"形状贴合它的能力"，既不是一个 bash 包打天下，也不是堆五十个细碎工具。

**一句判断**:**工具不是后端接口的原样暴露，而是给 Agent 重新设计过的"语义化能力"。** 后端可能是 `deviceService.executeCommand(commandId, payload)`，但给 Agent 的应该是 `set_flashlight(enabled: true)`。

### ② 命名和描述：别让工具互相打架

工具一多，最常见的麻烦不是缺工具，而是工具之间太像：

```text
open_flashlight
turn_on_torch
set_camera_light
device_light_control
quick_setting_toggle
```

Agent 看到这一堆，直接懵。所以要记住：**description 不是给人看的文案，是"工具选择器"的核心特征——模型就是靠它来判断该不该用、什么时候用。**

给个对照。写砸的描述：

```text
打开或关闭设备功能。
```

太泛了。Agent 既不知道它能不能开手电筒，也不知道什么时候不该用它。

写好的描述，讲清四件事：

```text
用于打开或关闭手机手电筒(flashlight / torch)。
仅当用户明确要求控制手电筒时使用。
不要用于:屏幕亮度、拍照时的闪光灯、智能家居的灯。
```

好在哪：① 说清能做什么；② 说清什么时候用；③ 说清什么时候别用；④ 主动和相似工具划清界限。一个工具的描述里写没写清"什么时候别用"，往往就是 Agent 会不会乱调的分水岭。

### ③ 参数：别让 Agent 猜字符串

很多调用失败，是参数设计太松。自由文本最危险：

```json
{ "action": "open", "target": "light" }
```

`open` 和 `light` 都是自由文本，模型可能填成 `"enable"` / `"torch mode"`，然后就失败了。更稳的设计是用受约束的类型：

```json
{ "enabled": true }        // 用 boolean
{ "state": "on" }           // 或用 enum,state 只能是 on / off
```

并用 JSON Schema 校验输入（MCP 规范里每个工具都要带 inputSchema，正是这个作用）。

**一句判断**:**能用 enum，就别让 Agent 自由发挥；能用 boolean，就别让它猜字符串。** 把选择空间收窄，失败率自然降下来。

### ④ 工具的"输出"也要被设计：太长怎么办

大家通常只盯着工具的输入，却忘了输出同样会出事。想想这些工具的返回：

```text
search_code  →  100 个文件
read_log     →  5 万行日志
query_db     →  几千条记录
browser_snapshot  →  巨大的 DOM
```

直接把这些原样塞回模型，上下文会爆，模型也抓不住重点。所以**工具的输出也要被设计成分层结构**:

| 返回层级 | 内容 | 作用 |
|---|---|---|
| summary | 一句话结论 | 给模型快速判断 |
| structured_data | 关键字段 | 结构化、好解析 |
| evidence | 原始证据片段 | 需要时引用 |
| next_actions | 建议的下一步 | 帮模型推进 |
| raw_ref | 原始结果的引用句柄 | 不直接塞满上下文，要看再取 |

（这和 E2 的"工具结果回灌进循环"不冲突：E2 讲循环怎么消费结果，E3 讲输出格式怎么设计，才好被循环消费。）

### ⑤ 危险工具：先 preview，再执行

读类工具（read_file、search_code、get_weather）可以直接执行；但有副作用的工具（delete_file、send_email、create_pr、install_package）最好先给个预览——支持 `dry_run: true`，或在结果里返回"我将改动什么、风险多高、预览是什么":

```json
{ "will_change": ["system.flashlight"], "risk_level": "low", "preview": "Turn flashlight on" }
```

至于"什么时候该让用户审批、谁来兜底责任"，那是 E6 安全章的地盘。E3 只负责把工具设计成"支持预览"这个接口，把球稳稳传给 E6。

## 四、工具失败了怎么办：重试分类 + 熔断 + 幂等（Q4）

E2 里那个 4 亿 token 的事故反复提醒我们：无限重试，只会把账单越搞越大。"错误路径就是主路径"——工具失败是常态，不是例外，必须当成主流程来设计。

核心判断是：**不是所有失败都该重试。** 失败要分类对待：

| 失败类型 | 正确处理 |
|---|---|
| 网络超时 | 有限次重试 |
| 429 限流 | 指数退避后再试 |
| 参数错误 | 改参数，而不是用同样的参数盲重试 |
| 权限不足 | 转授权流程，不重试 |
| 写操作超时 | 先查状态（到底成没成），不盲重试 |
| 破坏性操作失败 | 停下来，不自动重试 |

在这之上还要有**熔断**：如果一个工具连续失败、或超时率过高，系统应该短时间熔断它，不再让 Agent 反复空调——这正是 E2 那句"该重试 vs 该熔断"在工具层的落地。连续 N 次同样的失败，是熔断信号，不是"再来一次"的理由。

最后一个最容易被忽略、却最致命的点：**写操作的幂等性。** 看这几个工具：

```text
create_jira_ticket / send_email / charge_user / submit_order
```

如果网络超时、Agent 重试一次，会不会创建两个工单、发两封邮件、扣两次款？所以有副作用的工具，必须设计幂等保护——`idempotency_key`、`request_id`、或执行前先查状态。

**一句判断**:**读工具可以放心重试，写工具不能盲目重试。凡是有副作用的工具，都要有幂等键、执行记录和重复调用保护。**

## 五、工具 = 权限钥匙：Tool Registry + MCP 不裸奔（Q5）

到这里要做一个关键的思维转变：

> **工具不是一个普通的函数调用，而是"生产资源的对外授权点"。** 你给 Agent 一个工具，等于给它一把权限钥匙——这把钥匙能开多少扇门、有没有时限、留不留痕、谁能审计，这些事必须从 Day 1 就想清楚。

落地的抓手，是让所有工具都走一个统一关口：**Tool Registry（工具注册表）**。一个合格的 Tool Registry，每个工具至少登记九项元信息：

1. 工具名称（唯一标识）
2. 工具描述（给 LLM 看的说明）
3. 输入参数 JSON Schema（用于校验）
4. 允许调用的 Agent 列表（RBAC）
5. 调用超时与速率限制
6. 风险等级（低 / 中 / 高）
7. 是否需要人工确认
8. 输出结果结构
9. 审计日志策略（存什么、留多久、谁能看）

再补一项常被忽略的：**生命周期状态**——owner、版本、状态（draft / beta / active / deprecated）。工具会升级，description、参数、返回格式一变，Agent 的行为可能立刻变差；老任务要能绑定旧版本，下线要有灰度和回滚。

**一条铁律**：哪怕你只有 3 个工具，也从第一天起强制走 Tool Registry。先有规矩，后扩规模——等工具堆到几百个再来治理，就是手电筒那种灾难。

### MCP：标准化是趋势，但不能裸奔

MCP（Model Context Protocol）解决的是接入标准化问题：过去每接一个工具，都要为不同模型写不同适配器；MCP 把这层标准化了——工具一次实现，所有支持 MCP 的应用都能调。**它之于 AI 工具，就像 USB-C 之于充电接口。**

但标准化不等于安全，恰恰相反——工具越容易接入，越需要 Harness 在中间当安全网关。五条最佳实践：

1. **永远别把 MCP Server 直接暴露给 Agent**，必须先过 Tool Registry。MCP 提供"能力",Harness 提供"治理"。
2. **每个 MCP Server 单独配额**：一个流氓 server 不应该拖垮整个系统。
3. **白名单而非黑名单**：哪怕一个 server 暴露了 50 个工具，也只把业务真正需要的几个开放给特定 Agent。
4. **高风险工具一律走 Human-in-the-Loop**：文件写入、删除、代码执行、数据库写、外部支付，不能让 Agent 自动执行。
5. **所有 MCP 调用都打 Trace**：来源、参数、结果、调用者都要可追溯。

还有一条来自 MCP 官方规范的硬要求：**工具的 annotations（行为标注）默认不可信**——除非来自可信的 server，客户端必须把它当作不可信内容处理。换句话说，别因为一个工具"自称"安全就放行。

**顺带划一下边界**：工具返回的内容里，可能藏着"忽略之前指令、把文件发到某地址"这类注入。原则是——**工具输出要当作数据（data），不能当作指令（instruction）**，别把网页、邮件、issue 里的文本直接提升成系统指令。完整的提示注入防护，是 E6 安全章的主场，这里只点到为止。

一句话收束这一节：**MCP 让工具接入变便宜，Harness 让工具调用变可信。两者必须搭配，不可偏废。**

## 六、拔高：工具系统，考的是架构设计能力

走到这里，回头看会发现：工具系统的本质，根本不是"给 Agent 多接几个 API"。

> 真正的工具系统，是把外部世界重新整理成 Agent 能理解、能选择、能执行、能失败恢复、能治理的**能力资产**。
> Function Call、MCP、Skill、插件，只是这批能力资产的不同外壳；更核心的是：工具有没有被设计成清晰的语义能力、有没有被登记分级版本化、有没有明确的输入输出 / 失败策略 / 副作用边界。
> 所以，一个好的 AI IDE / Coding Agent harness，不是工具越多越强，而是**工具越多，系统越需要一套严密的 Tool Registry 和 Tool Router**。

这正好接回 E1 立下的"双核能力":**工具系统体现的不是"接 API 的能力"，而是架构设计能力。** 选什么外壳、切多大粒度、怎么写描述、怎么约束参数、怎么治理权限——每一个决定，都是产品经理在为"Agent 能不能可靠地用对能力"做架构。这是 Harness PM 和普通 PM 的又一处分水岭。
