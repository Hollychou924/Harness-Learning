## 0. 开篇钩子(约 400 字)

- 承接 E1: E1 说"Harness 是 Agent 的操作系统",**循环就是这台机器的 CPU 怎么一拍一拍执行,编排就是多任务怎么被调度**。任务能不能"跑起来"且"不跑飞",全看这一层。

# 第一幕 · 循环:单个 Agent 怎么转

## 1. 什么是 Agent 循环:定位 + 本质(约 700 字)= Q1

- **本质一句话**: Agent 循环 = 单个 Agent "想 → 做 → 看 → 再想"不断重复,直到完成或交还控制权。它就是一个带工具调用的 `while` 循环——不是普通问答。
- **三组边界,各一句话(不展开)**:
  - 循环 vs 工具调用: 工具调用是循环里"做"的那一下;循环是把"想-做-看"串起来反复转。
  - 循环 vs 编排: 循环是"一个 Agent 怎么转"(微观);编排是"多个步骤/多个 Agent 谁先谁后、谁调谁"(宏观)。本章先讲循环,再讲编排。
  - 循环/编排 vs Harness: 它们是 Harness 这台操作系统里的"执行+调度"模块,对应 E1 七层里贯穿 E/T/L 的主循环。
- 引 `entities/codex.md`(模型+工具+状态的循环)、`topics/tencent-*`(Agent 本质=while 循环+工具调用)。

## 2. 主循环怎么设计:TAO + 三件必须想清的事(约 1100 字)= Q2【新补 · JD 正面命中】

- **TAO 骨架**: Think(决定下一步)→ Act(调工具/改文件)→ Observe(读结果)→ 回灌进上下文 → 再 Think。配 Mermaid 主循环图。
- **真实锚点**: Codex 官方原话——"在用户输入、模型推理、工具调用、工具结果回填之间循环,直到生成最终回复或交还控制权"。
- **设计一个能用的主循环,必须想清三件事**:
  1. **循环什么时候该停?** 正常终止(产出最终结果/交还控制权)vs 异常终止(到达上限)。这里埋下第 5 节"四道硬闸"的引子——停止条件不能只靠模型自觉。
  2. **工具结果怎么回灌进下一轮?** Observe 到的结果(成功输出/报错/空结果)要结构化地塞回上下文,让下一次 Think 看得到。回灌质量直接决定会不会"原地打转"。(注: 工具结果的格式与回流细节属 E3,本节只讲"回灌进循环"这件事。)
  3. **单步怎么不跑偏?** 每一轮都要能对齐原始目标(避免越走越远)、能识别"这步没进展"(避免重复同一个失败动作)。
- **原创判断**: 循环本身好写,难的是给它装上"刹车、油门、仪表盘"——也就是停止条件、结果回灌、防跑偏。**一个不会停、看不见结果、不知道自己有没有进展的循环,就是 4 亿 token 事故的起点。**
- 引 `entities/codex.md`、`topics/tencent-*`。

## 3. 边做边想 vs 先出计划(约 800 字)= Q3

- **两条路线**:
  - ReAct(边想边做): Think→Act→Observe 灵活推进,适合探索型任务;缺点: 易"走着忘了目标"、易死循环。
  - Plan-and-Execute(先计划后执行): 先产出计划再逐步落地,目标稳、可审查、可中途接管;缺点: 计划僵化时返工贵。
- **怎么选(原创判断)**: 任务越长、越要可控/可审计/可人工接管 → 越靠"先出计划";越短、越探索 → 越可"边做边想"。成熟产品多是**混合**: 先出计划,执行中允许局部 ReAct 微调。
- **声明式 vs 命令式计划**: Planner 最好输出"声明式计划"(打算做这几件事),而非"命令式调用"(直接执行),把"做不做、按什么预算做"的决定权留给 Harness(埋第 5 节伏笔)。
- 引 `concepts/quest-mode-agent-development.md`、`topics/agentway-harness-books.md`。

# 第二幕 · 编排:多个怎么协同不跑飞

## 4. 多 Agent 协调、隔离、冲突裁决(约 800 字)= Q4

- **前提声明**: 本节默认任务"已经拆好了"——该不该拆、拆几个、量化信号(读 >10 文件 或 ≥3 独立子任务、3-10 倍 token 值不值)全部见 **E5**。E2 只讲拆完之后怎么编排着协同跑。
- **三件事**:
  - 顺序/调度: Lead Agent 分配,Worker 并行执行,完成交回确认。
  - 隔离(不互踩): Worker 各自在独立代码副本(Git Worktree)干活。**硬原则: 并行 Agent 不共享可写状态。**
  - 冲突裁决(谁说了算): 结果不一致由 Lead/Gatekeeper 拍板;Gatekeeper 拒绝合入后,Worker 在原副本修复再提交。
- **真实反例**: 4 亿 token 第三轮——Worker 1 在副本里改了 8 文件 238 行,却被监控误判"卡死"强杀(隔离做了,但监控逻辑错,埋第 5/6 节)。
- 引 `topics/tencent-*`(Worktree、Lead/Worker)、`entities/claude-code.md`(多 Agent 取舍)。

## 5. 不跑飞:四道硬闸 + 谁踩刹车(约 1000 字)= Q5【全章高潮】

- **反面案例(全章脊梁)**: 4 亿 token Day 1 多米诺失败链——Watchdog 检测崩溃 → 诊断 Agent 跑满 20 轮没搞定 → 误判"已修复"重启 → 环境变量被继承连续 3 次拒启 → 连续失败保护退出 → Watchdog 自己也停了。配 Mermaid 时序图。
- **解法一: 四道硬闸(由 Harness 统一管,不让 Agent 自己判断要不要继续)**:
  - 最大步数 max_steps、最大 token max_tokens、最大时长 max_duration、最大工具调用 max_tool_calls。
  - ✅ 收尾清单硬指标: 给一组**可直接抄的保守默认值**(草拟: 单任务 max_steps 50、max_tool_calls 30、max_duration 30 分钟、max_tokens 按预算定),并说明怎么按任务长短上调。数字在 draft 定稿。
- **解法二: 谁踩刹车——"让 Agent 出主意,让 Harness 拿决定"**: 预算、并发、是否继续、是否重启,阀门握在 Harness 手里,不交给 Agent/Planner 自决。
- **原创判断**: 区分"该重试"和"该熔断"——同一工具连续 N 次同样失败 = 熔断信号,不是再试一次的理由。
- 呼应第 2 节: 这就是给主循环装上的"刹车"。
- 引 `topics/tencent-*`(4 亿 token Day1)。

## 6. 能恢复且用户安心:断点续跑 + 降低等待焦虑(约 1000 字)= Q6【PM 出彩节】

- **技术面——崩了怎么不从头来**:
  - 状态属于主业务(state 落在可持久化处,不在 Agent 脑子里)。
  - 线程/会话持久化(Codex: 线程/轮次/项目持久化)。
  - checkpoint: 关键步骤存档,崩溃后从最近检查点续跑。
  - 原创判断: "恢复的目标是继续工作,不是回到原状"——有些副作用回不去,关键是保住"执行叙事一致"。
- **产品面——让用户看得见、不焦虑(你最该出彩处)**:
  - 把"对话流"变"任务流": 任务卡片(运行中/等待确认/已完成)、执行计划可见、推理过程可见、决策点(放行/改方向/打回)。
  - 实时流式进度(每做一步写一行),既降焦虑也是第 7 节/监控的基础。
  - 原创判断: 用户焦虑不来自"等得久",来自"不知道它在干嘛、还要多久、能不能插手"。**可见性本身就是产品力。**
- 引 `concepts/quest-mode-agent-development.md`(对话流 vs 任务流)、`entities/codex.md`(线程持久化、思考流/执行流穿插、随时插队)。

## 7. 划算且算得清:编排控成本 + 好编排自查(约 700 字)= Q7

- **控成本(真实数字)**: 多 Agent 往往 3-10 倍 token;4 亿 token 里 92% 被低成本模型吃掉、真正调贵模型不到 2.5%。**关键: 便宜模型干粗活、贵模型干关键决策;别让循环无意义空转(回到四道硬闸)。**
- **什么算好编排——一张轻量自查表(深度评估见 E8)**:
  - [ ] 循环有没有明确停止条件?该熔断时熔断了吗?
  - [ ] 任务有没有"有效推进"(不是进程活着就算)?
  - [ ] 并行有没有隔离?冲突有没有明确裁决者?
  - [ ] 成本结构合理吗(贵模型只用在刀刃上)?
  - [ ] 用户看得见进度、能插手吗?
- 引 `topics/tencent-*`、`topics/agent-evaluation-system.md`(过程指标)。

## 8. 拔高 + 下期预告(约 500 字)

- **拔高——循环与编排对 PM 意味着什么(呼应 E1 双核)**:
  - 架构设计能力的体现 = 设计"谁出主意、谁拿决定、出事谁兜底"的调度结构(角色边界 + 流程编排 + 安全护栏,正是 E1 第六节那几条)。
  - 评估能力的体现 = 坚持用"是否有效推进"而非"进程是否活着"量好坏(直接对应 4 亿 token 监控教训)。
  - 一句话: **循环与编排,是 PM"双核能力"第一次真正落地的地方——你设计的不是功能,是一套让 AI 不失控的调度秩序。**
