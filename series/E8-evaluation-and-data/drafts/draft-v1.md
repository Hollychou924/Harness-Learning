## 开篇:别只答"看准确率"

E7 反复强调,Harness 自进化靠的是"能评好坏"——建评测集、跑回归、防刷分。可到底**怎么科学地评好坏**?这一篇专门讲这件事。

先看个面试场景。面试官问你:"你怎么评测一个 Agent?"——只会答"看准确率、看成功率"的,基本就出局了。为什么?因为 **Agent 是个多步骤系统,最终答案对,过程可能全是错的**:

- 最终报告对了,但中间用了未授权的数据源;
- 最终代码能跑,但 Agent 调了十几次无意义的工具,贵得离谱;
- 最终回答完整,但关键事实来自一次错误的检索,只是碰巧蒙对;
- 某次任务成功了,只是因为重试撞上了正确答案。

只看终点,你看不见这些。所以评估一个 Agent,核心其实就两件事:

> **用什么题考(评测集),按什么标准打分(评测标准)。**

把这两根支柱搭好,再加上"怎么用评测结果做上线/迭代决策",就是一套能驱动产品每天变好的反馈系统。一句话立住基调:

> **评测不是上线后补写的一份报告,是产品的方向盘和仪表盘。**

下面分三段:支柱一(评测集)、支柱二(评测标准)、决策落地。

# 支柱一 · 评测集:用什么题考

## 一、评测集从哪来、怎么建得合理(Q1)

两种最常见的错误做法:① 根本没有评测集,凭感觉上线、凭感觉说"这版更好了";② 直接拿公开榜单(GAIA / AgentBench / SWE-bench)当自己的评测集。

**先说公开榜单的正确用法**:GAIA、AgentBench、SWE-bench 这些是"入门锚点"——用来对标行业水位、防止自己刷榜自嗨(benchmark hacking)。但**垂直业务必须自建评测集**,因为公开集覆盖不了你的真实用户分布。你的 Agent 服务的是你的场景,不是 leaderboard。

**自建评测集,三路并用**:

| 来源 | 作用 | 说明 |
|---|---|---|
| **真实采集** | 地基 | 从线上真实日志里捞 case,最贴近生产分布 |
| **故障注入** | 补边界 | 主动构造异常/边界 case,补真实集采不到的极端情况 |
| **模拟生成** | 补长尾 | 生成低频但重要的场景 |

**真实案例(带数字)**:阿里云智能运维的故障根因定位 Agent,自建了覆盖"输入 / 代码 / 资源 / 状态"四个维度的评测集,真实采集 + 故障注入 + 模拟三路并用;积累 2000+ 原始案例、发布 200+;首批 200 个案例做到根因召回 87.5%、定位准确率超 80%。这个量级和方法,就是"认真建评测集"长什么样。

**车载的差异化(这是我所在领域的真实考量)**:车载 Agent 根本不能套 coding 的 SWE-bench——它要自建一套覆盖**语音识别、方言、多轮中断、分心驾驶**等输入维度的 Benchmark。"用户在高速上、带口音、说一半被打断"这种 case,任何公开集里都没有,只能自己建。这也说明一个通用道理:评测集的维度,必须长在你自己业务的真实输入上。

每条 case 怎么设计?带**难度等级 + 评分维度 + ground truth**,而且要"Golden 标准答案 + 真实采样 + 反例"三种都有,不能只放好做的题。

> **一句判断**:"先评测集,再上线"是生产心态,不是洁癖。公开榜单只能告诉你"行业到哪了",**自建评测集才能告诉你"你的用户在你这儿过得好不好"。**

## 二、评测集怎么不过时、不被污染(Q2)

评测集最大的两个坑:**过时**和**被污染**。

**会过时**:评测集建完三个月,业务变了、用户变了、场景变了,它就和现实脱节了。最危险的信号是——**评测分数还在涨,线上却在崩**。因为你在用一套过期的题考试,考得再好也没用。

破解靠一条心法,叫 **Look at your data**(Hamel Husain 总结的三年评估心法):**定期(比如每周)拿 20-50 条真实线上日志,亲自一条条看**,而不是只盯着分数。分数是"薛定谔的好看"——看着挺高,你不知道它好在哪、漏在哪;只有翻真实 case,才知道用户真正卡在哪里。

配套一个持续维护机制(正好呼应 E7 的错误固化):**每发现一个新的 badcase,就把它加进评测集,变成一道回归题。** 这样评测集会跟着真实数据一起自我增长,任何模型/Harness 改动都要在最新的评测集上回归通过。

**被污染**(呼应 E7 数据飞轮):训练数据绝不能污染评测集,holdout(留出集)必须独立。否则你看到的"分数提升",可能只是模型把测试题背下来了,不是能力真变强——这是自欺。

> **一句判断**:评测集是活的,不是建一次就完。**不"看真数据"的评测会过时,不做"训练/评测隔离"的评测会自欺。** 让评测集跟着真实数据一起长大,它才一直有效。

# 支柱二 · 评测标准:按什么标准打分

## 三、除了成功率,按什么标准打分、怎么量化(Q3)

回到开篇那句:Agent 是多步骤系统,**只看最终答案会被骗**。所以评测标准必须同时覆盖"结果"和"过程",而且要可量化。一套实用的 5 类指标:

| 类别 | 量化指标 | 回答什么 |
|---|---|---|
| **任务完成率** | 成功率、测试通过率、回归失败数 | 它到底干成没 |
| **过程质量** | 平均步数、工具调用正确率、上下文命中率、错误恢复次数、轨迹合理性 | 怎么干成的 |
| **成本** | token 消耗、缓存命中率、首 token 延迟、单任务耗时 | 划不划算 |
| **安全合规** | 危险操作拦截数、越权次数、注入拦截率、偏见/隐私合规 | 安不安全 |
| **用户满意度** | 采纳率、合并率、人工接管率、取消/重试率、赞踩 | 用户认不认 |

几个关键点:

- **"过程质量"最容易被忽略,却最能区分真假**。LangChain 的轨迹评估器(Trajectory Evaluator)就是把"输入 + 最终输出 + 中间步骤"一起评分,而不是只看输出——这样才抓得住"答案对但过程烂"的情况。
- **成本是硬约束,不是加分项**:一个高性能但资源密集型的 Agent = 不可部署。质量再好,单任务烧几美元,产品就跑不起来(呼应 E4 cache、E5 多 Agent token)。
- **不同业务,权重完全不同**:车载语音 Agent 重"识别准确率 + 响应延迟 + 分心驾驶安全";coding Agent 重"测试通过率 + 采纳率 + token 成本"。同一套指标,权重按场景调。
- **过程指标从哪来?** 靠 Trace(执行轨迹)——这就接到第五节的可观测。

> **一句判断**:评测标准不是一个"成功率"数字,是一套"结果 + 过程 + 成本 + 安全 + 满意度"的多维仪表盘。**只看成功率,等于开车只看车速、不看油表和方向盘——迟早出事。**

## 四、谁来打分:人工 vs LLM-as-Judge(Q4)

题目和标准有了,还有个问题:**谁来打分?** 三种打分员各有位置:

- **程序化判断**:能确定性判的优先(测试通过/不通过、regex 匹配、SQL 结果对不对)——最可靠、最便宜,能用就用。
- **人工**:是 ground truth(金标准),但贵、不可扩量。
- **LLM-as-Judge(用 AI 评 AI)**:可扩量,但有噪声、可能不准。

**LLM-as-Judge 什么条件下能信?** 大淘宝"放我家"智能导购给了实战参考:它的 LLM-as-Judge 做到了 **91.9% 的评判准确率**——但前提是**三件套缺一不可**:

1. 结构化 Benchmark(题目和评分维度都结构化,不是开放打分);
2. 多评审投票(多个 judge 投票,降低单个的偏差);
3. 人工抽样校准(人工抽查 judge 的打分准不准)。

而且有一条铁律:**先小批量人工标注,验证 LLM judge 和人工的一致性达到 90%+,才能上规模。** 这个 90%+ 是实战参考线。

**噪声问题别忽略**:同一个 Skill、同一份测试数据,LLM judge 跑 4 次,分数能在 0.79~0.92 之间漂移。你改了一行规则、分数从 0.85 到 0.87,根本分不清是真改进还是 judge 今天心情好。解法:多次采样取均值、用 pairwise 对比代替绝对打分、固定 judge prompt 版本、关键 case 优先用程序化判断。

**裁判本身要被"元评估"**:当裁判变成 LLM,你得有一个 ground truth 数据集,定期验证"这个裁判打分到底准不准"。芝麻信用的"评测 Agent"做到机审率 80%+,靠的就是用元评估给裁判兜底。

**瑞士奶酪模型(Anthropic 官方方法论)**:任何单层评估都有洞——单看人工(贵、慢、覆盖少)、单看 LLM(有噪声)、单看程序(只能判确定性的)。**多层叠加才可靠**:程序判确定性的、LLM 扩量主观的、人工抽样校准,三层的洞错开,才能补上。

> **一句判断**:能程序判的别用 LLM,能 LLM 扩量的别全堆人工;但 LLM 当裁判前,必须先用人工把它校准到 90%+ 一致、并持续元评估。**一把没校准的 LLM-as-Judge,就是没刻度的尺子——量得越多,错得越自信。**

# 决策落地 · 怎么用评测做上线/迭代决策

## 五、离线过了线上怎么验证:A/B 多步骤挑战 + 可观测(Q5)

评测集和标准都有了,最后一步:**怎么用它做上线和迭代决策?** 这是一条完整的评估 Sequence:

```text
离线评测集(上线前拦住明显问题)
 → 灰度(小流量先试)
 → 在线 A/B(真实用户对比新旧版本)
 → 持续回归(每次改动都跑回归集)
```

缺任何一环都不稳:只有离线没在线,你会被真实长尾打脸;只有在线没离线,明显的 bug 会直接冲给用户。

**A/B 测试在多步骤 Agent 场景的特殊挑战(AI PM 面试高频题)**:传统网页 A/B 一次交互一个结果,统计很直接;但 **Agent 一次任务要跑 20 步、每一步都影响最终结果**,这让 A/B 变复杂——

- 统计单元要按"任务"而不是"单步",否则一个任务的 20 步会被当成 20 个独立样本,严重高估样本量;
- 样本量、置信区间、显著性检验的算法都要重算,不能直接套网页 A/B 那套;
- 还要控制任务复杂度的分布,否则两组的任务难度不一样,对比就没意义。

**可观测三支柱升级(AWS 秘籍七)**:传统监控的 Metrics → Logs → Traces 三件套,在 Agent 场景只能回答"发生了什么",回答不了"为什么这么干"。Agent 是黑盒,必须额外加三层:

- **决策原因**:模型为什么选这个动作;
- **行为链条**:完整的 think-act-observe 轨迹;
- **结果质量**:每一步产出到底好不好。

把这两件事合起来,就是一套"离线 + 在线 + Trace + 反馈"四件套:离线冒烟+回归集(上线前)、在线 A/B(上线后)、Trace+Span(还原过程)、显式反馈(赞踩/重试)+ 隐式反馈(成功率/放弃率/用户在哪步离开)。

> **一句判断**:离线分数高 ≠ 线上好用——评测集再全,也覆盖不了真实世界的长尾。**必须走"离线→灰度→A/B→回归"全链路,A/B 要按 Agent 多步骤特性重设统计口径,可观测要能看见"为什么",而不只是"发生了什么"。**

## 六、拔高:评测是产品的方向盘和仪表盘

回到 E1 的"双核能力"。评测集 + 评测标准,正是那两个核心能力里"**评估能力**"的两根支柱;而评估,又是 E7 自进化飞轮的**引擎**——没有"能评好坏"这个引擎,E7 讲的错误固化、门控、回归、防刷分,全都转不起来。

把三段连起来看:

- **评测集 = 用什么题考**(架构设计能力:设计一套覆盖真实分布、不断生长的题库);
- **评测标准 = 按什么标准打分**(评估能力:多维量化 + 校准过的裁判 + 多层叠加);
- **决策落地 = 让评测真正驱动上线和迭代**,而不是写完报告就归档。

一句话收束:

> **评测不是上线后补的一份报告,而是产品的方向盘和仪表盘——它告诉你现在好不好(标准)、拿什么衡量(评测集)、下一步往哪改(决策)。一个不会评测的团队,只能凭感觉做 AI 产品,在"A 场景变好、B 场景变差"的循环里打转;一个会评测的团队,能让产品每一次迭代都踏在实地上,每天自动变好。**

### 评估自查清单(拿着就能用)

- [ ] 有自建评测集吗?还是只套了公开榜单 / 凭感觉?三路来源(真实采集 / 故障注入 / 模拟)齐吗?
- [ ] 评测集会过时吗?每周 Look at your data 了吗?新 badcase 进回归集了吗?训练/评测隔离了吗?
- [ ] 打分只看成功率,还是结果 + 过程 + 成本 + 安全 + 满意度多维?权重按业务调了吗?
- [ ] 用 LLM-as-Judge 了吗?校准到 90%+ 一致了吗?有元评估吗?关键 case 程序判优先了吗?
- [ ] 上线走"离线 → 灰度 → A/B → 回归"全链路了吗?A/B 按多步骤特性重设统计口径了吗?
- [ ] 可观测能看到"为什么"(决策原因 / 行为链条 / 结果质量),不只是"发生了什么"吗?
