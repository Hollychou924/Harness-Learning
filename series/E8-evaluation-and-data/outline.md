## 0. 开篇钩子(约 450 字)
- 承接 E7:E7 反复说自进化靠"能评好坏"——建评测集、跑回归、防刷分。但到底怎么科学地"评好坏"?
- **钩子(面试场景)**:面试官问"你怎么评测一个 Agent?"——只会答"看准确率/成功率"的,基本出局。因为 Agent 是多步骤系统,最终答案对,过程可能全错(用了未授权数据源 / 调了十几次无意义工具 / 关键事实来自错误检索 / 只是重试撞对了)。
- **立论**:评好坏,核心就两件事——**用什么题考(评测集)、按什么标准打分(评测标准)**。把这两根支柱搭好,再加"怎么用评测做上线决策",就是一套能驱动产品每天变好的反馈系统。
- 一句话:**评测不是上线后写份报告,是产品的方向盘和仪表盘。**

# 支柱一 · 评测集:用什么题考

## 一、评测集从哪来、怎么建得合理(约 1100 字)= Q1
- **错误做法**:① 没有评测集,凭感觉上线;② 直接套公开榜单(GAIA/AgentBench/SWE-bench)当评测集。
- **行业参考的正确用法**:GAIA / AgentBench / SWE-bench 是"入门锚点"——用来对标行业水位、防 benchmark hacking,但**垂直业务必须自建**,因为公开集覆盖不了你的真实分布。
- **三路并用建评测集(给可抄)**:
  - 真实采集:从线上真实日志里捞 case(最贴近生产分布,是地基)
  - 故障注入:主动构造边界/异常 case(补真实集覆盖不到的边界)
  - 模拟系统:生成长尾 case(补低频但重要的场景)
- **真实案例(带数字)**:阿里云智能运维故障根因 Agent——自建覆盖"输入/代码/资源/状态"四维评测集,真实采集 + 故障注入 + 模拟三路并用;2000+ 原始案例、200+ 已发布;首批 200 案例根因召回 87.5%、定位准确率超 80%。
- **车载差异化(周浩视角)**:车载 Agent 不能套 coding 的 SWE-bench,必须自建覆盖语音识别、方言、多轮中断、分心驾驶等输入维度的 Benchmark——这些是车载特有、公开集完全没有的。
- **每条 case 怎么设计**:带难度等级 + 评分维度 + ground truth("Golden" + 真实采样 + 反例)。
- **一句判断**:"先评测集,再上线"是生产心态,不是洁癖。公开榜单只能告诉你"行业到哪了",**自建评测集才能告诉你"你的用户在你这儿过得好不好"。**

## 二、评测集怎么不过时、不被污染(约 800 字)= Q2
- **会过时**:评测集建完三个月,业务变了、用户变了,它就和现实脱节了。指标还在涨,线上却在崩——因为你在考一套过期的题。
- **Look at your data(Hamel Husain 三年心法)**:定期(每周)拿 20-50 条真实日志亲自看,而不是只盯分数。分数是"薛定谔的好看",真实 case 才告诉你哪里真出问题。
- **持续维护机制(呼应 E7 错误固化)**:每发现一个新的 badcase,就把它加进评测集变成一道回归题——评测集自我增长,模型/Harness 改动永远要在最新评测集上回归通过。
- **防污染(呼应 E7 数据飞轮)**:训练数据绝不能污染评测集,holdout 必须独立。否则"分数提升"可能只是把测试题背下来了,不是能力真变强。
- **一句判断**:评测集是活的,不是建一次就完。**不"看真数据"的评测会过时,不做"训练/评测隔离"的评测会自欺。** 让评测集跟着真实数据一起长。

# 支柱二 · 评测标准:按什么标准打分

## 三、除了成功率,按什么标准打分、怎么量化(约 1200 字)= Q3
- **核心认知**:Agent 是多步骤系统,**只看最终答案会被骗**——答案对,过程可能用了未授权数据、调了十几次废工具、靠重试撞对。所以要同时看结果和过程。
- **5 类指标(给可抄量化表)**:
  | 类别 | 量化指标 | 说明 |
  |---|---|---|
  | 任务完成率 | 成功率、测试通过率、回归失败数 | 它到底干成没 |
  | 过程质量 | 平均步数、工具调用正确率、上下文命中率、错误恢复次数、轨迹合理性 | 怎么干成的(轨迹评估器 Trajectory Evaluator 把输入+输出+中间步骤一起评) |
  | 成本 | token 消耗、缓存命中率、首 token 延迟、单任务耗时 | 划不划算(高性能但资源密集 = 不可部署) |
  | 安全合规 | 危险操作拦截数、越权次数、注入拦截率、偏见/隐私合规 | 安不安全(呼应 E6) |
  | 用户满意度 | 采纳率、合并率、人工接管率、取消/重试率、赞踩 | 用户认不认 |
- **不同业务权重不同**:车载语音 Agent 看"识别准确率+响应延迟+分心驾驶安全";coding Agent 看"测试通过率+采纳率+token 成本"。同一套指标,权重按场景调。
- **过程指标怎么拿**:靠 Trace(执行轨迹)——这就接到第五节可观测。
- **一句判断**:评测标准不是一个"成功率"数字,是一套"结果 + 过程 + 成本 + 安全 + 满意度"的多维仪表盘。**只看成功率,等于只看车速不看油表和方向——迟早出事。**

## 四、谁来打分:人工 vs LLM-as-Judge(约 1100 字)= Q4
- **三种打分员**:① 程序化判断(能确定性判的优先,如测试通过/regex/SQL 结果——最可靠最便宜);② 人工(ground truth,但贵、不可扩量);③ LLM-as-Judge(用 AI 评 AI,可扩量,但有噪声、可能不准)。
- **LLM-as-Judge 什么条件下能信(放我家 91.9% 实战)**:达到 91.9% 评判准确率,但**三件套缺一不可**:结构化 Benchmark + 多评审投票 + 人工抽样校准。**先小批量人工标注,验证一致性达 90%+ 才能上规模**——这是实战参考线。
- **噪声问题**:同一 Skill 同一 GT,LLM 跑 4 次分数 0.79~0.92 漂移。解法:多次采样取均值 / pairwise 对比 / judge prompt 固定版本 / 关键 case 用程序化判断优先。
- **裁判要被元评估**:当裁判变成 LLM,必须有 ground truth 数据集定期验证"这个裁判打分准不准"(芝麻信用"评测 Agent 评测 Agent"机审率 80%+,就靠元评估兜底)。
- **瑞士奶酪模型(Anthropic 官方)**:任何单层评估(单看人工/单看 LLM/单看程序)都有洞,**多层叠加才可靠**——程序判确定性的、LLM 扩量主观的、人工抽样校准,三层错开洞才补上。
- **一句判断**:能程序判的别用 LLM,能 LLM 扩量的别全靠人工,但 LLM 当裁判前必须先用人工把它校准到 90%+ 一致、并持续元评估。**没校准的 LLM-as-Judge,是在用一把没刻度的尺子量东西。**

# 决策落地 · 怎么用评测做上线/迭代决策

## 五、离线过了线上怎么验证:A/B 多步骤挑战 + 可观测(约 1100 字)= Q5
- **评估 Sequence(给可抄)**:离线评测集(上线前拦住明显问题)→ 灰度(小流量试)→ 在线 A/B(真实用户对比)→ 持续回归(每次改动跑回归集)。这是一条从实验室到真实世界的完整链路,缺一环都不稳。
- **A/B 在多步骤 Agent 的特殊挑战(AI PM 面试高频)**:传统 A/B 一次交互一个结果;Agent 一次任务跑 20 步、每步都影响最终结果,**样本量、置信区间、显著性检验的算法都不一样**——不能直接套网页 A/B 那套。要按"任务"而非"单步"做统计单元,且要控制任务复杂度分布。
- **可观测三支柱升级(AWS 秘籍七)**:传统监控的 Metrics → Logs → Traces 三件套,在 Agent 场景只能回答"发生了什么",回答不了"为什么这么干"。Agent 黑盒必须额外加三层:
  - 决策原因:模型为什么选这个动作
  - 行为链条:完整的 think-act-observe 轨迹
  - 结果质量:这一步产出到底好不好
- **离线+在线+Trace+反馈 四件套**:离线冒烟+回归集(上线前)/ 在线 A/B(上线后)/ Trace+Span(还原过程)/ 显式(赞踩重试)+隐式(成功率/放弃率/在哪步离开)反馈(持续优化)。
- **一句判断**:离线分数高 ≠ 线上好用(评测集再全也覆盖不了真实长尾)。**必须走"离线→灰度→A/B→回归"全链路,且 A/B 要按 Agent 的多步骤特性重新设计统计口径,可观测要看得到"为什么",不只是"发生了什么"。**

## 六、拔高 + 自查表 + 下期预告(约 600 字)
- **拔高(呼应 E1 双核 + E7)**:评测集 + 评测标准,正是 E1 那两个核心能力里"**评估能力**"的两根支柱;而评估又是 E7 自进化飞轮的**引擎**——没有"能评好坏",E7 的"错误固化、门控、回归、防刷分"全都转不起来。
  - 评测集 = 用什么题考(架构设计:设计覆盖真实分布的题库)
  - 评测标准 = 按什么标准打分(评估能力:多维量化 + 校准过的裁判)
  - 决策落地 = 让评测真正驱动上线/迭代,而不是写完报告就归档。
- 一句话:**评测不是上线后补的一份报告,是产品的方向盘和仪表盘——它告诉你现在好不好(标准)、用什么衡量(评测集)、下一步往哪改(决策)。一个不会评测的团队,只能凭感觉做 AI 产品;一个会评测的团队,能让产品每天自动变好。**
- **文末:评估自查清单**:

  - [ ] 打分只看成功率,还是结果+过程+成本+安全+满意度多维?权重按业务调了吗?
  - [ ] 用 LLM-as-Judge 了吗?校准到 90%+ 一致了吗?有元评估吗?关键 case 程序判优先了吗?
  - [ ] 上线走"离线→灰度→A/B→回归"全链路了吗?A/B 按多步骤特性重设统计口径了吗?
  - [ ] 可观测能看到"为什么"(决策原因/行为链条/结果质量),不只是"发生了什么"吗?
