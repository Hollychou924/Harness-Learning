## 0. 开篇钩子(约 400 字)

- **场景**: 你的 AI 产品用了市面最强模型,效果却不如对手用差一档模型做的产品。问题不在模型,在 Harness。
- **核心论断**: Agent = Model + Harness,且 Harness 决定真实表现。模型定上限(地板),Harness 定天花板。
- 社区金句钩子: "模型是地板,Harness 才是天花板";"Prompt 是入口,Context 决定可用度,Harness 决定能不能进生产"。

## 1. 什么是 Harness + 发展轨迹(约 1400 字)= Q1

- **一句话定义**: Prompt 下指令、Context 给材料、Harness 控执行。Harness = 让模型在边界里可靠干活的整套运行环境。
- **三层框架**(对外讲清楚用): Prompt / Context / Harness 各管什么(引 `concepts/prompt-context-harness.md` 的三层表)。
- **发展轨迹 5 阶段**(引 `timelines/agent-harness-design-evolution.md`):
  1. 固定 Prompt——解决"怎么说清任务"
  2. 动态 Prompt + Context——解决"怎么让模型看到对的材料"
  3. Harness——解决"怎么在边界里可靠执行"
  4. Harness 七层分类(ETCLOVG)——解决"怎么把执行环境拆成可评审、可验收的模块"
  5. 自进化 Harness——解决"怎么把执行经验变成下一次能力"
- 已否决路径(加分项,体现判断): 只堆长 Prompt / 只做工具调用 / 只做训练反馈,各自不够在哪。

## 2. 为什么需要 Harness + 它有多重要(约 1300 字)= Q2 + ROI

- **核心证据(干货最硬一节)**: 引 `lessons/prompt-only-agent-is-not-production.md`
  - 阿里商旅: AI Coding/任务准确率 24.86% → 90.54%(好 Harness 的成功路径)
  - 腾讯: 4 亿 token 换来的 5 个教训(没 Harness 的踩坑路径)
  - 公式: 代码产出 = AI 能力 × 上下文质量;Demo 易、泛化难
- **重要性三句话**: 模型通用、业务具体,Harness 是把通用模型转成专用系统的那一层;没好架构再强模型也发挥不出;有好架构普通模型也能做出好产品。
- **ROI 怎么算才诚实**: 同时摆成功路径(阿里)和踩坑成本(腾讯 4 亿 token),才是诚实估算。
- 🖊️ 占位: 现成产品 Harness "卡住你"的具体经历(你来填)。

## 4. 怎么判断一个产品的 Harness 做得好(约 1100 字)= Q4

- 把 ETCLOVG 七层变成一张**可打分的检查表**(这是体现专业度的加分节):

  | 维度 | 判断问题 | 做得好的信号 |
  |---|---|---|
  | 执行环境 E | 它在哪里安全动手? | 有沙箱/隔离,不直接碰生产 |
  | 工具接口 T | 调工具有没有边界? | 工具有权限/参数约束、失败有提示 |
  | 上下文 C | 每步看什么、怎么省? | 有压缩/缓存/按需加载,不无脑塞 |
  | 生命周期 L | 长任务能否暂停恢复、人工接管? | 有计划/检查点/接管 |
  | 可观测 O | 出错能不能看懂发生了什么? | 有轨迹/日志/成本/失败原因 |
  | 验证评测 V | 怎么确认结果真可用? | 有测试/评测/回归/验收闸门 |
  | 治理安全 G | 谁能让它做什么? | 有权限/审批/审计/防注入 |

- **一句话标准**: Harness 不是单点功能强,而是这七项组成一套完整的执行秩序;看一个产品好不好,就看它把哪几项做成了系统级硬约束而不是靠提示词软约束。

## 6. Harness 时代的 AIPM 能力模型(约 900 字)= Q6【本章差异化,你定的双核】

- **收敛结论**: Harness 时代 AIPM 与传统 PM 的根本区别,最终收敛到两个最本质能力——**架构设计能力 + 评估能力**。

- **① 架构设计能力(骨架能力)**: 传统 PM 设计"功能",Harness AIPM 设计"能自主完成任务的智能系统"。含:
  - 角色与边界架构、流程编排架构(含人机回环)、安全护栏架构(软约束→系统级硬约束)、记忆与工具架构、多 Agent 协作架构。
  - 为什么是第一核心: Harness 本质就是系统架构;没好架构再强模型也发挥不出,有好架构普通模型也能做出好产品。

- **② 评估能力(眼睛能力)**: Agent 是概率性的,没量化评估就没科学决策。含:
  - 多维评估体系(任务完成率 + 可控率 + 安全合规率 + 成本)、可观测性设计、错误归因(架构?Prompt?工具?模型?)、持续迭代。
  - 为什么是第二核心: Agent 永远不完美,评估是从"能用→好用→可靠"的唯一途径。

- **双核闭环**: 架构设计("做什么")→ 系统实现 → 评估测试 → 错误归因 → 架构优化("做得更好"),首尾相接。

- **为什么不可替代**: 工程师能实现架构但不设计架构(只看技术可行,不看业务目标);数据科学家能分析数据但不定义"业务上的好";模型研究员能优化模型但不负责产品可用/可靠。只有 AIPM 站在业务+用户+技术交叉点同时具备这两项。

- **收尾——与工程师对话的检查清单**(收尾标志产出):
  - [ ] 我们 Agent 的"操作系统"是现成的还是自建的?
  - [ ] 五大组件/七层里,当前最弱是哪一项?
  - [ ] ROI 估算同时算了成功路径和踩坑成本吗?
  - [ ] 我们的架构有没有把关键约束做成硬约束?评估有没有量化指标?
