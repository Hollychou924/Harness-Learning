## 0. 开篇钩子(约 400 字)
- 承接 E4:E4 问过"为什么 Agent 跑久了会变笨——忘约束、乱调用、把错误经验记下来"。E7 反过来问:**怎么让它越用越聪明?**
- 区分两种"变好":一种是等模型升级(被动,等厂商);一种是让你这套 Harness 自己进化(主动,你能控)。本章只讲后者——这才是 PM 能掌控的杠杆。
- **核心命题**:Harness 自进化的本质,不玄,就一句——**把每个错误,工程化地变成"永不再犯"。** 它有个名字叫 Harness Engineering(Hashimoto 的定义)。

## 一、把一个错误固化成"永不再犯"(约 1300 字)= Q1【最具体、最有案例】
- **Hashimoto 的本质定义**:Harness Engineering = 每发现一个错误,就工程化地消除它再次发生的可能。不是"提醒模型别再犯"(软的、会忘),是"用工程手段让它犯不了"(硬的)。
- **错误固化闭环(给可抄)**:
  发现错误 → 把这个错误加进评测集(变成一道回归题)→ 改 Harness(改 Rules/Skills/Wiki/工具)→ 重跑回归集通过 → 才算"修好"。
  关键:**没进评测集的修复,等于没修**——下次还会回来。
- **真实案例**:阿里把单仓 AI 代码率从 24.86% 拉到 90.54%,靠的不是换模型,就是这个闭环跑了一周——AGENTS.md + Spec + 工具化环境 + 测试门禁 + 多 Agent 评审五件套,每发现一类错就补一条规则/一道回归。
- **压缩并入:修之前先定位(原 Q1 退化诊断,压成一小节)**:效果变差时,先分清是"模型退化(换了模型/模型抽风)、Harness 腐化(规则堆太多互相打架)、还是数据漂移(用户/场景变了)"——三者修法不同。但定位方法很朴素:**先有 baseline 和回归集**,效果掉了先跑回归集看是哪类题挂了,再"Look at your data"看真实 case。不展开成大节,点到为止(完整评估方法在 E8)。
- **一句判断**:Harness 会不会进化,就看一件事——**每个错误是不是都变成了一道回归题 + 一条工程改动。** 只在对话里骂模型"你给我记住",它永远学不会;把"记住"写进评测集和规则里,它才真的不再犯。

## 二、Skill 怎么从"能跑"打磨到"自己迭代自己"(约 1300 字)= Q2【从 E5 挪入】
- **场景**:Skill 写完能跑,但效果时好时坏。"能跑的 Skill 和真好的 Skill 之间,有一条到西天取经的路。"怎么系统地打磨,而不是凭感觉改?
- **Skill 自进化:8 阶段 Loop(给可抄)**:Phase0 Setup(建 workspace+评测计划+baseline)→ 每轮循环:Phase1 Review(读 memory+trace)→ Phase2 Ideate(从 trace 诊断失败、提原子化改动)→ Phase3 Modify(只改一处)→ Phase4 Commit(先 commit 再验证,保审计轨迹)→ Phase5 Verify(三层评测)→ Phase6 Gate(5 维 AND 门控)→ Phase7 Log → Phase8 Loop(继续/升层/停止)。
- **三层评测(成本递增,便宜的每轮跑、贵的条件触发)** + **5 维 AND 门控**:为什么是 AND 不是加权求和?——加权求和可能"质量+1 但 token 翻倍"也给 PASS;AND 逻辑里只要一维退步就 discard,不通过直接 git revert,没商量。
- **灵魂金句**:**与其写更长的 prompt 来"说服"模型守规矩,不如把规矩写进代码——门控不过就 git revert,程序掌握控制流,LLM 只管单点生成。** (这正好呼应 E6"软约束变硬约束"和第一节"错误固化"。)
- **真实坑**:LLM 评测有噪声(同一 Skill 同一 GT,Claude 跑 4 次结果 0.79~0.92 漂移)——解法是跑 3 次取均值,但成本翻 3 倍,天下没有免费午餐。
- **一句判断**:Skill 自进化的关键不是"让 AI 自由发挥改",是"给它评测 + 门控 + 回滚"——能评好坏、能卡住退步、能一键回滚。把打磨 Skill 这件事,从"凭感觉调"变成"有门控的迭代实验"。

## 三、数据飞轮:Harness 数据怎么反哺模型(约 1200 字)= Q3
- **先立硬数据**:纯模型能力代差在快速收敛(SWE-Bench 上各家差距已不是用户能直接感知的量级);真正拉开差距的是 Harness——**同一个模型,在不同 Harness 里的实际完成率能差 15-30 个百分点**(Anthropic SWE-Bench 报告)。而 Harness 跑出来的真实 task trace,是训练高质量 Agent 模型最稀缺的语料,比任何开源数据集都贴近生产分布。
- **5 种数据回流模式(给可抄横评表)**:
  | 模式 | 信号粒度 | 训练适用 | 自动化 | 隐私 | 代表 |
  |---|---|---|---|---|---|
  | A Trace 全量回流 | session 级 | SFT+RL 最广 | 全自动 | 高 | Cursor/Manus |
  | B Failure Log opt-in | task 级带标注 | 高质量负样本 | 半自动 | 低 | Claude Code/Codex/Hermes |
  | C Human Takeover 采样 | step 级 paired | DPO/RLHF 金矿 | 全自动无感 | 中 | Cursor/Manus |
  | D Eval Set 自动扩充 | task 级带 GT | 训练目标+回归门 | 全自动 | 低 | Claude Code(SWE-Bench)/Hermes |
  | E Replay-in-Sandbox | step 级可重放 | 离线大规模 A/B | 全自动 | 低 | 无公开实现 |
- **PM 视角的关键判断**:用户接管点(模式 C)是质量最高的负样本——"模型本来要这么干、用户否决了",这种 paired(rejected, chosen)是 DPO/RLHF 训练的金矿。但隐私最敏感,要无感打点 + 用户同意才上传。
- **一句判断**:数据飞轮的左半边是"回流什么数据"(5 模式),右半边是"多久能影响下一代模型"(Eval 基础设施)。产品上线第一天起,就该想清楚:我这套 Harness 在为下一代模型攒什么语料、攒什么评测?

## 四、两条路线:谁该承载"变聪明"(约 1100 字)= Q4
- **核心问题**:让 Agent 变聪明,是把秩序"训进模型",还是"放进 Harness"?业界分两条路:
  - **路线一:轻数据 + 重 Eval(秩序放 Harness)**——Claude Code。opt-in 少量数据 + 内置 Eval 流水线 + 模型与 Harness 同周发布;模型只要一个规范模型,Eval 维度最薄。to B 友好(合规)。
  - **路线二:重数据 + 自研模型(秩序训进模型)**——Cursor。Harness 行为数据全量回流训练自研 Composer 模型,用数据飞轮抵消模型代差;多模型并存(Tab 补全/Composer 长任务/Apply 应用),Eval 分层。to C/SMB 友好。
  - **第三态:不训基础模型,把进化放 Harness**——Manus。跟随 Claude/GPT 节奏,但把"共进化"重心搬到自己的 context engine 和 Skill router。
- **给 DeepSeek 的 0→1 飞轮(报告 5 条建议精选)**:① Day0 上 opt-in Failure Log(合规底线)、Day30 上 Human Takeover 无感采样(拿 paired 金矿);② 模型路由先选"对话级"(像 Codex),别一上来做工具级(Cursor 那是体量大了才玩得起);③ Eval Set 自动扩充 + 公开锚点(SWE-Bench)做回归门。
- **一句判断**:没有标准答案,但有判断逻辑——**合规优先、单模型、to B → 学 Claude(轻数据重 Eval);要用数据飞轮抵消模型代差、能自研模型 → 学 Cursor(重数据);不训基础模型 → 学 Manus(把进化放 Harness)。** 关键是别又想要 Cursor 的数据飞轮、又不想付它的工程和合规成本。

## 五、拔高 + 自查表 + 下期预告(约 600 字)
- **拔高(呼应 E1 双核)**:Harness 自进化,本质是 E1 那两个核心能力合起来转动的飞轮——
  - **评估能力**负责"发现错误、量好坏"(回归集、门控、数据信号);
  - **架构设计能力**负责"把改进固化下来"(改 Rules/Skills/工具、设计数据回流通道)。
  - 两者一转,就是"错误→评测→改 Harness→回归→再也不犯"的飞轮。
- 一句话:**不进化的 Harness 会腐化(规则越堆越乱、错误反复犯);会进化的 Harness,产品上线第一天起就在为下一代攒语料和评测。让 Agent 越用越聪明,靠的不是等模型升级,是 PM 设计的这套自进化飞轮。**
- **文末:Harness 自进化自查清单**:
  - [ ] 每个错误都变成"一道回归题 + 一条工程改动"了吗?还是只在对话里骂模型?
  - [ ] 修复有没有进评测集?(没进=没修,下次还来)
  - [ ] Skill 是凭感觉改,还是有评测+门控+回滚的迭代?
  - [ ] 效果变差时,能分清是模型退化/Harness 腐化/数据漂移吗?有 baseline 和回归集吗?
  - [ ] 这套 Harness 在为下一代模型攒什么数据(5 模式选了哪个)、攒什么评测?
  - [ ] 选对路线了吗?(轻数据重 Eval / 重数据自研 / 进化放 Harness)
- **下期预告**:E8 评估、可观测性与数据驱动迭代——自进化的前提是"能评好坏"。E8 专门讲:怎么科学证明这次迭代真的让 Agent 变好了,而不是自我感觉良好。
