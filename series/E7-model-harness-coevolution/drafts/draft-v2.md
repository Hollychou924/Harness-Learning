## 开篇:越用越聪明,靠的不是等模型升级

E4 里我们问过一个扎心的问题:为什么同一个 Agent,刚开始很聪明,跑久了反而变笨——忘约束、乱调用、还把错误经验记下来?这一篇反过来问:

> **怎么让它越用越聪明?**

先分清两种"变好"。一种是被动的——等模型厂商发新版本,你跟着升级。这条你控制不了,只能等。另一种是主动的——让你这套 Harness 自己进化:每用一阵,它就比上一阵更靠谱。**这第二种,才是 PM 真正能掌控的杠杆。**

而 Harness 自进化的本质,一点都不玄,就一句话:

> **把每个错误,工程化地变成"永不再犯"。**

这件事有个名字,叫 Harness Engineering。它最朴素的定义来自 Mitchell Hashimoto——他原话是:**你花时间去工程化地设计一个解法,让 agent 永远不再犯那个错误**("engineer a solution such that the agent never makes that mistake again")。注意是"工程化地设计解法",不是"提醒模型别再犯"——后者是软的、会忘的;前者是硬的、犯不了的。下面五节,把这件事拆成一套可落地、又不会把自己改坏的飞轮。

## 一、把一个错误固化成"永不再犯"(Q1)

这是 Harness 自进化最具体的一步,也是最有真实案例撑的一步。

很多人"修 bug"的方式是:跟 Agent 说"你给我记住,以后别这么干了"。这是无效的——它下次大概率还会犯,因为你只是在对话里说了句软话,没有任何东西真正改变。

**有效的做法是一条闭环**:

```text
发现错误
 → 把这个错误加进评测集(变成一道回归题)
 → 改 Harness(改 Rules / Skills / Wiki / 工具)
 → 重跑回归集,通过
 → 才算"修好"
```

这条闭环里最关键的一句话是:**没进评测集的修复,等于没修。** 因为你下次改别的东西时,很可能把这个 bug 又带回来,而没有回归题盯着,你根本不会发现。

**真实案例**:阿里一个团队把单仓库的 AI 代码率从 24.86% 拉到 90.54%。靠的不是换了更强的模型,就是这条闭环跑了一周——用 AGENTS.md + Spec + 工具化环境 + 测试门禁 + 多 Agent 评审这"五件套",每发现一类错误,就补一条规则、加一道回归题。错误一个个被工程化地消除,完成率就一个台阶一个台阶上去了。

### 1.1 不是所有错误都用同一种方式修:错误分类 → 固化到哪一层

固化"永不再犯"还有一层讲究:**不同类型的错误,该固化到不同的系统层。** 修错了层,要么没修住、要么把别的搞坏:

| 错误类型 | 典型表现 | 应该固化到哪一层 |
|---|---|---|
| 偏好/风格类 | 命名风格、注释习惯不对 | Memory / 规则文件(CLAUDE.md) |
| 规则约束类 | 违反团队规范、踩了禁区 | Rule(AGENTS.md / 规则) |
| 流程类 | 漏步骤、顺序错 | Skill(把验证过的流程固化) |
| 工具使用类 | 调错工具、参数错 | Tool 定义 / Tool Registry(改 description、加 schema) |
| 安全/权限类 | 越权、碰了高危操作 | Policy / 硬约束(绝不能交给模型自觉) |
| 能力盲区类 | 模型确实不会 | 评测集 + 考虑训练(Eval / 模型层) |

**一句判断**(配合改动分层):一个偏好问题别去改 Policy,一个安全问题更别只写进 CLAUDE.md 指望模型自觉。**错误分类决定改动分层——修到对的层,才叫真修好。**

**"偏好类固化到 Memory"这条,正在被产品自动化。** 过去是人手写进 CLAUDE.md,现在头部产品开始让 Agent **自己**把偏好沉淀成长期记忆——OpenAI 2026-06 给 ChatGPT 上线的 Dreaming 系统就是典型:它在后台从大量对话里自动提炼用户偏好和约束,合成进长期记忆,还会随时间作废过期事实。效果有数据撑:偏好遵循从便签时代的 31.4% 升到 71.3%。

但这恰恰印证了本章的治理主线——**自动进化必须受治理**。Dreaming 没有让记忆"黑箱自更新",而是配了一个用户可见、可改、可"以后不要再提"、还能**回滚到旧版本**的记忆摘要页。换成 E7 的话说:Agent 可以自己提议把偏好写进 Memory,但这条"自写记忆"必须**可见、可纠错、可回滚**,而不是默认就信(这点和 E4 Q6 完全一致)。

> **可迁移判断**:偏好固化进 Memory 这一层,正在从"人手写规则"走向"Agent 自动合成";但自动合成越强,越要把"用户可见 + 可纠错 + 可回滚"这套治理界面配齐——否则记错了,用户连改的入口都没有,自进化就变成了自污染。


### 1.2 修之前先定位:是模型退化、Harness 腐化,还是数据漂移

效果变差时,先分清三种(完整评估方法在 E8,这里点到为止):

- **模型退化**:换了模型、或模型本身抽风。
- **Harness 腐化**:你自己堆的规则太多、互相打架(一条 MUST 和一条 NEVER 矛盾),把 Agent 搞糊涂了。
- **数据漂移**:用户变了、场景变了,老规则不适配了。

定位方法很朴素:**先有 baseline 和回归集**,效果一掉,先跑回归集看哪类题挂了,再"Look at your data"翻真实 case。没有 baseline 和回归集,你连"退化了没、退化在哪"都说不清。

> **一句判断**:Harness 会不会进化,就看一件事——**每个错误是不是都变成了"一道回归题 + 一条(固化到对的层的)工程改动"。** 只在对话里骂模型"你给我记住",它永远学不会;把"记住"写进评测集和对应的系统层,它才真的不再犯。

## 二、Skill 怎么从"能跑"打磨到"自己迭代自己"(Q2)

上一节讲"错误怎么固化",这一节讲一个更进阶的:**一个能力(Skill)怎么从"能跑"打磨到"真好用",甚至让它自己迭代自己。**

痛点很普遍:Skill 写完能跑,但效果时好时坏。有句话说得好——"能跑的 Skill 和真正好用的 Skill 之间,有一条到西天取经的路。"问题是,怎么系统地走这条路,而不是凭感觉乱改?

(顺带一提:Skill 本来就不该被当成"一段提示词"。Anthropic 官方把 Skill 定义成文件夹形式的能力包,文档里专门讲 planning、testing、iteration、distribution——也就是说,Skill 是可以被测试、迭代、分发和维护的能力资产。)

社区有人做了一个"让 Skill 自己训练自己"的方案,把深度学习里"训练-评测-门控"的思路搬到了 Skill 上,很有借鉴价值。它的骨架是一个 **8 阶段 Loop**:

```text
Phase0 Setup:建 workspace + 生成评测计划 + 建立 baseline(一次性)
---- 以下每轮循环 ----
Phase1 Review:读 memory + 执行 trace
Phase2 Ideate:从 trace 诊断失败原因,提出原子化改动
Phase3 Modify:只执行一个改动
Phase4 Commit:先 commit 再验证(保留审计轨迹)
Phase5 Verify:三层评测流水线
Phase6 Gate:5 维 AND 门控
Phase7 Log:写结果
Phase8 Loop:继续 / 升层 / 停止
```

两个设计点特别值得 PM 记住:

**① 三层评测,成本递增**:便宜的评测(程序能判的,如 contains/regex)每轮都跑,贵的评测(要 LLM 语义判断的)条件触发。既保证每轮有反馈,又不让评测成本失控。

**② 5 维 AND 门控,不过就回滚**:为什么是 AND(全部维度都不退步)而不是加权求和?因为加权求和可能给你一个"质量 +1 但 token 翻倍"或"质量升了但安全退步"的改动判 PASS;AND 逻辑里只要有一维退步,这个改动就被 discard,**直接 `git revert`,没有商量余地。**

这就引出这一节的核心金句:

> **与其写更长的 prompt 去"说服"模型守规矩,不如把规矩写进代码——门控不过就 git revert。程序掌握控制流,LLM 只管单点生成。**

(这正好呼应 E6 的"软约束变硬约束",和上一节"错误固化"是同一种思维:别靠模型自觉,靠工程兜底。)

**一个真实的坑**:LLM 评测本身有噪声。同一个 Skill 状态、同一份测试数据,Claude 跑 4 次,分数能在 0.79~0.92 之间漂移。你改了一行规则,分数从 0.85 变 0.87——这到底是你的功劳,还是 LLM 今天心情好?分不清。解法是跑 3 次取均值,但评测成本也翻 3 倍。天下没有免费的午餐。

> **一句判断**:Skill 自进化的关键不是"让 AI 自由发挥地改",是"给它评测 + 门控 + 回滚"——能评好坏、能卡住退步、能一键回滚。把打磨 Skill 这件事,从"凭感觉调参"变成"有门控的迭代实验"。

## 三、数据飞轮:Harness 数据怎么反哺模型(Q3)

前两节讲的是"在 Harness 内部进化"。这一节往外看一层:**Harness 跑出来的数据,怎么反哺模型训练?**

先立一个判断,但口径要稳。模型基准分(SWE-Bench 这类)当然仍然重要;但在真实产品体验里,**模型之间的差距,会被 Harness 的上下文、工具、权限、评测、回滚能力进一步放大或缩小。** Anthropic 自己讲 SWE-Bench 成绩时也强调:他们是在模型外面"搭了一套 agent / scaffolding"——同一个模型,外面的 harness 搭得好不好,最终完成率差别很大(具体能差多少,以各家内部报告为准,公开数字会随 leaderboard 快速变化)。

更关键的是:Harness 跑出来的真实 task trace(用户 prompt → 模型决策 → 工具调用 → 文件 diff → 测试结果),是训练高质量 Agent 模型**最稀缺的语料**——它比任何开源数据集都贴近生产分布。这就是为什么"数据飞轮"是新一代 Agent 的护城河:**产品上线第一天起,就在为下一代模型攒别人拿不到的语料。**

业界目前能观察到 5 种数据回流模式(这是公开可观察的产品倾向,不代表各家确切的内部数据策略):

| 模式 | 信号粒度 | 训练适用 | 自动化倾向 | 隐私/合规成本 | 公开倾向接近 |
|---|---|---|---|---|---|
| **A Trace 回流** | session 级 | SFT+RL 最广 | 高 | 高 | 重数据路线(Cursor/Manus 方向) |
| **B Failure Log opt-in** | task 级,带标注 | 高质量负样本 | 半自动(用户触发) | 低 | Claude Code/Codex/Hermes |
| **C Human Takeover 采样** | step 级,paired | DPO/RLHF 金矿 | 高(无感打点) | 中 | Cursor/Manus 方向 |
| **D Eval Set 自动扩充** | task 级,带 ground truth | 训练目标 + 回归门 | 高 | 低 | Claude Code(SWE-Bench)/Hermes |
| **E Replay-in-Sandbox** | step 级,可重放 | 离线大规模 A/B | 高 | 低 | 暂无公开完整实现 |

其中 PM 最值得记住的是**模式 C(用户接管点采样)**:当用户撤销 commit、终止 agent run、强行切回手动编辑的那一刻,Harness 打点——这是质量最高的负样本,因为它精确记录了"模型本来要这么干、但用户否决了"。这种 paired(rejected, chosen)数据,正是偏好学习(DPO/RLHF)最稀缺的格式,是金矿。代价是隐私最敏感,必须无感打点 + 用户同意才上传。

**两条红线(数据飞轮必须先过)**:
- **数据合同先行**:任何回流都要先过隐私、脱敏、授权——外部内容/敏感代码默认不进训练集,用户不同意不上传。
- **训练集与评测集隔离**:绝不能让训练数据污染评测集(holdout 必须独立)。否则"分数提升"可能只是模型把测试题背下来了,不是能力真变强——自进化就退化成了"刷分"。

> **一句判断**:数据飞轮的左半边是"回流什么数据"(5 种模式各有取舍),右半边是"多久能影响下一代模型"(Eval 基础设施)。但开闸之前先签两条红线:数据合同(隐私/授权)和训练评测隔离(防刷分)。产品上线第一天就该想清楚:我这套 Harness 在合规前提下,为下一代攒什么语料、攒什么评测?

## 四、两条路线:谁该承载"变聪明"(Q4)

让 Agent 变聪明,到底是把秩序"训进模型",还是"放进 Harness"?业界分成两条主路线,外加一个第三态:

- **路线一 · 轻数据 + 重 Eval(秩序放 Harness)——Claude Code**:数据回流走 opt-in(少而精)、配一套内置 Eval 流水线,模型与 Harness 同周发布。它只要一个规范模型,Eval 维度最薄,把"秩序"做在 Harness 里而不是训进模型。合规友好,to B 路线。
- **路线二 · 重数据 + 自研模型(秩序训进模型)——Cursor 方向**:更依赖真实交互数据飞轮,训练自研模型(如 Composer),用数据飞轮抵消模型代差;多模型并存(补全/长任务/diff 应用各一个),Eval 必须分层。to C/SMB 路线。
- **第三态 · 不训基础模型,把进化放 Harness——Manus 方向**:跟随主流模型节奏,但把"共进化"重心搬到自己的 context engine 和 Skill router 上。

**到底什么训进模型、什么留在 Harness?一张判断表**:

| 情况 | 放哪 |
|---|---|
| 高频、需泛化、语言/推理习惯 | 训进模型 |
| 低频、企业定制、权限敏感 | 放 Harness |
| 明确流程、固定工具调用 | Skill / Tool |
| 高风险安全边界 | Policy(绝不交给模型) |
| 要快速迭代 | Harness(改了就生效,fine-tune 周期长、回滚慢) |
| 长期通用能力 | Model |

**给 DeepSeek 的 0→1 飞轮建议(精选 3 条)**:① Day0 上 opt-in Failure Log(合规底线)、Day30 上 Human Takeover 无感采样(拿 paired 金矿);② 模型路由先选"对话级"(像 Codex),别一上来做工具级(那是体量大了才玩得起);③ Eval Set 自动扩充 + 公开锚点(SWE-Bench)做回归门。

> **一句判断**:没有标准答案,但有判断逻辑——合规优先、单模型、to B → 学 Claude(轻数据重 Eval);要用数据飞轮抵消代差且能自研模型 → 学 Cursor 方向(重数据);不训基础模型 → 学 Manus 方向(进化放 Harness)。**最忌讳的是:又想要重数据飞轮,又不想付它的工程和合规成本。**

## 五、自进化系统,怎么不把自己改坏(Q5)

"自进化"听起来很美,但有个危险的误读:让 Agent 自己随便改自己的 Harness。**这是不行的——自进化不是让 Agent 自由发挥,是一套有版本、有门控、有治理权的工程流程。** 否则它很容易把自己越改越坏(规则越堆越乱、为刷 eval 过拟合、甚至改坏安全规则)。

**生产发布链路(每个自进化改动都要走)**:

```text
候选改动 → 全部过 Eval/回归 → 高风险改动人工审核
→ 灰度发布 → 线上监控 → 一键回滚兜底
```

每个改动必须**版本化**(可追溯、可回滚);所有候选改动必须**过评测**;高风险改动必须**人工审核**;上线先**灰度**、配**线上监控**和**一键回滚**。

**治理权(一条红线)**:**Agent 可以"建议"自进化(提出候选改动),但不能"擅自修改"生产 Harness——尤其是安全规则,普通 Agent 绝不允许自改。** 改动权握在人/审核流程手里。这和 E6"权限先于能力"是同一条原则:能提议,不等于能执行。

**防过拟合(别把自进化变成刷分)**:如果自进化的目标函数就是"评测分更高",它很可能学会"针对评测集刷分"而不是"真变强"。防御靠第三节那条——训练/评测数据隔离 + 独立 holdout + 定期换新真实 case 进评测集。

**怎么判断 Harness 已经在腐化(给信号)**:规则冲突数增加、prompt 越来越长、Skill 误触发升高、token 成本升高、回归集通过但线上失败、用户接管率上升、同类错误反复出现——出现这些,就是该清债、该治理了。

> **一句判断**:自进化最大的风险不是"进化得不够快",是"自己把自己改坏"。所以它必须是"Agent 提议、人/流程审核、过门控、灰度、可回滚"的受治理过程——**能自我改进,但改进权受治理;能越用越聪明,但绝不能越用越脏。**

## 六、拔高:自进化,是双核能力转起来的飞轮

回到 E1 的"双核能力"。Harness 自进化,本质上就是那两个核心能力**合起来转动的一个飞轮**:

```text
真实任务运行
 → Trace / Failure / 用户接管点
 → Case 标注
 → 加入 Eval / 回归集
 → 生成 Harness 改动(按错误类型固化到对的层)
 → 过门控 / 回归 / 成本检查
 → 灰度发布
 → 线上观测
 → 新数据回流(回到起点)
```

这个飞轮转一圈,产出两条线:

- **短期输出**:Rules / Skills / Tools / Policy 变好——同类错误不再犯。
- **长期输出**:训练数据 / 评测集 / 模型路线变好——产品运行本身成了下一代的训练场。

而驱动这个飞轮的,正是双核:**评估能力**负责"发现错误、量好坏、防刷分"(回归集、门控、数据隔离);**架构设计能力**负责"把改进固化到对的层、设计数据回流通道和治理流程"。

一句话收束:

> **Harness 自进化,不是让 Agent 自己变魔法,而是把真实世界里的每一次失败,变成可复现的评测、可验证的改动、可回滚的发布、可复用的数据资产。短期看,它让同类错误不再犯;长期看,它把产品运行本身变成下一代模型和下一代 Harness 的训练场。** 让 Agent 越用越聪明,靠的不是干等模型升级,而是 PM 亲手设计、并加以治理的这套自进化飞轮——这也是"与研究员协作"的真正含义。

### Harness 自进化自查清单(拿着就能用)

- [ ] 每个错误都变成"一道回归题 + 一条工程改动"了吗?固化到对的层了吗(偏好→Memory、安全→Policy)?
- [ ] 修复有没有进评测集?(没进 = 没修,下次还来)
- [ ] Skill 是凭感觉改,还是有评测 + 门控 + 回滚?门控不过会自动 revert 吗?
- [ ] 效果变差时,能分清模型退化 / Harness 腐化 / 数据漂移吗?有 baseline 和回归集吗?
- [ ] 数据飞轮过了两条红线吗:数据合同(隐私/授权)、训练评测隔离(防刷分)?
- [ ] 选对路线了吗(轻数据重 Eval / 重数据自研 / 进化放 Harness)?别又要飞轮又不想付成本。
- [ ] 自进化受治理吗:版本化、过门控、高风险人工审核、灰度、可回滚?Agent 能建议但不能擅改生产/安全规则?
