## 开篇:越用越聪明,靠的不是等模型升级

E4 里我们问过一个扎心的问题:为什么同一个 Agent,刚开始很聪明,跑久了反而变笨——忘约束、乱调用、还把错误经验记下来?这一篇反过来问:

> **怎么让它越用越聪明?**

先分清两种"变好"。一种是被动的——等模型厂商发新版本,你跟着升级。这条你控制不了,只能等。另一种是主动的——让你这套 Harness 自己进化:每用一阵,它就比上一阵更靠谱。**这第二种,才是 PM 真正能掌控的杠杆。**

而 Harness 自进化的本质,一点都不玄,就一句话:

> **把每个错误,工程化地变成"永不再犯"。**

这件事有个名字,叫 Harness Engineering。它最朴素的定义来自 Mitchell Hashimoto:**每发现一个错误,就用工程手段消除它再次发生的可能。** 注意是"工程手段消除",不是"提醒模型别再犯"——后者是软的、会忘的;前者是硬的、犯不了的。下面四节,就是把这件事拆成可落地的飞轮。

## 一、把一个错误固化成"永不再犯"(Q1)

这是 Harness 自进化最具体的一步,也是最有真实案例撑的一步。

很多人"修 bug"的方式是:跟 Agent 说"你给我记住,以后别这么干了"。这是无效的——它下次大概率还会犯,因为你只是在对话里说了句软话,没有任何东西真正改变。

**有效的做法是一条闭环**:

```text
发现错误
 → 把这个错误加进评测集(变成一道回归题)
 → 改 Harness(改 Rules / Skills / Wiki / 工具)
 → 重跑回归集,通过
 → 才算"修好"
```

这条闭环里最关键的一句话是:**没进评测集的修复,等于没修。** 因为你下次改别的东西时,很可能把这个 bug 又带回来,而没有回归题盯着,你根本不会发现。

**真实案例**:阿里一个团队把单仓库的 AI 代码率从 24.86% 拉到 90.54%。靠的不是换了更强的模型,就是这条闭环跑了一周——用 AGENTS.md + Spec + 工具化环境 + 测试门禁 + 多 Agent 评审这"五件套",每发现一类错误,就补一条规则、加一道回归题。错误一个个被工程化地消除,完成率就一个台阶一个台阶上去了。

**修之前,先定位**(这里压缩讲一下,完整评估方法在 E8):效果变差时,先分清是三种里的哪一种——

- **模型退化**:换了模型、或模型本身抽风。
- **Harness 腐化**:你自己堆的规则太多、互相打架(一条 MUST 和一条 NEVER 矛盾),把 Agent 搞糊涂了。
- **数据漂移**:用户变了、场景变了,老规则不适配了。

三者修法完全不同,但定位方法很朴素:**先有 baseline 和回归集**,效果一掉,先跑回归集看是哪类题挂了,再"Look at your data"翻真实 case。没有 baseline 和回归集,你连"到底退化了没、退化在哪"都说不清。

> **一句判断**:Harness 会不会进化,就看一件事——**每个错误是不是都变成了"一道回归题 + 一条工程改动"。** 只在对话里骂模型"你给我记住",它永远学不会;把"记住"写进评测集和规则里,它才真的不再犯。

## 二、Skill 怎么从"能跑"打磨到"自己迭代自己"(Q2)

上一节讲"错误怎么固化",这一节讲一个更进阶的:**一个能力(Skill)怎么从"能跑"打磨到"真好用",甚至让它自己迭代自己。**

痛点很普遍:Skill 写完能跑,但效果时好时坏。有句话说得好——"能跑的 Skill 和真正好用的 Skill 之间,有一条到西天取经的路。"问题是,怎么系统地走这条路,而不是凭感觉乱改?

社区有人做了一个"让 Skill 自己训练自己"的方案,把深度学习里"训练-评测-门控"的思路搬到了 Skill 上,很有借鉴价值。它的骨架是一个 **8 阶段 Loop**:

```text
Phase0 Setup:建 workspace + 生成评测计划 + 建立 baseline(一次性)
---- 以下每轮循环 ----
Phase1 Review:读 memory + 执行 trace
Phase2 Ideate:从 trace 诊断失败原因,提出原子化改动
Phase3 Modify:只执行一个改动
Phase4 Commit:先 commit 再验证(保留审计轨迹)
Phase5 Verify:三层评测流水线
Phase6 Gate:5 维 AND 门控
Phase7 Log:写结果
Phase8 Loop:继续 / 升层 / 停止
```

两个设计点特别值得 PM 记住:

**① 三层评测,成本递增**:便宜的评测(程序能判的,如 contains/regex)每轮都跑,贵的评测(要 LLM 语义判断的)条件触发。这样既保证每轮有反馈,又不让评测成本失控。

**② 5 维 AND 门控,不过就回滚**:为什么是 AND(全部维度都不退步)而不是加权求和?因为加权求和可能给你一个"质量 +1 但 token 翻倍"的改动判 PASS;AND 逻辑里只要有一维退步,这个改动就被 discard,**直接 `git revert`,没有商量余地。**

这就引出这一节的灵魂金句:

> **与其写更长的 prompt 去"说服"模型守规矩,不如把规矩写进代码——门控不过就 git revert。程序掌握控制流,LLM 只管单点生成。**

(这正好呼应 E6 的"软约束变硬约束",和上一节"错误固化"是同一种思维:别靠模型自觉,靠工程兜底。)

**一个真实的坑**:LLM 评测本身有噪声。同一个 Skill 状态、同一份测试数据,Claude 跑 4 次,分数能在 0.79~0.92 之间漂移。你改了一行规则,分数从 0.85 变 0.87——这到底是你的功劳,还是 LLM 今天心情好?分不清。解法是跑 3 次取均值,但评测成本也翻 3 倍。天下没有免费的午餐。

> **一句判断**:Skill 自进化的关键不是"让 AI 自由发挥地改",是"给它评测 + 门控 + 回滚"——能评好坏、能卡住退步、能一键回滚。把打磨 Skill 这件事,从"凭感觉调参"变成"有门控的迭代实验"。

## 三、数据飞轮:Harness 数据怎么反哺模型(Q3)

前两节讲的是"在 Harness 内部进化"。这一节往外看一层:**Harness 跑出来的数据,怎么反哺模型训练?**

先立一个硬判断。纯模型能力的代差,正在快速收敛——SWE-Bench 上几家头部模型的差距,已经不是用户能直接感知的量级了。真正拉开差距的是 Harness:

> **同一个模型,放在不同 Harness 里,实际任务完成率能差 15-30 个百分点。**(Anthropic SWE-Bench 报告)

而且,Harness 跑出来的真实 task trace(用户 prompt → 模型决策 → 工具调用 → 文件 diff → 测试结果),是训练高质量 Agent 模型**最稀缺的语料**——它比任何开源数据集都贴近生产分布。这就是为什么"数据飞轮"是新一代 Agent 的护城河:**产品上线第一天起,就在为下一代模型攒别人拿不到的语料。**

业界目前能观察到 5 种数据回流模式:

| 模式 | 信号粒度 | 训练适用 | 自动化 | 隐私 | 代表产品 |
|---|---|---|---|---|---|
| **A Trace 全量回流** | session 级 | SFT+RL 最广 | 全自动 | 高 | Cursor、Manus |
| **B Failure Log opt-in** | task 级,带用户标注 | 高质量负样本 | 半自动(用户触发) | 低 | Claude Code、Codex、Hermes |
| **C Human Takeover 采样** | step 级,paired | DPO/RLHF 金矿 | 全自动(无感打点) | 中 | Cursor、Manus |
| **D Eval Set 自动扩充** | task 级,带 ground truth | 训练目标 + 回归门 | 全自动 | 低 | Claude Code(SWE-Bench)、Hermes |
| **E Replay-in-Sandbox** | step 级,可重放 | 离线大规模 A/B | 全自动 | 低 | 无公开实现 |

其中 PM 最该记住的是**模式 C(用户接管点采样)**:当用户撤销 commit、终止 agent run、强行切回手动编辑的那一刻,Harness 自动打点——这是质量最高的负样本,因为它精确记录了"模型本来要这么干、但用户否决了"。这种 paired(rejected, chosen)数据,正是 DPO/RLHF 训练最稀缺的格式,是金矿。代价是隐私最敏感,必须无感打点 + 用户同意才上传。

> **一句判断**:数据飞轮的左半边是"回流什么数据"(5 种模式各有取舍),右半边是"多久能影响下一代模型"(Eval 基础设施)。产品上线第一天就该想清楚:我这套 Harness 在为下一代模型攒什么语料、攒什么评测?攒不到,就只能永远等厂商升级。

## 四、两条路线:谁该承载"变聪明"(Q4)

让 Agent 变聪明,到底是把秩序"训进模型",还是"放进 Harness"?业界分成两条主路线,外加一个第三态:

- **路线一 · 轻数据 + 重 Eval(秩序放 Harness)——Claude Code**:数据回流走 opt-in(少而精)、配一套内置 Eval 流水线,模型与 Harness 同周发布。它只要一个规范模型,Eval 维度最薄,把"秩序"做在 Harness 里而不是训进模型。合规友好,to B 路线。
- **路线二 · 重数据 + 自研模型(秩序训进模型)——Cursor**:Harness 行为数据全量回流,训练自研的 Composer 模型,用数据飞轮抵消模型代差。多模型并存(Tab 管补全、Composer 管长任务、Apply 管 diff 应用),Eval 必须分层。to C/SMB 路线。
- **第三态 · 不训基础模型,把进化放 Harness——Manus**:跟随 Claude/GPT 的节奏,但把"共进化"重心搬到自己的 context engine 和 Skill router 上。

**给 DeepSeek 的 0→1 飞轮建议(报告里 5 条精选 3 条)**:
1. **Day0 上 opt-in Failure Log(合规底线),Day30 上 Human Takeover 无感采样**——光靠 opt-in 数据量太小、long-tail 覆盖不全;Takeover 拿到的 paired 数据才是 DPO/RLHF 的金矿。
2. **模型路由先选"对话级"(像 Codex),别一上来就做工具级(Cursor 那套)**——工具级路由是体量起来后的产物,工程成本极高;0→1 阶段对话级切换 A/B 信号清晰、Eval 维度可控。
3. **Eval Set 自动扩充 + 公开锚点(SWE-Bench)做回归门**——让评测集自我增长,新模型不会在老用户的真实场景下回归。

> **一句判断**:没有标准答案,但有判断逻辑——**合规优先、单模型、to B → 学 Claude(轻数据重 Eval);要用数据飞轮抵消模型代差、且能自研模型 → 学 Cursor(重数据);不训基础模型 → 学 Manus(把进化放 Harness)。** 最忌讳的是:又想要 Cursor 的数据飞轮,又不想付它的工程和合规成本。

## 五、拔高:自进化,是双核能力转起来的飞轮

回到 E1 的"双核能力"。Harness 自进化,本质上就是那两个核心能力**合起来转动的一个飞轮**:

- **评估能力**负责"发现错误、量好坏"——建回归集、设门控、采数据信号;
- **架构设计能力**负责"把改进固化下来"——改 Rules/Skills/工具、设计数据回流通道。

两者一转,就是贯穿全章的那条飞轮:**错误 → 进评测集 → 改 Harness → 跑回归 → 再也不犯 → 攒成下一代的语料和评测。**

一句话收束:

> **不进化的 Harness 会腐化——规则越堆越乱、错误反复犯、Skill 越积越脏;会进化的 Harness,产品上线第一天起就在为下一代攒语料和评测。让 Agent 越用越聪明,靠的不是干等模型升级,而是 PM 亲手设计的这套自进化飞轮。** 这也是"与研究员协作"的真正含义:PM 不是去帮研究员调模型,而是设计好这套飞轮,让产品自己源源不断地产出训练语料和评测信号。

### Harness 自进化自查清单(拿着就能用)

- [ ] 每个错误都变成"一道回归题 + 一条工程改动"了吗?还是只在对话里骂模型?
- [ ] 修复有没有进评测集?(没进 = 没修,下次还来)
- [ ] Skill 是凭感觉改,还是有评测 + 门控 + 回滚的迭代?门控不过会自动 revert 吗?
- [ ] 效果变差时,能分清是模型退化 / Harness 腐化 / 数据漂移吗?有 baseline 和回归集吗?
- [ ] 这套 Harness 在为下一代模型攒什么数据(5 模式选了哪个)、攒什么评测?
- [ ] 选对路线了吗?(轻数据重 Eval / 重数据自研 / 进化放 Harness)别又要飞轮又不想付成本。
