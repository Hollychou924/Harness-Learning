## 开篇:先把四个概念分清,再讲为什么 Agent 会"越跑越笨"

工具调得对,只说明 Agent 有手有脚;上下文和记忆管得好,才说明它有脑子。E3 讲清了"怎么让 Agent 调对工具",这一篇要回答一个更靠近"智力"的问题:**一次对话,Agent 到底带着什么在思考?**

先说一个几乎每个用过 Agent 的人都遇到过的现象:

> 同一个 Agent,刚开始很聪明,跑了几十轮之后,开始忘记你早就交代过的约束、乱调工具、回复变慢、token 越烧越多,甚至把这一轮踩的坑当成"经验"记下来,下次接着错。

很多人以为这是"模型忘性大"。不是。**这是信息架构没设计好。** 而要把信息架构讲清楚,必须先把四个最容易被混为一谈的概念拆开:

| 概念 | 本质 | 解决什么问题 | 典型错误 |
|---|---|---|---|
| **Context 上下文** | 本轮模型能看到的信息 | 让模型这一轮答对 | 什么都塞进去 |
| **Memory 记忆** | 跨会话保留的信息 | 让 Agent 下次还能接着做 | 什么都永久记住 |
| **Cache 缓存** | 复用已计算过的前缀 KV | 降低成本和延迟 | 动态内容放前面导致 miss |
| **Compact 压缩** | 把长上下文变短 | 防止窗口爆掉 | 压掉关键约束和证据 |

记住这一句,本章就不会乱:

> **上下文决定 Agent 这一轮看见什么;记忆决定下次还记得什么;缓存决定重复内容要不要重新算;压缩决定窗口满了以后怎么活下来。**

这个拆分不是我编的。Anthropic 官方在 Claude Code 的文档里就明确把两套机制分开:每个会话都从一个**空白的上下文窗口**开始,靠两套东西跨会话带知识——`CLAUDE.md`(你手写的持久指令)和**自动记忆**(Claude 根据你的更正和偏好自己记下的笔记)。前者是上下文里的人工规则,后者是记忆。两层,不是一层。

下面分两幕。上半场讲"这一轮的上下文窗口怎么装满又不装乱",下半场讲"跨会话的长期记忆怎么管才不脏"。

# 上半场 · 上下文窗口怎么装

## 一、一次对话带着什么:Context Stack(Q1)

先看镇文图的请求体部分(§2 + §6A)。发给模型的一次完整请求,其实由三大块拼成:**system(系统提示词)+ tools(工具定义)+ messages(消息历史)**。但这三块里面还有更细的分层。我建议你把它理解成一个 **Context Stack(上下文栈)**——每一层有不同的内容、不同的稳定性、不同的风险:

| 层级 | 内容 | 稳定性 | 主要风险 |
|---|---|---|---|
| **System Prompt** | 角色、边界、基础规则(Intro / System / Tasks / Actions) | 高(静态区) | 写太长,挤占任务预算 |
| **Tools 工具定义** | 每个工具的 input_schema 等,占 token、影响预算 | 中 | 工具太多→选不对(见 E3) |
| **Rules / 三源注入** | CLAUDE.md(基础规范)+ Skills/Plugins(能力)+ MEMORY.md(长期记忆) | 中 | 全量静态注入,不按任务剔除 |
| **Messages 历史** | user + assistant 的关键消息,过滤掉低价值/冗余 | 低(动态区) | 单调增长,迟早撑爆 |
| **Repo / 检索片段** | 按需读进来的代码、文档 | 低 | 读了很多,但没读对 |
| **Tool Results** | 工具返回(可能超长) | 低 | 撑爆上下文(见 E3 的分层返回) |

这张栈里有一条贯穿全章的分界线:**静态区(Static Sections)vs 动态区(Dynamic Sections),中间隔着一条 Boundary(边界)。** 镇文图 §6A 画得很清楚:静态区放 Intro / System / Tasks / Actions 这些"整个会话基本不变"的东西(大约 7 个 section);动态区放 Session / Memory / Env 这些"每轮都在变"的东西(8 个以上)。

为什么要分这条线?现在先记住"静态区更稳、更适合缓存命中"这一句——第五节讲 cache 时,你会看到这条 Boundary 切在哪,直接决定了产品的成本生死。

> **一句判断**:Agent 的"上下文"不是一坨文本,是一个分层的栈。每一层的稳定性不同、风险不同,管理方式也就不同。把它当成一坨,就只能"无脑塞";把它当成栈,才能分层优化。

### 1.1 三源注入里最常踩的坑:AGENTS.md / CLAUDE.md 怎么写才真起作用

上面那张栈里有一层叫"三源注入"——CLAUDE.md(基础规范)、Skills/Plugins(能力)、MEMORY.md(长期记忆)。其中 `AGENTS.md` / `CLAUDE.md` 是大多数团队写的第一份"规则文件",也是最多人踩坑的地方:**明明写了一大堆规则,Agent 就是不遵守。**

根因通常是两个:

**① 把规则文件写成了"百科",而不是"目录"。** 很多人把项目所有知识、所有规范一股脑塞进 AGENTS.md,几百上千行。结果它要么挤爆上下文预算、要么被淹没在噪声里,Agent 根本抓不住重点。正确的做法是:**AGENTS.md 要做"目录",不要做"百科"——控制在 100 行以内,只写最关键的约定和"去哪查",深层知识让 Agent 用工具按需读取。** 这和第二节的"渐进式披露"是同一个哲学:不是让 AI 知道更多,而是让 AI 在恰当的时间知道恰当的事。

**② 把 Prompt/规则当成了"人格台词",而不是"分层宪法"。** Prompt 不是写给 Agent 的角色扮演剧本,而是一套有分层、有优先级的"宪法":系统层、模型层、Hook、CLAUDE.md / AGENTS.md、用户偏好分层注入,并保留各自的优先级。规则之间一旦冲突(比如一条 MUST 和一条 NEVER 打架),Agent 的行为就会变得不确定——它不知道该听谁的。

顺带分清两份文件的定位(来自社区对 Claude Code 与 Codex 的对比):**CLAUDE.md 偏"经验收编 / 现场记忆"(把踩过的坑、用过的偏好沉淀下来),AGENTS.md 偏"制度挂载 / 结构化注入"(把团队制度、规范结构化地挂上去)。** 一个是现场长出来的,一个是制度自上而下定的。

> **一句判断**:规则文件写了 Agent 不遵守,十有八九不是 Agent 不听话,是你把它写成了百科、或写成了互相打架的台词。好的规则文件是一份 100 行内、分层、不冲突的"目录式宪法"。

## 二、不是越多越好:怎么"选"上下文(Q2)

这是真实产品里最难、却最容易被跳过的一步。

大多数人一想到"上下文管理",第一反应是"压缩"——窗口快满了,砍掉一些。但**压缩只是泄压手段,选择才是第一性问题。** 在压缩之前,有一个更根本的判断:

> **这一轮,到底该带什么上下文?**

带得多,不等于带得对。塞进去一堆不相关的代码和历史,模型反而被噪声干扰、更容易选错。所以上下文工程的第一步不是"塞",是"选"。业界有三种由被动到主动的选择机制:

**① 条件加载(Cursor 的 glob 触发)**:Cursor 把规则拆成 `.cursor/rules/*.mdc`,每个规则带 frontmatter 声明它对哪些文件(glob)生效。没命中的规则根本不进 prompt——这是一种"懒加载式选择"。局限是:它只能按文件路径触发,不能按任务语义判断。

**② 渐进式披露(Skills 两层)**:先只给模型工具/技能的名称和一句话描述,命中了再加载完整说明(这是 E3 讲过的)。本质也是"先判断要不要,再决定带不带"。

**③ Agent 自己找(RAG → Grep 的演进)**:这是最主动的一种。早期大家用 RAG——预先把代码库索引好,每轮检索相关片段"喂"给模型。后来 Anthropic 发现,与其喂,不如给 Claude 一个 Grep 工具,让它**自己**去代码库里搜、自己一层层建上下文;再进一步,用一个子 Agent 在它自己的上下文里搜文档、只把答案回传,主对话的上下文始终保持干净。"喂 context"变成了"Agent 自己选 context"。

选择的本质,是**排序 + 优先级**:按相关性、新鲜度、来源可信度、token 预算给候选上下文排个序,只带最相关的那一批。

> **一句判断**:带得多不等于带得对。上下文工程的第一步不是"塞",是"选"——按任务语义决定这一轮让模型看见什么。选错了,后面压缩得再好也是在优化噪声。

## 三、窗口快满了:压谁、留谁、丢谁(Q3)

选完了还是会满——长任务跑几十轮,历史只会越来越长。这时才轮到压缩。压缩的关键不是"砍长度",是**先分类,再决定动谁**:

| 类别 | 处理方式 | 例子 |
|---|---|---|
| **常驻(必须保留)** | 永不压 | 系统规则、关键约束、当前任务目标 |
| **按需召回** | 压成引用,要用再取回 | 历史对话、读过的文件、检索片段 |
| **可丢弃** | 直接删 | 过期的 tool result、冗余/低价值消息 |

分好类,压缩就只动后两类——把"按需召回"的内容卸载到外部(要用时再拉回来)、把"可丢弃"的直接删掉,而**永远不碰常驻区的关键约束**。

这套"卸载"的威力有真实数字。腾讯云一个 Agent Memory 实践里,把上下文正确地卸载到外部(而不是全堆在窗口里),**省了 61% 的 Token、成功率提升了 52%**。Manus 也用同一个思路:把 observation、长文件内容这些易变的大块下沉到文件系统,当作外部记忆,既不撑爆窗口,也不污染那段需要稳定的前缀。

> **一句判断**:压缩不是无脑砍长度,是先按"常驻 / 按需召回 / 可丢弃"分类,只动后两类。关键约束一旦被压没,Agent 就开始"忘事"——这正是开篇那个"跑久了忘约束"现象的根因之一。

## 四、四级 Compact:它是 checkpoint,不是摘要(Q4)

现在拆镇文图的 §5(四级压缩链)和 §4(token 预算)。先纠正一个最常见的误解:

> **Compact 不是"给对话做个总结",而是"泄压保命 + 保留关键状态"的 checkpoint(检查点)。**

把它当摘要,你会把它做成"概括一下前面聊了啥";把它当 checkpoint,你才会去想"压完之后,Agent 还能不能接着干活"。镇文图把它画成一条**从轻到重的四级降级泄压链**:

| 级别 | 程度 | 动作 | 何时触发 |
|---|---|---|---|
| ① **MicroCompact** | 轻 | 移除过期的 Tool Result / 低价值结果 | 轻微超量,先清垃圾 |
| ② **Snip** | 中轻 | 删除冗余消息 / 低价值消息 | 还超,删可丢弃的 |
| ③ **AutoCompact** | 中重 | 对历史对话做摘要、保留必要状态 | 仍超,把中间大量消息归纳成 Summary、两头保留关键消息 |
| ④ **Reactive Compact** | 重 | 熔断式压缩,prompt 太长时兜底保留核心信息 | prompt too long,最后一道防线 |

注意这是**逐级降级**:先用最轻的手段(清过期垃圾),不够再删冗余,还不够才做摘要,实在不行才熔断式兜底。压完会重新计算 Context,如果还超长,就继续往下一级升级压缩(镇文图 §5 那条回流虚线)。

**为什么要留 Compact Buffer?** 看镇文图 §4 的 token 预算公式:

```
总 tokens ≈ 历史上下文 + 新增消息 + 下次返回预留 + Compact Buffer
```

Compact Buffer 是"给后续续期预留的安全余量"。很多人觉得留 buffer 是浪费窗口,其实正相反——**留 buffer 是为了让压缩本身有空间执行**。如果你把窗口塞到 100% 才想起来压缩,压缩动作自己都没地方落脚了。

这里有个 Anthropic 官方亲自记录的真实痛点,放在它的 memory 文档 troubleshoot 章节里:"**Instructions seem lost after /compact**"(压缩之后,指令好像丢了)。这恰恰印证了第三节那条铁律:关键约束必须放在常驻区、不能进压缩;而且 compact 的目标是"保留状态"而不是"做个摘要把约束也概括没了"。

> **一句判断**:好的 compact 是一条分级泄压的 checkpoint 链——从清垃圾、删冗余、做摘要到熔断兜底,逐级降级;关键约束永远在常驻区,不被任何一级压掉。

## 五、section 切分 → cache 命中率:这是 prompt 架构,不是服务端优化(Q5)

这一节是整章的"经济学基础"。先纠偏:

> **Cache(缓存)不是推理框架的服务端优化细节,而是 PM 在设计 prompt 时就要定的架构问题。**

拆镇文图的 §1(三级缓存)和 §2(静态/动态区)。

**为什么 cache 对 Harness 是生死攸关的基础设施?** 一个 coding agent 的单次请求由四段组成:system prompt(5k-20k tokens 的工具说明、风格指引)+ 工具定义(2k-10k tokens)+ 历史对话(变长)+ 当前消息。前两段在整个 session 内几乎不变——这是 prefix-stable workload 的教科书形态。而一个写代码任务平均要跑 20-50 轮,每一轮都重发完整上下文。如果没有 cache,等于把那段几万 token 的 system prompt 重传 50 次。

**section 怎么切,直接决定能不能命中 cache。** 原理很简单:缓存匹配的是"前缀"——只有从头开始连续相同的那一段才能复用。所以:

- **稳定内容放前面、放静态区**:system、tools、项目规则,放在请求最前面;缓存标记(cache breakpoint)落在这段稳定片段的末尾。
- **动态内容放后面、放动态区**:session、env、时间戳、随机 ID、临时状态,统统放到 Boundary 之后。
- **绝对不要**把时间戳、随机 ID、每轮都变的东西塞进前缀里——那会让整个 cache 前功尽弃。

把镇文图 §2 那条 Boundary 切对了,稳定前缀就能稳定命中。

**五个产品的 cache 策略横评(给你一张可抄的对照表):**

| 产品 | 控制粒度 | 命中折扣 | 命中率 | 特点 |
|---|---|---|---|---|
| Claude Code | 显式 cache_control breakpoint(最多 4 个) | 命中仅付 10% | 10 轮后稳定 85-92% | 控制最细,可分段续约 TTL |
| Codex | 自动前缀匹配(>1024 token 启用) | 命中付 50% | 70-85% | 零配置,但边界不可控 |
| Cursor | 透传上游模型 | 看上游 | 不公开 | @-mention/多文件重排是隐形 miss 来源 |
| Manus | 应用层 KV + 文件 offload | 自实现 | 前缀近 100% | append-only + stable tool ordering + masked tool selection 三原则 |
| Hermes | 无(函数调用专注) | N/A | N/A | cache 由推理框架处理 |

**几个能直接用的真实数字**:典型 coding session 在 10 轮后命中率稳定在 85-92%;盈亏平衡点在 2-3 轮,之后每轮节省约 80% 的输入成本;Manus 靠文件 offload 把 50k+ token 的上下文压到 5k 级,单任务 LLM 成本降到 1/10。

**一个高阶但极其重要的冲突——cache × compact 天然矛盾**:compact 会改变上下文内容,而一旦它动到了原本稳定的前缀,就会破坏缓存、触发 cache miss。**cache 要的是稳定,compact 要的是动态泄压,二者本质上是打架的。** 解法在镇文图里其实已经画出来了:把**稳定 section**(放前面、进缓存)和**动态的 compact state**(Compact Buffer 区)分开,别让压缩动作污染稳定前缀。

> **一句判断**:cache 命中率是固定价格 Agent 产品的生死线——命中率从 60% 提到 85%,毛利可能从 -10% 翻到 +40%。它不是后端调参,是 PM 设计 prompt 架构时就要拍板的:哪些放前面、哪些放后面、Boundary 切在哪、compact 会不会污染前缀。

# 下半场 · 长期记忆怎么管

## 六、记忆生命周期:写入 / 召回 / 更新 / 隔离 / 遗忘 / 防投毒(Q6)

上半场讲的都是"这一轮"的事。但 Agent 真正的价值在于跨会话——今天交代过的偏好,明天还记得;这个项目的规则,下个任务还在。这就是记忆。先纠偏:

> **Memory(记忆)不是一个存储方案(选 Redis 还是数据库),而是一套生命周期管理。**

**两套机制(Anthropic 官方)**:每个 Claude Code 会话都从空白上下文开始,靠两套东西跨会话带知识——`CLAUDE.md`(你手写的持久指令)和**自动记忆**(Claude 根据你的更正和偏好自己写的笔记,有 `MEMORY.md` 入口)。关键是:自动记忆需要 **workspace trust 门控**才生效(和 hooks 同一道安全门)。这背后的产品判断是——**Agent 自己写的东西,不能默认就信。**

**记忆要分层(三层,真实工程方案,不能一把 KV 通用)**:
- 短期(秒级响应)→ Tair 路线
- 中期(语义召回)→ Mem0 路线
- 长期(持久化与索引)→ Tablestore 路线

分层不是炫技,是被工程坑逼出来的。淘宝闪购的 Tair 短期记忆实践里就踩过:**并发上来后,记忆延迟从 5ms 涨到 50ms、在途请求膨胀 10 倍**。记忆系统一旦扛不住并发,整个 Agent 的响应都会被拖垮。

把记忆当生命周期,要管六个环节:

- **写入**:什么该进长期记忆?——稳定偏好、反复出现的项目规则、经用户确认的事实、架构决策、明确的禁止事项。什么**不该**进?——临时任务细节、未验证的猜测、敏感信息、外部网页里的指令、一次性状态。
- **召回**:按相关性和 scope 取,不是每次全量塞回去。
- **更新**:新事实覆盖旧的,做版本化,别让过期事实和新事实并存打架。
- **隔离**:给记忆分作用域和可信度——

  | 维度 | 分类 |
  |---|---|
  | 按 scope | user / project / repo / task / agent 级 |
  | 按可信度 | human-authored(人写)/ system-generated(系统生成)/ agent-inferred(Agent 推断)/ external-imported(外部导入) |
- **遗忘**:TTL 过期、用户手动删除、低频使用降权、被新事实覆盖、失效标记、tombstone(墓碑标记,防止删掉的旧记忆又被复活)。记忆系统不会遗忘,迟早被垃圾撑死。
- **防投毒**:OWASP 专门立了一个项目叫 **Agent Memory Guard**,定义了 memory poisoning(记忆投毒)——持久记忆被污染,导致 Agent 失准、甚至数据外泄。防护要点:外部内容默认**不能直接写**长期记忆;写入前做安全扫描;每条记忆记录 source(来源)和 trust level(可信度);区分 data 和 instruction(别把网页/邮件里的文本当成系统指令);高风险记忆需要人工审批。(完整的提示注入防护是 E6 的主场,这里只点到记忆这一侧。)

要特别强调一条:**记忆投毒不一定是攻击,也可能是 Agent 自己压缩时错误归纳。** Agent 在 compact 时把一次失败的尝试错误地总结成"经验",下次就接着错——这就是开篇说的"把错误经验记到下次"。所以:

> **关键判断**:Agent 自己写的记忆,默认不可信。应该给它低初始置信度、标记来源、可审计、可删除,必要时要人工或系统确认。

> **一句判断**:记忆决定 Agent 下一轮是"越做越聪明"还是"越做越脏"。没有生命周期管理的记忆,迟早把错误经验、投毒指令、过期事实越攒越多,把一个本来好用的 Agent 喂坏。

## 七、拔高:上下文与记忆,是 Agent 的"信息操作系统"

走到这里回头看,E4 表面讲的是上下文和记忆,实际讲的是 **Agent 的信息架构**——它的"信息操作系统"。

- 一个**不成熟**的 harness,会把所有东西都一股脑塞进 prompt。
- 一个**成熟**的 harness,会把信息分成:常驻 / 按需 / 可丢弃 / 可缓存 / 可记忆 / 必须隔离。

这正好接回 E1 立下的"双核能力":

- **架构设计能力**在这里的体现 = section 怎么切、Boundary 切在哪、记忆怎么分层和隔离。这些都不是"功能",是为了缓存命中、成本可控、长程可靠而做的**架构决策**。
- **评估能力**在这里的体现 = 用 cache 命中率、token 成本、"压缩后有没有丢约束"这些可量化的指标,来判断这套信息架构到底好不好。

一句话收束:

> **上下文与记忆,不是模型能力的附属品,而是 AI Agent harness 的信息操作系统。** 上下文管理决定 Agent 这一轮能不能做对;记忆管理决定它下一轮会越做越聪明还是越做越脏;cache 和 compact 则决定这套系统能不能在真实成本下长期跑起来。Agent 不是忘性大,是信息架构没设计好。
