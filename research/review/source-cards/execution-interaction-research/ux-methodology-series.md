# UX 方法论与过程可视化调研摘要（4篇）

## 08 · Agent 时代的界面为什么反着做（人人都是产品经理）

**来源**：https://www.woshipm.com/ai/6410617.html

**核心观点**：
- 上一代AI界面是"输入→等→输出"，Agent时代反着做："输入→看它干→随时拦→得结果"
- 过程可见是核心，但本质是一条"安全带"——为还不可靠、还慢的模型量身配的

**"大事清楚小事糊涂"设计黄金法则**：
- 大事(改文件/删数据/花钱/发消息)：必须让用户清楚、必须确认
- 小事(读文件/搜索/草稿)：糊涂过去，不打扰
- 不是所有事都透明=好，信息轰炸=通知盲，用户反而忽略一切

**过程可见的隐性价值**：
1. 建立信任：让用户看到思考过程，才敢把更重要任务交出去(玩具→工具的关键一步)
2. 培训用户能力边界：看多了就知道什么任务能信任、什么要干预(隐性培训)
3. 纠错入口：过程可见让用户能在出错前提早拦截

**自治度滑块（Karpathy）**：
- 人机协作的自治度是个滑块，会往上滑
- 今天停在偏"人盯"那档，因为模型还没到；模型到了，滑块往自治走，精心设计的闸门就到退场时
- 终局：界面退到背景，只在真正需要做决定的岔路口浮上来

**对终局的预判**：
- "看着它干"本身就是瓶颈，让人成了流水线最慢的一环
- 今天拼命设计的"看得见的过程"，也许正是为了有一天让它们体面消失
- 但只要模型不是100%可靠，界面不会完全消失，会演变成"例外处理模式"——只在置信度低时请求确认

**对交互设计的启示**：
- 小蓝鲸一期处于"模型还不够可靠"阶段，过程可见是必须的安全带，但要做"大事清楚小事糊涂"
- 自治度滑块 = 小蓝鲸的权限模式(Safe→Balanced→Auto)，一期停在 Balanced(偏人盯)，Auto 是滑块上滑方向
- 交互设计要为"闸门退场"留口子：今天做的过程展示，架构上要能随模型变强而减少打扰

## 09 · Agent UX & Transparency（Max Gherman）

**来源**：https://www.max-gherman.dev/design/anatomy-of-ai-agent/agent-ux-and-transparency/

**流式部分结果（Streaming）**：
- 最大的UX改进：流式。用户看 token 逐字出现，感知2秒为"快"；盯空白屏2秒感知为"坏"
- 流式给用户早期信号判断 Agent 是否理解意图，可在完成前中断重定向
- 工具执行时用户看到"nothing"——除非你告诉他们。必须发状态事件消除 dead air

**渐进式披露三层模型（核心）**：
- **Layer 1 — The Answer**：最终答案 + 置信度。默认展示。用户90%时间只需这个
- **Layer 2 — The Evidence**：来源、工具结果摘要、可点击引用。折叠或可展开。用于验证
- **Layer 3 — The Trace**：完整思维链、每个工具调用(含参数和响应)、时序数据、模型选择。调试和审计用，绝不强制

**自适应详情级别**：
- 低风险高置信：只展示答案。如"函数定义在第42行"
- 中风险中置信：答案+证据。如"基于3个匹配文档，政策允许..."
- 高风险低置信：答案+证据+主动暴露不确定性。如"我找到冲突信息——这是两种解读"

**推理结构化摘要**：
- 不要 dump 原始思维链(冗长、重复、对非专家困惑)
- 提取关键决策点：做了什么动作 / 为什么(1-2句) / 考虑了什么替代方案 / 置信度
- 给用户清晰"审计轨迹"而无需解析数百 token 内部 deliberation

**引用与归属**：
- 答案依赖检索信息时，必须链接回来源
- 两个目的：让用户验证声明；让无支撑声明视觉上明显(没引用=危险信号)，限制幻觉损害

**对交互设计的启示**：
- 小蓝鲸的"默认轻、展开重"= 这里的三层模型。主界面=Layer1，详情抽屉=Layer2，开发者视图=Layer3
- 当前动作卡按风险/置信度自适应展示详细度
- 推理展示用"阶段意图"而非原始思维链——正是 Word 文档"不再每步显示思考，改阶段意图"的依据
- Work 场景的报告必须有来源引用(归属)，Code 场景的 diff 必须有自然语言解释

## 10 · Identifying Necessary Transparency Moments In Agentic AI（Smashing Magazine）

**来源**：https://www.smashingmagazine.com/2026/04/identifying-necessary-transparency-moments-agentic-ai-part1/

**两种极端都错**：
- Black Box(黑盒)：全隐藏保简单 → 用户无力、不信任
- Data Dump(数据倾倒)：流每行日志每个API调用 → 通知盲，用户忽略一切直到出事，出事又缺上下文修

**Decision Node Audit（决策节点审计）方法**：
- 把设计师和工程师拉到一起，把后端逻辑映射到 UI
- 找到 AI 做"判断"的精确时刻(概率性步骤、置信度判断)，这些才需要透明
- 不是所有步骤都需要展示，只展示"判断点"

**Impact/Risk 矩阵**：
- 把决策节点按影响×风险排序
- 高风险高影响：必须展示(Intent Preview，动作前展示意图)
- 低风险低影响：简单日志条目即可
- 矩阵决定每个节点用什么展示模式

**保险理赔案例**：
- 黑盒："计算理赔状态" → 用户不知道有没有看警察报告 → 不信任
- 审计后：发现3个概率性步骤(图像分析/文本审查/赔付计算) → 每个给明确状态消息
- "对比当地供应商价格以保障你的周五配送" 比 "计算置信度阈值" 好100倍

**内容设计师角色**：
- 工程师写技术准确字符串："计算供应商匹配的置信度阈值"
- 内容设计师翻译成建立信任的短语："对比当地供应商价格以保障你的周五配送"
- 用户理解动作和结果

**信任是设计选择**：
- 信任是可预测沟通的机械结果，不是好体验的情感副产品
- 在正确时间展示正确信息建立信任；淹没用户或隐藏机制则摧毁信任

**对交互设计的启示**：
- 小蓝鲸要做 Decision Node Audit：7步闭环里哪些是"判断点"(目标契约、计划、工具选择、质量校验)，这些才需要展示意图
- Impact/Risk 矩阵 = 小蓝鲸权限四级(low/medium/high/critical) 的展示版：不同风险用不同展示模式
- 权限文案必须用"用户影响"表达，这正是内容设计师翻译的实践
- "动作前展示意图(Intent Preview)" = 小蓝鲸高风险确认卡的核心

## 11 · 透明度与可解释性：思考图谱可视化（CSDN）

**来源**：https://blog.csdn.net/weixin_41455464/article/details/156517085

**思考图谱 vs 检索依据（两个可解释性维度）**：
- **思考图谱(Thought Map)**：从接收请求到生成响应的完整内部推理过程。包括分解问题、选工具、执行、观察、自我修正的每步
- **检索依据(Retrieval Basis)**：推理中为获取外部信息做的检索，以及用了哪些信息片段形成决策

**三级透明性设计**：
1. **概要级**：只展示最终结论和关键依据来源
2. **过程级**：展示思考步骤的序列(非线性，含分支/回溯/重试)
3. **原始级**：完整事件流(每个LLM调用的prompt/output、每个工具的input/output)

**事件埋点类型**：
- AGENT_START / LLM_CALL_START / LLM_CALL_END / TOOL_CALL_START / TOOL_CALL_END / RETRIEVAL_START / RETRIEVAL_END / AGENT_THOUGHT / AGENT_DECISION / AGENT_ERROR / AGENT_END

**复杂多步骤推理的挑战**：
- 动态性：执行路径非预设，根据实时环境和LLM决策动态生成
- 非线性：可能分支、回溯、重试
- 工具调用：外部工具的输入输出也需追踪
- 一个复杂会话可能几十上百步，UI 实时展示需高效捕获/传输/渲染

**对交互设计的启示**：
- 小蓝鲸的 event_log 表(04契约)已经覆盖了这些事件类型(thinking/tool_call/tool_result/approval_request等)
- 三级透明性 = 普通用户视图(概要级) / 开发者视图(过程级+原始级)
- 思考图谱的非线性(分支/回溯/重试)在 UI 上要简化：默认线性阶段展示，回溯/重试折叠进"查看详情"
- 检索依据 = Work 场景的来源卡，每条结论指回来源
