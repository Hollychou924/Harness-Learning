# Vibe Coding 神级Prompt与思维模型全景

> 一句话：卡兹克的"第一性原理+对抗式审查"只是一个代表，实际上国内外已经形成了一个围绕「怎么跟AI写代码最有效」的完整方法库。这些方法按作用可以分为四大类：**思考类**（让AI想清楚再做）、**验证类**（做完全面检查）、**工作流类**（怎么分工、分步、迭代）、**元规则类**（让AI用正确的姿态和你合作）。

## 0. 背景：Vibe Coding到底是什么

这个词是 **Andrej Karpathy**（OpenAI联创、前Tesla AI总监）2025年2月在X上首次提出的：

> "There's a new kind of coding I call 'vibe coding', where you fully give in to the vibes, embrace exponentials, and forget that the code even exists. It's possible because the LLMs (e.g. Claude Sonnet) are getting too good. I just see things, say things, run things, copy paste things, and it mostly works. I code a project on the weekend that's actually pretty decent and I haven't written a single line of code in the normal sense. I just speak, and the code happens."
> （有一种新的编程方式叫"vibe coding"，你完全顺着感觉走，拥抱指数级变化，甚至忘了代码本身存在。我周末写了个挺像样的项目，但按正常意义我一行代码都没写——我只是说话，代码就出来了。）

但他后来也补了一句关键的话：**vibe coding最大的风险是"你以为它能跑"——而实际上它没跑，或者在你没看见的地方炸了。** 所以这一轮围绕vibe coding的方法论，本质上都是在回答一个问题：**当你"不动手写代码只动嘴"时，怎么保证最后出来的东西是对的。**

下面按四大类整理。

---

## 一、思考类Prompt：让AI在做之前先想清楚

这类Prompt作用在"动手之前"，防止AI直接上手写一坨看起来对其实有问题的东西。**核心特征：逼AI慢下来、回到本质、想清楚再做。**

### 1. 从第一性原理出发（卡兹克）

**用法**：在修bug/做方案/设计架构时，在Prompt末尾加一句"从第一性原理出发"。

**作用**：打断AI的"类比推理"惯性——AI倾向于在训练数据里找最像的方案拼一个出来。这句话逼它回到问题本质重新推导，而不是抄"行业惯例"。

**案例**：AIHOT飞书推送bug，AI一开始说"是测试国产模型改坏了OpenAI抓取，修一下就行"；加了这句话之后它找到了底层流量路由层的一个深层隐患——表层修只是给破船打补丁。

### 2. 先给我计划，我确认后再做 / Plan Mode（几乎所有社区）

**用法**："先别写代码，给我一个详细的计划，包含要改哪些文件、每一步做什么、可能的风险，我确认之后再开始写。"

**作用**：逼AI在动手前把思路显性化，避免它一头扎进某个方向改了一堆东西才发现走错。这也是Codex/Cursor/Claude Code都内置"Plan mode"的原因。

**来源**：
- 所有头部Agent工具都内置Plan模式
- superpowers技能`writing-plans`和`executing-plans`
- 宝玉xp提示词Level 7："给AI一个清晰的输出格式要求"

### 3. Rubber Duck Debugging / 小黄鸭调试（海外社区）

**用法**："先不要修bug，给我一段一段解释这段代码是怎么工作的，就像我是一只橡皮鸭一样。"

**作用**：强迫AI先把它对代码的理解"讲出来"，很多bug在讲的过程中就暴露了。人类调试的经典方法套到AI身上照样管用。

### 4. 五问为什么 / 5 Whys（源自丰田生产方式）

**用法**："连续问五次'为什么'，直到找到根因而不是表面原因。"

**作用**：和"第一性原理"思想一致，但更结构化。第一次回答通常是直接原因，连续追问才能挖到根因。superpowers的`systematic-debugging`技能就是这套逻辑的工程版：**"NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST"**（没有根因调查前不许动手修）。

### 5. Step-by-step思考 / 分步骤思考（经典CoT）

**用法**："Think about this step by step. Don't skip any steps."（中文："一步一步想，不要跳步。"）

**作用**：最基础也最有效。让AI显式列出推理链条，减少跳步出错。OpenAI o1/Claude Opus等模型虽然内置了thinking模式，但在复杂任务上显式要求"分步骤"仍然能显著提升准确率。

### 6. 假设自己是这个领域最顶尖的专家（Persona设定）

**用法**：
- "你是一个有20年经验的资深SRE，以极端保守、考虑所有边界情况著称"
- "你是一个吹毛求疵的代码审查员，以发现别人看不见的bug为生"
- "你现在的任务是找出这段代码所有可能出问题的地方，而不是让它跑通"

**作用**：给AI一个明确的角色设定，改变它的默认"讨好式输出"倾向。这是"对抗式审查"的前置工作——让它进入挑刺模式而不是帮忙模式。

## 二、验证类Prompt：做完之后全面检查

这类Prompt作用在"交活前"，防止AI自己觉得做完了但实际上漏洞百出。**核心特征：切换视角、主动找错、留证据。**

### 1. 对抗式审查 / Adversarial Review（卡兹克）

**用法**："开启多Agent并发，站在恶意攻击者/挑剔用户/三个月后维护者三个角度挑毛病，覆盖边界/性能/正确性/文档一致/回归/安全六个维度。"

**作用**：卡兹克的原创。通过多Agent并发从不同视角主动攻击代码，找出自己写的时候想不到的坑。典型案例：6月初审查AIHOT找出了OOM死循环（大任务爆内存→被杀→重试→再爆）、未来时间污染（信源时区错导致"明天的文章"排到信息流最前面污染所有渠道）等多个隐藏炸弹。

### 2. Evidence Before Claims / 证据先行（superpowers: verification-before-completion）

**用法**：在AI准备说"搞定了""测试通过了""bug修好了"的时候，强迫它：
1. 明确"什么命令能证明这个说法？"
2. 现场跑一遍那个命令
3. 看完输出、数一下失败数
4. 输出"xxx测试通过，34/34全部pass"，附上证据
5. 才可以说"搞定了"

**铁律**：**NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE**（没有当场验证的证据，不许说完成）。禁止"应该可以了""我很确定""看起来没问题"这类措辞。

**作用**：治AI最常见的毛病——"我觉得我改完了"但根本没跑过。这是vibe coding最致命的坑。

### 3. Red Team / 红队测试（海外安全社区）

**用法**："假设你是一个恶意用户，你的目标是让这个系统崩溃/绕过权限/泄露数据/做出危险操作。你会怎么做？把所有攻击路径列出来。"

**作用**：比"对抗式审查"更聚焦安全和权限边界。适合涉及用户数据、支付、权限、文件系统操作的功能。

### 4. Edge Case Enumeration / 边界穷举

**用法**："列出所有可能的边界情况：空输入、超大输入、特殊字符、未来时间、网络失败、并发、权限不足、磁盘满、依赖缺失、第三方接口变了、编码问题、时区问题... 一个个看代码是否处理了。"

**作用**：vibe coding出来的代码最容易在happy path上跑得好好的，边界全是洞。穷举边界是系统化补洞的最快方法。

### 5. Self-Critique / 自我批判

**用法**："现在切换角色，你是一个严格的批评家。指出你刚才给的方案里所有的问题、风险、假设和可以改进的地方。"

**作用**：让AI自己跟自己辩论。相比一次生成，可以把错误率降低30-50%。这也是OpenAI和Anthropic内部都在用的"Self-Consistency"技术的口语版。

### 6. Read the diff as if you're reviewing a PR（像审PR一样看你自己的diff）

**用法**："现在把你刚才写的所有改动当成一份别人提交的PR，像你是最严格的reviewer一样读一遍。每个你会评论的地方都列出来。"

**作用**：让AI跳出"我刚写的肯定是对的"的自利偏差，用审查别人代码的苛刻眼光审查自己的代码。Cursor的code-review模式、Claude Code的code-review命令都是这个逻辑。

### 7. Regression Check / 回归检查

**用法**："你这次改动，可能会影响哪些已有功能？列出来，一个个检查是不是还正常工作。"

**作用**：修了A坏了B是vibe coding第二大死因（第一大是"以为自己修好了"）。显式让AI列回归清单并逐个验证。

## 三、工作流类Prompt：怎么分工、分步、迭代

这类Prompt不是一锤子买卖，而是把整个开发过程组织起来的方法论。

### 1. Spec-Driven Development / 规格驱动（小爱PM规则第7条）

**核心思想**：**先写清楚"什么算好"，再让AI做。** 写评测/spec/acceptance criteria，而不是写PRD。AI时代PM最核心的技能从"写需求文档"变成"写评测标准"。张和的说法叫"Evals as PRD"（评测即需求）。

**做法**：写一份明确的spec：输入是什么、输出长什么样、边界条件、成功标准、反例。然后让AI"先让所有测试通过"。这是superpowers工作流的核心，也是工业界最推荐的vibe coding升级路径。

### 2. TDD with AI / AI辅助测试驱动

**用法**："先写测试（描述我期望的行为），测试现在应该是失败的，然后再写实现让测试通过。"

**作用**：经典的TDD套在AI上更有效——测试就是那份"什么算好"的规格书，AI最擅长的就是"让测试通过"这件事。superpowers的`test-driven-development`技能强调：80%+覆盖率要求。

### 3. Baby Steps / 小步前进（海外社区共识）

**用法**："一次只改一个东西，改完验证一次，通过了再改下一个。不要一次改一大坨。"

**作用**：vibe coding最容易翻车的方式就是一次让AI改十几个文件。小步提交、每次可以回滚、每个步骤有验证点，出问题能快速定位。这也是`verification-before-completion`的配套心法。

### 4. Subagent Delegation / 子任务委派（Anthropic/Cursor主推）

**用法**："对于独立的子任务，启动一个子Agent去做，完成后由主Agent整合并验证。主Agent负责规划和验收，子Agent负责具体执行。"

**作用**：
- 一个Agent上下文太长时容易"走神"和遗忘，拆给子Agent可以保持每个Agent专注
- 可以并行跑多个子任务，速度快
- 主Agent可以作为"审查者"，比自己做自己审更客观

Claude Code的Subagents、Cursor的Background Agents、Codex的多任务都是这套。Anthropic官方博客专门写过"什么时候用subagent"。

### 5. Parallel Attack / 并发攻击（卡兹克的Ultracode玩法）

**用法**："针对这个问题/这次审查，启动N个独立Agent并发处理，最后投票/合并结果。"

**作用**：利用AI每次运行有随机性的特点。对审查/找bug/方案设计这种"找多不嫌多"的任务，并发跑几个Agent独立做，合并结果比单个Agent做得深得多。卡兹克说"开40个Agent并发审查"就是这招。

### 6. Context Management / 上下文管理（Cursor强调）

**用法**："我不需要你把所有文件都读进来。只读你真正需要的最小相关文件。需要更多上下文的时候主动告诉我，我来决定给不给你。"

**作用**：上下文窗口不是越大越好——塞太多无关内容AI反而会"迷路"。Cursor的"动态上下文发现"就是这个思路：给AI工具让它自己按需拉上下文，而不是提前喂一大堆。

### 7. Iterative Refinement / 迭代打磨（Simon Willison等）

**用法**："先给我一个粗糙的能跑的版本，不要追求完美。然后我们一轮一轮改，每一轮只改一个点。"

**作用**：对抗"过度工程化"。vibe coding最容易的另一个毛病是AI一次就想给你搞一个"完美架构"，结果做了一堆你不需要的东西。先有再优，迭代打磨。

### 8. Session Reset / 开新会话（实战经验）

**用法**：如果一个会话里Agent越聊越糊涂、开始犯低级错误、或者上下文已经塞了太多失败尝试，果断开新会话，把问题用干净的上下文重新描述一遍。

**作用**：上下文污染是Agent开发的隐性杀手。一个长会话里前面的错误尝试会持续影响后面的判断。Simon Willison和很多独立开发者都强调"长会话超过一定长度就开新的"。

## 四、元规则类Prompt：调整AI和你合作的姿态

这类Prompt不是解决某一次任务的，而是设定AI整体的工作态度和规则，可以写进全局AGENTS.md/CLAUDE.md里长期生效。

### 1. 沟通铁律（小爱PM版/你的全局Codex AGENTS.md已经有）

核心几条：
- 全程大白话，不要堆术语
- 只说能做什么/不能做什么/有什么影响/需要我做什么决策
- 不确定的地方停下来问，不要瞎猜
- 先把你的理解和假设说给我听
- 结论必须告诉我"你查了哪些地方，覆盖全没全"

### 2. 怎么简单怎么来（KISS原则）

**用法**："用最少的代码/最小的改动解决问题。不要画蛇添足。不要加我没要求的功能。不要抽象、不要通用化、不要加配置项。不可能发生的情况不处理。"

**作用**：对抗AI的"过度工程化"倾向——AI喜欢给你做一个高度抽象、高度可配置、高度通用的版本，结果你本来要修个按钮却出来一个框架。这条是很多独立开发者最常挂在嘴边的规则。

### 3. 动最少的代码/只改必须改的

**用法**："严格跟现有代码风格保持一致，不顺手改旁边的代码/注释/格式。只删你自己改出来的没用东西，原来的不要动。看到旧代码没用告诉我就行，不要自己删。"

**作用**：防止AI当"热心邻居"——帮你改完bug顺便把旁边它觉得写得不好的代码也"优化"了，结果引入新bug。这一条你已经在Codex AGENTS.md里写了，非常重要。

### 4. Honesty Policy / 诚实政策

**用法**："如果你不确定，直接说你不确定。如果你不知道，说你不知道然后去查。不要猜。不要编。不要因为想让我高兴就说做完了。"

**作用**：治AI最烦人的一个毛病——"幻觉式完成感"（它根本没做完/没跑过，但会用非常自信的语气告诉你做完了）。

### 5. Ask Before Acting on Destructive Actions / 先问再做

**用法**："删除文件、运行危险命令、推送到远程、修改数据库、发消息之前，必须先告诉我要做什么、会有什么后果，等我明确说可以再做。"

**作用**：安全底线。几乎所有Agent工具都有approval mode，但显式写在全局规则里双保险。

### 6. Constraints First / 约束先行（卡兹克"洁癖Skill"）

**用法**："开始之前先明确告诉我这次任务的约束：哪些文件不能动、哪些风格要遵守、用什么技术栈、不能引入什么依赖、性能/安全有什么硬要求。"

**作用**：卡兹克的"洁癖Skill"就是这套——提前把所有约束列清楚，避免AI按自己的喜好做出来不符合你口味的东西。

### 7. Teach Back / 复述确认

**用法**："开始之前先用你自己的话复述一下我的需求和你理解的任务，确保你理解对了。如果有歧义就问我。"

**作用**：防止"我以为你说的是X其实你说的是Y"的误解浪费时间。特别是需求描述不清、任务模糊的时候最有用。

## 六、这些Prompt/规则怎么组合用——实战推荐工作流

把上面这些按"做一次功能开发"的流程串起来，推荐的标准流程：

### 开始前（元规则层，全局写死在AGENTS.md）
- ✅ 沟通铁律（大白话、先说理解、不确定就问、结论带证据）
- ✅ 怎么简单怎么来
- ✅ 动最少的代码
- ✅ 诚实政策/没跑过不许说搞定
- ✅ 危险操作先问
- ✅ 开始前复述确认

### 接到需求（思考层）
1. **Teach Back**：先复述需求，确认理解一致
2. **Constraints First**：明确约束（不能动的、风格、技术栈）
3. **Plan Mode**：让AI给计划，你确认再做
4. **Spec/TDD**：如果是新功能，先写清楚"什么算好"或者写测试
5. 如果涉及找bug/做方案，加**"从第一性原理出发"**+**5 Whys**

### 开发中（工作流层）
6. **Baby Steps**：一次只改一个点，改完验证
7. **Subagent委派**：独立子任务让子Agent做
8. **Session Reset**：长会话糊涂了果断开新会话
9. **先粗糙能跑，再迭代打磨**

### 交活前（验证层）
10. **Evidence Before Claims**：跑测试/命令，有证据才能说"搞定了"
11. **对抗式审查/Red Team**：多视角并发挑毛病
12. **Edge Case穷举**：边界一个个查
13. **Read your own diff as a PR**：用审稿人眼光读自己的diff
14. **Regression Check**：老功能有没有受影响
15. **Self-Critique**：自己批判一遍自己的方案

### 定期（比如每2-3周）
16. **全局体检/full-audit**：第一性原理复盘整个项目+分模块对抗审查，找技术债和防线漏洞
17. **失败case沉淀进评测集/检查清单**：这次踩的坑下次自动拦住

这套组合下来，基本上覆盖了vibe coding 90%以上的翻车场景。

## 七、哪些Prompt值得立刻加进全局配置

6. **Constraints First / 先说约束**——动手前AI必须先问清/列出约束。

- 每次都穷举所有边界（日常小改动没必要）
- 每次都写Spec/TDD（探索性任务先粗做再迭代）

## 八、最后的判断：Vibe Coding的两个半真相

1. **真相一：vibe coding不是"跟着感觉走"，而是"感觉只在正确的脚手架里才靠谱"。** 卡兹克、Karpathy这些人看起来"顺嘴一说就搞定"，背后是一套严格的脚手架——计划、验证、对抗审查、小步迭代——只是这套脚手架用多了内化了，看起来像"vibe"。新手只看到表面的"随便说"，没看到底下的规矩，必然翻车。

2. **真相二：所有神级Prompt的本质都是"对抗AI的默认坏倾向"。** AI默认会：讨好你说做完了、抄最像的方案而不是最优解、一次改太多、对自己的输出过度自信、过度工程化、不主动说不确定。所有好用的Prompt都是在治某一个具体的坏倾向。没有万能的"一个Prompt让AI变神"，但有一组Prompt治一组病。

3. **半个真相：工具会进化，但这些思维模型不会过时。** 模型变聪明了、工具变强了，"想清楚再做、做完全面检查、小步迭代、诚实说不知道"这些原则不会变——这些是工程的本质，不是某个模型版本的trick。这也是为什么要把它们写进全局配置而不是每次手打。

## 附录A：神级System Prompt原文（可直接贴进全局配置）

这里是经过全球开发者验证、加进去就能显著提升Agent执行效果的**铁律级规则**。按"治什么病"分类。每条规则都是一个具体的行动指令，不是空话。

### A1. Evidence Before Claims（治"我以为我做完了"病）

**来源**：superpowers工作流`verification-before-completion`，社区公认最有效的单条规则。

**规则原文**：
```
NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE.
（没有当场验证的证据，不许说"做完了"。）

Before claiming anything is done/fixed/passing:
1. IDENTIFY: 什么命令能证明这个说法？
2. RUN: 当场跑一遍完整命令
3. READ: 看完完整输出，数失败数
4. VERIFY: 输出真的证明了你的说法吗？
5. ONLY THEN: 才能说做完了——并且要附上证据。

任何一步跳过 = 撒谎，不是验证。
```

**为什么有效**：AI最大的失败模式是"我觉得我改完了"但根本没跑过。这条规则把"感觉完成"变成"拿出证据才能说完成"。实测单独这条就能减少30-50%的"说做完了实际没做完"事故。

**禁用词**（出现这些词就说明在没验证地声称完成）：
- "应该可以了" / "should work now"
- "我很确定" / "I'm confident"
- "看起来没问题" / "looks good"
- "Great!"/"Perfect!"/"Done!"（在验证之前）
- "测试通过了"（但没贴出测试运行结果）

**对应的验证要求表**：
| 说法 | 必须出示的证据 | 不充分的证据 |
|---|---|---|
| 测试通过 | 测试命令输出：0 failures | 之前跑过、"应该通过" |
| Lint干净 | Lint输出：0 errors | 部分检查、主观推断 |
| 构建成功 | 构建命令 exit 0 | Lint过了、看起来可以 |
| Bug修了 | 复现原症状的测试通过 | 代码改了，以为好了 |
| Agent完成了 | VCS diff显示了改动 | Agent自述"成功" |
| 需求满足了 | 对照需求清单逐条核对 | 测试过了 |

---

### A2. Root Cause Before Fix（治"头痛医头"病）

**来源**：superpowers `systematic-debugging`，与卡兹克"第一性原理"精神一致，工程化表达。

**规则原文**：
```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST.
（没有根因调查前，不许动手修。）

修bug/异常行为的标准流程：
Phase 1 - 根因调查（必须先做完）：
  1. 认真读错误信息（不要跳过，答案经常就在里面）
  2. 能稳定复现吗？精确步骤是什么？
  3. 最近改了什么？git diff、新依赖、配置变化、环境差异
  4. 多组件系统里，在每个组件边界加诊断日志，跑一次看哪一层坏了
  5. 坏的值从哪来？顺着调用栈往上追，找到源头
Phase 2 - 根因定位后，才可以开始修复
Phase 3 - 修复后验证：复现步骤再跑一遍，确认真的修了
Phase 4 - 回归检查：相关的其他功能有没有被影响？

紧急情况下也不许跳Phase 1——急的时候乱猜保证返工。
```

**为什么有效**：和卡兹克"第一性原理"是同一件事，但多了"加诊断日志/分层追数据/复现验证"这些工程动作，不是一句口号。

---

### A3. Baby Steps + Atomic Changes（治"一次改太多收不回来"病）

**来源**：海外独立开发者社区共识、Karat创始人、Tibo、Simon Willison多次强调。

**规则原文**：
```
一次只做一个逻辑改动，改完立刻验证。验证通过了再做下一个。
禁止：
- 一次性改十几个文件然后期待它能跑
- 重构+加新功能+修bug混在一个提交里
- 看到"顺手"可以改的东西就一起改了（"drive-by refactoring"）

每个提交之前，你必须能一句话说清楚这个提交做了什么。
说不清楚就说明改动太大，拆小再做。
```

---

### A4. Teach Back Before Starting（治"我以为你说的是X其实是Y"病）

**来源**：临床沟通、管理学经典技巧，AI编程社区应用极广。

**规则原文**：
```
开始动手之前，先用你自己的话复述一遍你理解的任务：
- 目标是什么
- 要改哪些文件/模块（先列清单）
- 什么算做完（验收标准）
- 有什么约束（不能改什么、技术栈、风格要求）
- 你不确定的地方是什么（要先问清楚）

我确认你理解对了，你再动手。
不允许带着你自己猜的理解闷头做。
```

---

### A5. Plan First, Execute Second（治"上来就瞎写"病）

**来源**：Claude Code/Cursor/Codex全部内置Plan模式；superpowers `writing-plans`。

**规则原文**：
```
对于超过3个文件改动或超过30分钟工作量的任务，先给计划，等我确认再执行。
计划必须包含：
1. 当前状态分析（相关入口在哪里、现在怎么工作的）
2. 拟议方案（为什么选这个而不是别的方案）
3. 要改的文件清单和每个文件改什么
4. 风险点和回滚方式
5. 验证步骤（改完跑什么命令证明成了）

简单/明确/单文件的任务可以直接做，但要保持小步提交。
```

---

### A6. Honesty Policy: Don't Guess, Don't Hallucinate（治"不懂装懂"病）

**来源**：所有AI工具的top1痛点规则。

**规则原文**：
```
- 不确定就说不确定，不要猜。
- 没看过的代码不要说"看过了"，没跑过的命令不要说"跑过了"。
- 如果某个工具/文件/路径你不确定存不存在，先查再说。
- 回答"有几个/默认是什么/能不能"这类数量/是非问题时，必须真的去查代码或配置，不许凭印象说。
- 如果查了还是不确定，明确说"我查了X/Y/Z，但还没确认W"。
```

---

### A7. When Stuck, Stop and Ask（治"越修越歪"病）

**来源**：Simon Willison、swyx等实战派反复强调。

**规则原文**：
```
如果你满足以下任一条件，立刻停下来问我，不要继续瞎试：
- 连续两次修复都失败了
- 发现问题比你最初估计的大很多（动到核心架构/数据模型/公共接口）
- 需要改我没明确授权改的东西（删文件、改依赖、改配置、发请求、发消息、涉及生产数据）
- 发现之前你告诉我的"完成了"其实没完成——立刻告诉我，不要掩盖
- 跑了超过10分钟还没进展

不要为了"显得在工作"而做无谓的尝试。
卡住了就说卡住了，说清楚卡在哪、试过什么、接下来建议怎么做。
```

---

### A8. Red Team Review Before Hand-off（治"happy path病"——和卡兹克对抗式审查一脉相承）

**来源**：海外Red Teaming文化、Cursor代码审查、卡兹克对抗式审查。

**规则原文**：
```
说"做完了"之前，必须自己先当一次恶意审查员，对自己的代码做一遍Red Team：
1. 空输入/缺字段/格式错会怎样？
2. 超大输入/超长文本/100MB文件会怎样？
3. 网络失败/第三方接口返回500/超时时怎么办？
4. 并发调用/重复提交/竞态条件下会怎样？
5. 权限不足/文件不存在/路径写错/编码错误时优雅失败吗？
6. 错误信息会不会泄露敏感数据/栈信息给用户？
7. 日志足够定位问题吗？
8. 回滚怎么做？出了问题能不能一键恢复？
9. 有没有破坏老功能？列出"可能受影响的老功能"清单跑一遍。

发现的问题要么修掉，要么在报告里明确列出。
```

---

### A9. Constraints Up Front（治"做出一个不是我要的东西"病）

**来源**：卡兹克"洁癖Skill"、海外"constraints-first"方法论。

**规则原文**：
```
动手前先确认这些约束，不确认不开工：
- 哪些文件/模块绝对不能动
- 代码风格跟谁保持一致（我会告诉你参考哪个文件）
- 能不能引入新依赖
- 性能/安全/兼容性有没有硬要求
- UI/文案/语言有没有硬要求（比如中文还是英文，术语是否有标准译法）
- 要不要写测试、写文档

如果你发现缺关键约束，立刻问。不要默认你猜的是对的。
```

---

### A10. Prefer Simple, Obvious Solutions（治"过度工程化"病）

**来源**：Unix哲学、Codex/Cursor/Claude Code官方行为准则、你自己的Codex AGENTS.md里已经有了。

**规则原文**：
```
- 用最简单的方案解决问题。能50行搞定不要写200行。
- 只用一次的代码不要抽象、不要泛化、不要加配置项。
- 不要加"可能将来有用"的灵活性——YAGNI (You Ain't Gonna Need It)。
- 不可能发生的边界情况不处理（但要在注释里说明"假设X不会发生"）。
- 选择跟现有代码风格一致的写法，哪怕你觉得它不优雅。
- 如果你写了200行发现50行能搞定，重写。
```

### A11. Karpathy风格77行.cursorrules（综合精髓版）

Karpathy原版是2025年他自己用的.cursorrules，流传极广。下面是社区迭代后的精华版本（非逐字翻译，保留所有核心思想）：

```markdown
# Vibe Coding Rules (Karpathy-inspired)

## 思维模式
- 先思考，再写代码。每写一段代码前，先想清楚它为什么存在。
- 写代码的时候假设你是在给未来6个月后的自己解释——你会忘记一切。
- 解决问题的最好方法通常是最直接的那个。先让它工作，再让它好看。
- 复制粘贴是允许的，但要理解你粘贴的每一行。
- 不要和代码谈恋爱。写得不好的代码就删掉重写，不要试图挽救。

## 工作流程
- 小步提交。每做一件小事就提交一次，提交信息要说明"为什么"而不只是"改了什么"。
- 不要害怕删除代码。删错了可以git恢复，留着垃圾会永久拖慢你。
- 遇到奇怪的bug，先假设是你的错，不是库的错、不是框架的错、不是工具的错。99%是你的错。
- 调试的时候打印一切。不要猜，看真实数据。
- 如果某个行为你解释不了，回滚到最后一个能工作的版本，一点一点加回来。
- 不要在疲惫/生气/赶时间的时候做重大决策。提交、推送、合并前深呼吸看一遍diff。

## 代码质量
- 命名比注释重要。好名字让注释多余。
- 注释解释"为什么"，不是"是什么"。代码本身已经说了是什么。
- 函数要短。一个函数只做一件事。超过50行就该拆。
- 文件要聚焦。一个文件只服务一个主题。超过400行就该分。
- 错误要明确。要么成功要么明确失败，永远不要静默失败（silent failure）。
- 不要吞异常。catch了就要处理或重新抛出或至少打log。
- 不要留注释掉的代码。git会记得。
- TODO要写名字和日期：// TODO(zhouhao, 2026-06-30): xxx，不然永远不会处理。

## 交互规则
- 改之前先告诉我你要改什么、为什么。
- 不要擅自改我没让你改的地方。
- 如果发现更大的问题，先停下来告诉我，不要顺手"帮我修了"。
- 跑完命令要贴结果，不要只说"成功了"。
- 不知道的时候说不知道，不要编。
```

## 附录B：哪些规则值得立刻加进全局配置

对照你现有的全局配置（Codex AGENTS.md里的小爱PM铁律、CLAUDE.md里的第一性原理+对抗式审查），下面这几条是**目前缺的、单独就能显著减少翻车**的：

### 必须加（强烈推荐）
1. **Evidence Before Claims（证据先行）**—— 治"说做完了实际没做"的头号杀手
2. **Root Cause Before Fix（根因先行）**—— 把"第一性原理"工程化成可执行的步骤，不是口号
3. **Teach Back Before Starting（复述确认）**—— 防误解、防跑偏
4. **Honesty Policy（诚实政策）**—— 不猜不编，查了再说
5. **When Stuck, Stop and Ask（卡住就问）**—— 防越修越歪
6. **Baby Steps + Atomic Commits（小步原子提交）**—— 防一次改太多
7. **Prefer Simple Solutions（简单优先）**—— 你已经有但可再强化

### 应该加
8. **Plan First（先计划）**—— 复杂任务先出计划
9. **Red Team Self-Review（自审）**—— 对抗式审查的精简版，作为"交活前最后一步"
10. **Constraints Up Front（约束先行）**—— 动手前列清楚不能动的东西

### 可以按需加
11. Karpathy代码风格规则（命名、函数长度、注释规范等）
