# 主流Code Agent是怎么做真实场景任务跟踪与数据驱动迭代的

> 一句话：没有任何一家公司是靠"跑个公开榜单"来做产品决策的。真正驱动产品创新的，是一套"离线自建题库 + 线上真实行为埋点 + 受控A/B实验 + 人工抽样校准 + 自动化复盘"五层叠加的闭环系统；统计学严谨性体现在"多层防线互相校验、用置信区间而不是单次分数做决策、对裁判本身也要做裁判"。

## 1. 回答你最关心的问题：他们到底在跟踪什么？

大多数人以为Code Agent只看"代码能不能跑通"。但Cursor、Anthropic、OpenAI这三家公开发表的材料里，实际跟踪的指标可以归为六大类，每一类都对应一个产品改进方向：

| 层级 | 跟踪什么 | 它回答什么问题 | 一旦变差意味着什么 |
|---|---|---|---|
| 结果层 | 任务完成率、代码被保留率、PR解决率、测试通过率 | 用户要的事到底做成没 | 模型/框架退步了，或者方向走偏了 |
| 过程层 | 平均轮次、工具调用次数、工具错误率、上下文命中率、首次正确读取文件比例 | Agent是怎么做成的，中间绕了多少弯路 | 工具设计有坑、上下文检索没做好、提示词让模型犹豫 |
| 体验层 | 用户接管点、取消率、重试率、点赞/点踩、继续下一个任务 vs 贴报错堆栈 | 用户满不满意，有没有信任它继续用 | 输出看起来对但实际不好用，也就是"离线高分、线上翻车" |
| 成本层 | 每个任务的token消耗、缓存命中率、首token延迟、端到端耗时 | 规模化用起来贵不贵、快不快 | 新功能虽然效果好但成本翻倍，不能上 |
| 可靠性层 | 崩溃率、OOM率、超时率、长任务中断率、云端任务恢复率 | 能不能稳定跑一整天不炸 | 产品再聪明也没法日常使用 |
| 安全层 | 危险命令拦截率、出网请求审批率、沙箱越界次数 | 会不会搞坏用户东西或泄露数据 | 企业客户不敢用 |

核心判断：**结果层告诉你"有没有用"，其他五层告诉你"为什么有用/没用、能不能规模化用"。** 只看结果层，就会像Cursor早期那样，明明离线分数很高，用户却反馈"变笨了"。

## 2. 五家代表产品的真实做法（我一家一家查过）

### 2.1 Cursor：最完整的"线上-线下双闭环"样本

Cursor是目前公开披露最多的一家，它的做法代表了当前最成熟的范式：

**离线部分 - CursorBench（自建题库）**
- 题目的来源不是公开代码仓库，而是他们自己的内部代码库和受控渠道里工程师真实做过的任务，通过一个叫 Cursor Blame 的机制，把"最终提交的代码"追溯回"当时Agent收到的请求"，天然形成(提问, 标准答案)配对。
- 每隔几个月更新一次题库，因为用户让Agent干的事在变（以前是修bug，现在是跨多文件重构、跑长时间实验、排查生产日志），不更新题库就会考偏。
- 不只用一个分数。他们看"正确性得分 vs 完成所用token中位数"的散点图：右上角才是好模型——又准又省。
- 评分不是单一测试通过就完，而是多维：正确性、代码质量、效率、交互行为。

**线上部分 - 真实流量的受控实验**
Cursor的线上评估最值得学的有这几招：

1. **代码保留率（Keep Rate）**：Agent改完代码后，跟踪一段时间后这些改动还有多少比例留在用户代码库里。被改回去、被删掉、被重写，都算没保留。这是一个不需要用户显式反馈就能自动算的"隐性满意度"指标，比点赞点踩数据量大得多、偏差小得多。
2. **语义级满意度判断**：让一个语言模型去读用户对Agent输出的后续回复——用户接着说"继续做下一个功能"是强正面信号；用户贴一段报错堆栈是强负面信号。这比情感分析准得多，因为它直接基于后续行为。
3. **工具错误分类+异常检测告警**：把工具调用错误分成"参数无效""环境不符""服务商错误""用户中止""超时""未知错误"几类。未知错误一律当bug处理；预期错误则按"每个工具+每个模型"分别建立基线，显著高于基线就自动告警。
4. **消融实验（Ablation）**：判断某个组件到底有没有用，不是看加了之后分数涨不涨，而是在真实流量上"把它关掉"对比。比如他们判断语义搜索价值的时候，直接在一部分流量上完全移除语义搜索，看哪些场景变差了。
5. **自动化"日志医生"**：每周跑一个自动化Agent，让它自己搜索日志、找出新出现或突然增多的问题，自动到任务系统里建单。
6. **崩溃/OOM的双视角排查**：既做"Top-down"（把崩溃率和Statsig实验平台上的功能开关关联，通过A/B看哪个功能引起崩溃），也做"Bottom-up"（抓崩溃瞬间堆栈、鼓励用户自愿上传堆快照、低采样率持续运行堆分配分析），两种视角对不上就说明没找全。

**Bugbot（代码审查机器人）的闭环特别值得抄**
Bugbot是Cursor最经典的"用产品自己产生的数据训练自己"的案例：
- 核心指标叫"解决率"：PR合并时，用AI判断Bugbot提的每条评论里，作者实际修了哪些。这个指标直接回答"它提的问题是不是真问题"。
- 一开始纯离线优化，解决率从52%爬到70%+；后来发现每次PR审查本身就是一个"自然实验"，可以从三个信号反哺规则：用户点踩、评论里说哪里不对、人类审查员补充的Bugbot漏掉的问题。
- 这些信号自动汇总成候选规则，持续在新PR上评估效果，好的就激活、变差的就禁用。结果是11万个仓库启用学习，生成了4.4万条规则，解决率从70%爬到78%，领先第二名15个百分点。

### 2.2 Anthropic / Claude Code：瑞士奶酪多层防御模型

Anthropic在2026年1月发的官方长文是目前业界最系统的Agent评估方法论，核心思想一句话：**没有任何一层评估能抓住所有问题，必须像瑞士奶酪一样多层叠，一层漏掉的另一层兜住**。

他们明确把方法分成六类，各有各的盲区，必须组合用：

| 方法 | 优点 | 缺点 | 什么时候用 |
|---|---|---|---|
| 自动化离线评测 | 迭代快、可复现、每次提交都能跑、不影响用户 | 需要前期投入、题库会过时、可能和真实使用脱节 | 每次改代码/提示词/升级模型前 |
| 生产监控（埋点） | 反映真实行为、抓合成数据造不出来的问题、规模大 | 是事后的，问题已经影响到用户了；信号有噪声 | 上线后持续跑 |
| A/B测试 | 衡量真实用户结果、可控、系统化 | 慢（要几天到几周才有统计显著性）、只能测已经上线的改动 | 大的功能/模型/框架改动 |
| 用户反馈（点赞点踩、bug报告） | 能发现你想都没想到的问题 | 稀少、自选择偏差（只有极度满意或极度不满的才反馈）、非自动化 | 持续收集、用于触发人工查看 |
| 人工抽样阅读对话记录 | 建立失败模式的直觉、发现自动化看不到的细节质量问题 | 费时间、不能规模化、信号偏定性 | 每周固定抽样20-50条 |
| 系统化人工研究（训练过的标注员） | 黄金标准、可以处理主观任务、用来校准LLM裁判 | 贵、慢 | 校准模型裁判、评主观题的时候 |

他们明确说了Claude Code自己的演进路径：**一开始靠员工和外部用户的反馈快速迭代（dogfooding）**，后来才加上评测，先是针对"简洁性、文件编辑、过度工程化"这些具体问题的小评测，再慢慢扩展到复杂行为。评测不是第一天就有全套，而是随着规模被逼出来的。

Anthropic特别强调两个统计学概念，这是很多人忽略的：
- **pass@k vs pass^k**：同一个任务跑k次，pass@k是"k次里至少有1次成功"的概率（适合"只要有一次对就行"的场景）；pass^k是"k次里每次都成功"的概率（适合"用户希望每次都靠谱"的面向客户场景）。随着k增大，前者分数上升、后者下降。一个面向普通用户的产品要盯pass^k，而不是被pass@k的高分迷惑。
- **能力评测 vs 回归评测要分开**：能力评测用来发现"我们还不会什么"，初始通过率应该很低（给团队一个要爬的山）；回归评测用来保证"我们已经会的别退步"，通过率应该接近100%。能力评测跑熟了、通过率很高了，就"毕业"成为回归评测的一部分。

评分器（裁判）的设计上，他们主张三种结合：
- **基于代码的评分器**（跑测试、静态检查、状态校验、工具调用校验）：最快、最便宜、最客观、可复现，但对"多种有效解法"不宽容。
- **基于模型的评分器**（LLM按评分规则打 rubric 分、成对比较、多评委共识）：灵活、能处理开放题，但非确定性、贵、必须和人工校准。
- **人工评分器**：黄金标准，但贵、慢，主要用来校准模型评分器。

最重要的一条经验是：**评结果，别评路径。** 不要去检查Agent是不是按你预想的步骤调用工具的——前沿模型经常能找到你没想到的有效解法，你检查路径就会惩罚这些创造性方案，让评测本身变瞎。Opus 4.5在一个订机票任务上就是因为"发现了政策漏洞给了用户更好的方案"而被错误判负。

### 2.3 OpenAI Codex：以企业合规视角做全链路日志

OpenAI公开披露的重点在企业部署场景，最有价值的是它的**Agent原生遥测（Agent-native Telemetry）**设计理念：

传统安全日志只记录"某个进程启动了、某个文件被改了、某个网络请求发生了"——只能回答"发生了什么"，但回答不了"Agent为什么要这么做、用户当时的意图是什么"。

Codex的做法是支持用OpenTelemetry标准导出整个Agent生命周期的事件日志：用户原始提问、审批决策、工具执行结果、MCP服务器使用情况、网络代理的允许/拒绝事件。企业客户还能通过合规平台拿到活动日志。

他们在内部的用法很有代表性：
- 终端安全工具发现可疑行为后，把Codex日志喂给一个"AI安全分诊Agent"，让它读原始请求、工具活动、审批决策、网络策略决定，自动区分"正常Agent行为""无害错误""需要升级处理的活动"。
- 运营侧也用这些日志看：内部使用情况的变化趋势、哪些工具/MCP最常用、网络沙箱拦截/提示频率、哪些环节需要优化。

这给我们的启发是：**给Agent做的日志，字段设计要能还原"为什么"，而不是只记录"做了什么"**。这需要在Agent执行框架的每个关键节点埋点：用户意图是什么、模型判断是什么、调用工具前的决策理由是什么、审批结果是什么、工具返回了什么、模型如何根据返回调整计划。

### 2.4 GitHub Copilot：天然嵌入已有工作流的"被动"数据收集

Copilot走了一条和前三者很不一样的路：它不长在独立IDE/CLI里，而是长在GitHub本身上。这带来一个巨大的数据优势——**用户工作的所有环节（Issue、PR、Review、Actions、合并）都已经在平台上，天然就是验收场**。

从公开材料看，Copilot的数据闭环核心在于：
- **PR作为天然验收闸门**：Agent开分支、改代码、提PR，最后人类要不要合并、Review里提了什么意见、CI过没过、有没有被打回修改——这些都是现成的强信号，不需要额外埋点就能拿到。
- **Commit署名+Verified标签**：明确区分哪些commit是Agent写的、哪些是人写的、谁是co-author，方便做效果对比（"Agent写的代码后续被修改的比例是多少"）。
- **Session log + Audit log**：管理员可见的会话日志和审计日志，既服务企业合规，也反哺产品改进。
- **自动化规则的约束设计**：定时/事件触发的自动任务，默认不让作者自己批自己的PR，默认忽略无写权限用户触发的事件——这些规则本身就保证了自动化场景下收集到的反馈是真实可信的。

一句话总结Copilot范式：**不另造一套评估系统，而是把Agent塞进已有的企业研发流程里，让流程本身成为评估器**。

### 2.5 国内经验补充：张和的"评测直接驱动自动改代码"闭环

张和（前腾讯PM，AI产品经理大本营作者）分享了他自己做Excel Agent时更激进的一套做法，代表了2026年的新趋势：
- 评测集构建从人工挖掘标注进化到"全流程Agent化"：从挖掘、清洗到预标注，每一层都让Agent盯着质量。
- Trigger（触发器）从关键词匹配升级成语义理解：用小模型跑在抽样数据上，判断这条query有没有评测价值，而不是靠用户骂"你好笨"才发现翻车。
- 打分从"LLM-as-Judge"进化到"Agent-as-Judge"：不再是喂固定prompt让大模型打个分，而是让Agent逐条深度分析完整轨迹、横向对比顶尖模型的表现、再让另一个Agent聚类成报告。
- 最激进的是：**评测一旦暴露某个场景做得不好，Debug Agent直接钻进代码库，分析对应prompt、框架、代码，根据bad case直接改代码、跑通、提PR**。他自己的产品里这套夜间自动debug流程已经在跑，每天早上他能收到飞书发的bug列表和改好的PR。
- 统计学上的诚实：Agent打分准确率有浮动（90-95%），但只要下游有人工兜底，方差是可以算出来的；在输入端做补偿、再加异构数据源，就不会因为单个Agent出错而影响整体判断。

## 3. 统计学严谨性：他们是怎么避免"数字骗自己"的？

这部分是你特别问到的，也是国内很多团队最容易踩坑的地方。从各家公开的做法看，严谨性体现在七个动作上：

### 3.1 题库要"防污染、防饱和、防偏科"

- **防污染**：公开题库（SWE-bench等）的任务来自公开GitHub仓库，很可能已经进了模型训练数据，导致分数虚高。OpenAI已经发现前沿模型可以凭记忆背出标准答案、且60%未解决问题的测试本身有bug，干脆停止公开报告SWE-bench Verified分数。Cursor的做法是题库主要来自内部代码库和受控渠道，降低见过的概率。
- **防饱和**：当一个评测通过率到100%了，它就失去了"衡量进步"的价值，必须开发更难的新题。SWE-bench Verified一年间从30%爬到80%+，正在饱和。
- **防偏科/类别不平衡**：不能只考"Agent该做什么"的正向用例，也要考"Agent不该做什么"的负向用例。Anthropic做网页搜索评测时，专门平衡了"应该搜索"（查天气）和"不该搜索"（回答"谁创立了苹果"）两类题，否则会训出一个"什么都搜一下"的笨Agent。
- **难度分布要故意比真实世界更难**：张和的经验是40%简单中等、20-30%偏难、永远留10-20%极难题，真实世界里万分之一的极端case要放大到1%在题里出现——这和"撵驴"一个道理，一直让它能够到但不轻松。

### 3.2 裁判本身要被裁判（元评估）

- LLM-as-Judge不是直接就能用的，必须先小批量跟人工对打，达到90%以上一致性才能上规模。放我家团队的实践是结构化基准+多评委投票+人工抽样校准三件套，做到了91.9%的一致性。
- 要给LLM裁判一个"我不知道"的选项，避免它瞎编。
- 要按维度拆开打分，每个维度单独让一个LLM判，而不是一个LLM一把梭判所有维度。
- 有类任务不能交给LLM当裁判：安全相关、精确数值、代码能不能跑通、权限判断——这些必须用确定性程序判。
- 定期抽样人工复核，看裁判和人意见不一致的case到底是裁判错了还是人错了，持续校准。

### 3.3 A/B实验的正确打开方式

Code Agent的A/B和传统SaaS的A/B有本质区别，这几点Cursor和Anthropic都踩过坑：

- **实验单元是"任务"不是"用户"**：同一个用户一天可能做5个不同类型的任务，把用户锁死在A或B版本会污染数据。更合理的是用户发起任务时随机分配到A或B版本，同一个用户不同任务可能体验不同版本。
- **灰度放量要阶梯进行**：先5%流量跑3天只盯安全告警，再25%跑一周看功能体验指标，最后全量。Agent改坏了的代价远大于按钮颜色改坏。
- **设自动回滚条件**：任务完成率下降超过5%就自动回滚，不要等人工发现。
- **不能只看一个指标**：Cursor的经验是看一组高层代理指标的"一致变化方向"，而不是对单个指标单独优化。很多时候A版本在一个指标上好一点点，但在体验指标上全面崩，只看一个指标就会上当。
- **观测周期要够长**：Agent任务可能很长（几十分钟到几小时），效果不是立刻能看到的，至少要1-2周。
- **成本和效果一起看**：Cursor试过用更贵的模型做上下文摘要，发现对质量改善微乎其微、成本显著上升，果断砍掉。

### 3.4 要算置信区间，不要看单次分数

- 因为模型输出有随机性，同一个任务跑两次结果可能不同。所以每个任务要跑多次（trials），报告平均通过率+方差/置信区间，而不是单次跑分。
- 样本量小的时候（早期迭代、改动效果明显）可以少跑几次；系统成熟了、改动效果变小了，就需要更多样本才能区分"真的变好了"和"随机波动"。
- 看版本对比的时候要看效应量（effect size）和统计显著性，而不是"新版本82% vs 旧版本80%，涨了2个点"——2个点可能完全在噪声范围内。

### 3.5 分桶抽样，别被总分骗了

- 一个新版本总分涨了2%，但可能是"简单题涨了10%、难题跌了20%"平均出来的，实际对重度用户是变差了。
- 必须按场景分桶（代码库大小、任务类型、语言、任务长度、是否用了工具、用户细分）分别看指标，总分掩盖局部退步是最常见的坑。
- Cursor还专门按"每个工具+每个模型"分别建立错误率基线，就是因为不同模型用工具的出错率天然不同，搁在一起比会误报。

### 3.6 看数据，别看仪表盘（Look at your data）

这是Anthropic和张和都反复强调的一条"心法级"经验：
- 每周必须亲自抽样读20-50条真实对话记录，不要只看聚合数字。
- 当一个任务失败时，一定要读完整轨迹（transcript），判断是Agent真错了、还是你的评分器错杀了一个有效解法。
- Opus 4.5在CORE-Bench上从42%跳到95%，不是因为模型突然变强了，而是研究员发现评分器对数字精度要求过严、任务描述模糊、任务本身随机不可复现——是评测在胡说。
- 张和的话更狠："只要在几个关键阶段认真完成100个以上真实case的错误分析和归纳，你对产品技术边界的理解深度，很可能超过团队里包括算法工程师在内的所有人。"

### 3.7 环境隔离与可复现

- 每次评测跑必须从干净的隔离环境开始：文件残留、缓存、上一次跑留下的git历史、资源耗尽，都会让评分不准。Anthropic在内部评测里发现过Claude靠查看前几次跑留下的git历史"作弊"拿了虚高的分数。
- 涉及外部服务的任务（网络请求、数据库、第三方API）要做录制重放或者mock，不然今天跑和明天跑结果不一样，根本没法对比版本。
- 云端Agent的可靠性评测要专门考虑"节点挂了、推理服务抖动、pod被回收"这些真实分布式系统问题，Cursor把架构迁到Temporal才把可靠性从1个9提升到2个9以上。

## 4. 数据是怎么从"收集上来"变成"产品决策"的？

把各家做法串起来，能看到一条标准的数据流：

```
用户真实使用
   ↓（埋点采集：用户输入、Agent轨迹、工具调用、用户反馈、最终代码命运）
原始日志/轨迹  →  实时监控告警（未知错误爆发、崩溃率飙升、异常指标立刻报警）
   ↓
按任务/用户/场景分桶聚合  →  仪表盘看趋势（天/周维度的六类指标变化）
   ↓                                 ↓
离线题库扩充         A/B实验对比新版本（阶梯放量、多指标一致变化、置信区间校验）
   ↓                                 ↓
跑回归+能力评测       人工抽样读case、LLM裁判辅助归类失败模式
   ↓                                 ↓
定位根因（是模型问题？提示词问题？工具问题？环境问题？评分器问题？）
   ↓
改代码/改提示词/改工具/改模型  →  跑离线评测验证  →  再走A/B  →  上线  →  下一轮
```

几个关键的产品创新触发点：
1. **仪表盘上某个指标突然跳变** → 触发告警 → 定位回归。Cursor的工具错误率下降一个数量级就是这么来的。
2. **A/B发现新模型/新框架在某个分桶里明显好/差** → 做模型特化适配（比如给OpenAI模型用patch格式编辑、给Claude用字符串替换）。
3. **人工抽样发现反复出现的失败模式** → 变成新的评测题加进题库。Anthropic说"失败的案例变成测试用例，测试用例防止退步"。
4. **用户反馈/点踩聚集在某类任务** → 提示这类任务的prompt或工具设计有问题。Bugbot的学习规则就是从PR评论、点踩、人类审查意见里生出来的。
5. **Top失败case自动聚类后发现共性** → 要么改产品（比如Cursor发现大文件导致OOM，就加分块加载和熔断），要么加新功能（比如发现用户经常让Agent解释代码，就做专门的代码解释视图）。
6. **离线评测和线上信号对不上** → 说明题库过时了或者评分器有bug，必须先修评测再信评测。CursorBench能持续有效，关键就在它每几个月根据真实分布更新一次。

## 5. 对我们做同类产品的具体启示（可直接落地）

如果从0到1搭这套系统，按优先级顺序：

**第一阶段（用户量还小，几千到几万会话/天）**
- 先把"Agent全链路轨迹日志"做了：每轮用户输入、模型输出、工具调用（含参数和返回）、审批决策、最终代码diff、用户后续行为（是否继续、是否回退、是否点踩），这是后面一切的基础。
- 建最小化的内部题库：从真实用户失败case里挑20-50个有代表性的，覆盖高频任务类型和典型翻车场景。每条题都要有"两个专家能独立判断是否通过"的明确评分标准，并且要先写一个参考解法证明题是可解的。
- 每周固定读20-50条真实对话记录，手动标注成败原因，归纳失败模式。
- 先别搞复杂的LLM裁判，用确定性评分器（跑测试、静态检查、状态校验）+ 少量人工校准起步。

**第二阶段（开始规模化，有流量做实验）**
- 引入A/B实验系统，实验单元按"任务"随机化；建立阶梯灰度和自动回滚机制。
- 开始算"代码保留率"这种隐性结果指标，不要只靠点赞点踩。
- 把题库按能力/回归分开管理；能力评测给团队一个明确的"要爬的山"，回归评测保证别退步。
- LLM裁判小规模上线，但必须先和人工对齐到90%+一致性才信它；给它"不知道"选项；安全/数值/代码正确性类问题别用它判。
- 按关键维度分桶看指标（任务类型、代码库规模、语言、用户分层），别只看总分。

**第三阶段（成熟阶段，追求自动化飞轮）**
- 建自动case挖掘流水线：从用户点踩、错误爆发、人工标注的失败case里自动生成候选评测题，人工审核后加入题库。
- 跑"评测→自动定位根因→自动改代码/提示词→自动提PR→自动跑评测验证"的半自动闭环（人在最后审核PR）。
- 建异常检测和自动告警：不只是未知错误，预期错误也要按分桶建基线，突增就报警。
- 把全链路日志按OpenTelemetry这类标准格式输出，既能做产品分析也能做企业合规审计。
- 让每周的"读case"变成团队文化，而不只是数据分析师的事——产品经理、工程师、设计师都要轮流看真实用户的使用轨迹。

## 6. 最后几个反直觉但很重要的经验

1. **评测本身就是产品需求文档（Evals as PRD）**：在AI产品里，写清楚"什么算好"比写清楚"要做什么"更重要。一个好的评测case就是一条活的、可自动验证的需求。
2. **评测本身也要迭代、也要被评测**：不要相信你的评测是完美的。当分数怎么都上不去的时候，先怀疑是不是题出得有问题、评分器有bug、环境不公平——这种情况比模型真不行更常见。
3. **线上和离线必须互相对齐、互相校准**：Cursor的核心判断是"CursorBench分数和在线评估结果一致"才是有效基准；如果离线说A比B好但线上真实用户觉得B更好，信线上，回去补题库。
4. **"看起来很有前景"的想法经常在A/B里死掉**，这是好事——它省了你上线后伤害用户。Cursor就是这么放弃了"贵模型做摘要"的方案。
5. **大多数进步不是来自大跃迁，而是小优化的累积**。Cursor说"改进框架靠的是近乎执着地叠加一个个小优化"，Bugbot做了40次重大实验才把解决率从52%推到78%。不要看不起1-2个点的提升，累积起来就是代际差距。
6. **最容易被忽视但最值钱的数据是"用户在哪里接管"和"Agent改的代码最后有多少被留下来"**。这两个数据不需要用户主动反馈，却最诚实。
